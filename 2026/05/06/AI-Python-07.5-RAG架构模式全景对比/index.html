AI-Python-07.5: RAG 架构模式全景对比

传统 RAG（07.1-07.3）、GraphRAG（07.4）只是 RAG 的两种形态。2024-2025 年间，RAG 架构快速演化，出现了 LLM Wiki、Agentic RAG、CRAG、Self-RAG 等多种模式。本章全景梳理六大 RAG 架构的核心思路、优劣对比、选型决策，帮你为具体场景选对架构。

1. RAG 架构演化脉络

1.1 从简单到复杂的进化路径

    timeline
    title RAG 架构演化时间线
    2020 : Naive RAG : 原始论文（Lewis et al.）
    2023 : Advanced RAG : 混合检索 + Rerank
    2024 : GraphRAG : 微软发布知识图谱增强
    2024 : CRAG : 纠正性检索增强生成
    2024 : Self-RAG : 自反思检索增强生成
    2024 : Agentic RAG : 多 Agent 协同检索
    2025 : LLM Wiki : Karpathy 提出"预编译知识库"

1.2 六大架构一句话概括

架构 一句话总结 核心创新点
Traditional RAG 检索相关文档片段，让 LLM 基于片段回答 Embedding + 向量检索
LLM Wiki 提前把知识”编译”成 Wiki，直接问 LLM 预编译，省去运行时检索
GraphRAG 构建知识图谱，支持跨文档关系推理 实体-关系图谱 + 向量
Agentic RAG Agent 自主决定”搜什么、怎么搜” 检索策略动态决策
CRAG 检索结果不靠谱？自动纠正或换搜索引擎 检索质量评判 + 纠正
Self-RAG LLM 自我评判”要不要检索、检索结果好不好” 自反思机制

2. Traditional RAG（经典检索增强）

2.1 架构概览

    flowchart LR
    Q[用户提问] --> E[Embedding\n向量化]
    E --> V[向量数据库\n相似度检索]
    V --> C[Top-K 文档片段]
    C --> L[LLM\n生成回答]
    Q --> L

    style Q fill:#fff3cd,stroke:#ffc107,stroke-width:2px,color:#333
    style V fill:#d1ecf1,stroke:#17a2b8,stroke-width:2px,color:#333
    style L fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333

07.1-07.3 已详细讲解，这里只做定位回顾。

2.2 核心特征

特征 描述
检索方式 向量相似度（余弦/内积）+ 可选 BM25
知识形态 原始文档的文本片段（chunk）
运行时行为 每次查询都执行检索
基础设施 向量数据库（Chroma/FAISS/Milvus）

2.3 优势与局限

优势 局限
生态成熟，开箱即用 通用 Embedding 对领域术语理解有限
可扩展到百万级文档 分块策略影响质量，需反复调优
查询延迟低（100-500ms） 只能找”相似文本”，无法推理关系
增量更新简单 对结构化内容（代码、表格）效果不稳定

3. LLM Wiki（预编译知识库）

3.1 起源

2025 年初，Andrej Karpathy（OpenAI 联合创始人、前 Tesla AI 总监）提出了 LLM Wiki 模式：不依赖向量数据库，而是用 LLM 把原始资料”编译”成结构化的 Wiki 知识库，直接利用 LLM 的上下文窗口进行问答。

3.2 架构概览

    flowchart TB
    subgraph 三层架构
        direction TB
        L1["📁 Layer 1: Raw Sources（原始资料）\n不可变，只追加"]
        L2["📝 Layer 2: Wiki（LLM 编译后的知识）\n人工可审核"]
        L3["📋 Layer 3: Schema（index.md + CLAUDE.md）\n结构化索引"]
    end

    L1 -->|"Compile\nLLM 编译"| L2
    L2 -->|"Index\n自动索引"| L3

    subgraph 查询流程
        Q[用户提问] --> LLM[LLM]
        L3 -->|"读取索引"| LLM
        L2 -->|"读取相关 Wiki"| LLM
        LLM --> A[回答]
    end

    style L1 fill:#fff3cd,stroke:#ffc107,stroke-width:2px,color:#333
    style L2 fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333
    style L3 fill:#d1ecf1,stroke:#17a2b8,stroke-width:2px,color:#333
    style LLM fill:#ffd1d1,stroke:#e57373,stroke-width:2px,color:#333

3.3 四个核心操作

操作 说明 时机
Ingest（摄入） 把新资料（文章、视频笔记、网页剪藏）放入 Layer 1 新资料产生时
Compile（编译） LLM 读取原始资料，生成结构化 Wiki 页面 摄入后一次性执行
Query（查询） LLM 读取 index.md 定位相关 Wiki，直接回答 用户提问时
Lint（维护） 定期检查 Wiki 页面的准确性、去重、更新 定期执行

3.4 关键文件

index.md — 知识目录

log.md — 变更日志

3.5 为什么能不用向量数据库？

    flowchart LR
    subgraph "传统 RAG: 运行时检索"
        Q1[提问] --> S1[搜索向量DB] --> R1[返回碎片] --> L1[LLM 拼凑]
    end

    subgraph "LLM Wiki: 预编译"
        Q2[提问] --> I2[读 index.md] --> W2[读完整 Wiki] --> L2[LLM 直接答]
    end

    style S1 fill:#ffd1d1,stroke:#e57373,stroke-width:2px,color:#333
    style I2 fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333

核心数学： 126 篇博客 ≈ 40 万字 → 编译后约 15-20 万字 → 压缩后 index.md + 几篇相关 Wiki 约 2-5 万字 → 远小于现代 LLM 的 128K token 上下文窗口。

LLM 上下文窗口 可容纳字数 覆盖情况
8K tokens ~1.2 万字 不够
32K tokens ~5 万字 勉强
128K tokens ~20 万字 完全覆盖 126 篇编译后的 Wiki
200K+ tokens ~30 万字 裕度充足

3.7 优势与局限

优势 局限
零基础设施：不需要向量数据库、Embedding 模型 首次编译成本高：需要 LLM 逐篇编译
知识质量可控：编译后人工审核，非碎片化 chunk 规模上限：~500 篇文档后性能下降
天然 Git 管理：所有知识是 Markdown 文件 有损压缩：编译过程可能丢失细节
查询极快：直接读文件，无数据库查询开销 更新需重编译：修改原文后要重新编译对应 Wiki
可读性强：人可以直接阅读 Wiki，向量 DB 不可读 不适合精确搜索：”这个函数在哪用过”不如向量检索

4. GraphRAG（知识图谱增强）

详见 07.4 章节，此处仅做对比定位。

4.1 核心思路

    flowchart TB
    D[原始文档] --> NER[LLM 实体抽取]
    NER --> KG[知识图谱\n实体 + 关系]
    D --> E[Embedding]
    E --> V[向量数据库]

    Q[用户提问] --> R{问题类型?}
    R -->|关系推理| G[图谱查询\nCypher]
    R -->|语义匹配| S[向量检索]
    G --> F[结果融合]
    S --> F
    F --> L[LLM 生成回答]

    style KG fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333
    style V fill:#d1ecf1,stroke:#17a2b8,stroke-width:2px,color:#333
    style L fill:#ffd1d1,stroke:#e57373,stroke-width:2px,color:#333

4.2 对比定位

维度 定位
杀手锏 跨文档关系推理（”A 和 B 有什么关系”）
适用场景 人物关系、组织架构、事件关联、知识推理
不适用 简单 FAQ、代码查询、精确关键词搜索
成本 高（构建 + 维护知识图谱）

5. Agentic RAG（智能体检索）

5.1 核心思路

传统 RAG 的检索策略是固定的（总是向量检索 → Top-K → LLM）。Agentic RAG 让 Agent 自主决定检索策略。

    flowchart TB
    Q[用户提问] --> A[Router Agent\n路由 Agent]
    A -->|简单问题| D[直接回答\n无需检索]
    A -->|需要检索| B[Retriever Agent\n检索 Agent]
    A -->|需要计算| C[Tool Agent\n工具 Agent]

    B --> B1[选择检索策略]
    B1 -->|语义| V[向量检索]
    B1 -->|关键词| BM[BM25 检索]
    B1 -->|关系| G[图谱查询]
    B1 -->|Web| W[网络搜索]

    V --> J[Judge Agent\n评判结果]
    BM --> J
    G --> J
    W --> J

    J -->|满意| L[LLM 生成回答]
    J -->|不满意| B

    style A fill:#ffd1d1,stroke:#e57373,stroke-width:2px,color:#333
    style J fill:#fff3cd,stroke:#ffc107,stroke-width:2px,color:#333
    style L fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333

5.2 多 Agent 协作

5.3 优势与局限

优势 局限
灵活：不同问题走不同路径 延迟高：多轮 Agent 协商
智能：简单问题不浪费检索资源 成本高：每次查询多次 LLM 调用
可扩展：轻松接入新工具 调试难：Agent 决策链路不透明
容错：一条路径失败可换路 稳定性差：Agent 可能做出意外决策

6. CRAG（纠正性检索增强）

6.1 核心思路

CRAG（Corrective RAG）在传统 RAG 流程中增加一个检索评估器（Retrieval Evaluator），评判检索结果的质量，质量差时自动纠正。

    flowchart TB
    Q[用户提问] --> R[检索\n向量 / BM25]
    R --> E{检索评估器\nRetrieval Evaluator}

    E -->|"✅ 正确\n置信度 > 阈值"| G[LLM 生成回答]
    E -->|"❌ 错误\n置信度 < 阈值"| W[Web 搜索\n补充知识]
    E -->|"⚠️ 模糊\n置信度接近阈值"| F[知识精炼\n重排序 + 过滤]

    W --> G
    F --> G

    style E fill:#fff3cd,stroke:#ffc107,stroke-width:2px,color:#333
    style G fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333
    style W fill:#ffd1d1,stroke:#e57373,stroke-width:2px,color:#333

6.2 检索评估器

6.3 优势与局限

优势 局限
提高回答可靠性：过滤低质量检索结果 增加延迟：额外的评估步骤
自动降级到 Web 搜索 增加成本：每篇文档需 LLM 评估
实现简单：在传统 RAG 上加一层评估 评估器本身可能误判

7. Self-RAG（自反思检索增强）

7.1 核心思路

Self-RAG 让 LLM 在生成过程中自我反思三个问题：

反思标记 含义 时机
Retrieve “这个问题需要检索吗？” 生成前
ISREL “检索到的文档相关吗？” 检索后
ISSUP “我的回答有文档支撑吗？” 生成后 

    flowchart TB
    Q[用户提问] --> L1[LLM]
    L1 --> R1{"是否需要检索?\n[Retrieve]"}
    R1 -->|否| A1[直接生成]
    R1 -->|是| RET[执行检索]
    RET --> L2[LLM + 检索结果]
    L2 --> R2{"文档相关吗?\n[ISREL]"}
    R2 -->|是| G1[生成回答]
    R2 -->|否| RET2[重新检索]
    RET2 --> L2
    G1 --> R3{"回答有据吗?\n[ISSUP]"}
    R3 -->|是| FINAL[输出最终回答]
    R3 -->|否| G2[重新生成\n补充来源]
    G2 --> R3

    style R1 fill:#fff3cd,stroke:#ffc107,stroke-width:2px,color:#333
    style R2 fill:#fff3cd,stroke:#ffc107,stroke-width:2px,color:#333
    style R3 fill:#fff3cd,stroke:#ffc107,stroke-width:2px,color:#333
    style FINAL fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333

7.3 优势与局限

优势 局限
回答质量高：三重反思把关 延迟最高：可能多次检索+生成
减少幻觉：强制要求有据可依 成本高：每个查询 3-9 次 LLM 调用

8.3 选型速查表

你的场景 推荐架构 理由
个人博客/笔记知识库（< 500 篇） LLM Wiki 零基础设施，Git 管理，人可读
企业文档问答（500-10万 篇） Traditional RAG 成熟稳定，可扩展
需要跨文档关系推理 GraphRAG 图谱天然支持关系查询

9. 融合架构：LLM Wiki + 按需 RAG

9.1 为什么需要融合？

每种架构都有局限。融合架构取各家之长：

层 用什么 解决什么问题
Layer 1 LLM Wiki 小规模时零基础设施，知识可读
Layer 2 Traditional RAG 大规模时高效检索
Layer 3 GraphRAG（可选） 需要关系推理时叠加 
    flowchart TB
    subgraph "知识生产"
        RAW[原始文档] --> COMPILE[LLM 编译]
        COMPILE --> WIKI[Wiki 知识库]
    end

    subgraph "知识检索"
        Q[用户提问] --> ROUTE{路由判断}

        ROUTE -->|"规模 < 500\n或简单查询"| LLM_WIKI[LLM 直接读取\nWiki + index.md]
        ROUTE -->|"规模 > 500\n或精确搜索"| RAG[向量检索\nBM25 + Rerank]
        ROUTE -->|"关系查询"| GRAPH[图谱查询]
    end

    LLM_WIKI --> MERGE[结果融合]
    RAG --> MERGE
    GRAPH --> MERGE
    MERGE --> ANSWER[最终回答]

    style WIKI fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333
