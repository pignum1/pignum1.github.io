1. 为什么 Agent 需要眼睛

1.1 纯文本 Agent 的局限

场景 文本 Agent Vision Agent
用户发了一张架构图 “我看不了图片” 分析组件、关系、流程
截图中的报错信息 需要用户手动打字 直接读图识别错误
PDF 文档中的表格 无法处理 OCR 提取表格数据
产品图片对比 只能靠文字描述 视觉对比差异

核心就一件事：图片 + 文字提问 → 文字回答。Vision LLM 内部把图片编码成 token，和文字 token 一起喂给模型。

2.2 Prompt 精度影响输出质量

Prompt 输出质量
“描述这张图” 宽泛，可能遗漏关键细节
“列出图中的每个组件名称，并说明箭头方向代表什么” 精确，输出结构化、信息密度高

经验：Prompt 越具体，Vision LLM 的输出越有用。就像给人看图提问一样，”你看这图”和”这个图里第二个框写的是什么”得到的结果天差地别。

5. 应用场景与最佳实践

5.1 实用场景

场景 做法 效果
架构图理解 analyze_image + search_wiki 看图 → 识别组件 → 搜索相关文档 → 详细解释
代码截图 OCR 提取代码 + 解释 从截图识别代码并分析
文档 OCR 扫描件 → 文字提取 PDF/图片中的表格、文字提取
产品对比 多张图片分析 分别分析再对比差异

5.2 当前局限

局限 表现 应对
幻觉 描述图中不存在的内容 交叉验证，关键信息要求引用图中原文
细节丢失 小字、模糊文字识别不准 放大关键区域，或用专用 OCR 工具
复杂图表 多层嵌套图可能理解错 分区域描述，或简化图表再分析
成本较高 Vision 调用比纯文本贵 只在需要看图时调用，避免每次都传图

5.3 最佳实践

  1. Prompt 要具体 — “描述这张图” vs “列出图中每个框的名称和箭头含义”
  2. 只在需要时看图 — 纯文字问题不需要调 Vision
  3. 交叉验证 — Vision 结果 + 知识库搜索，双重确认
  4. 图片预处理 — 裁剪关键区域、提高分辨率、转为 PNG

6. 总结

关键概念

  • Vision LLM 不是 Agent 的大脑，是 Agent 的工具 — analyze_image 就是一个 @tool
  • Text LLM（决策）+ Vision LLM（看图）= Vision Agent
  • Prompt 精度直接决定 Vision 输出质量

实战练习

  1. 入门：修改 exercise_08，用本地的一张截图做 OCR 测试
  2. 进阶：在 exercise_09 的 Agent 中增加一个”对比”功能，支持分析两张图片的差异
  3. 挑战：实现 Vision + Supervisor 多 Agent 团队（参考 08.3 + 本文 4.2 节）
