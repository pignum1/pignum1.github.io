09.1 让 Agent 长出了”眼睛”，本章让它长出”嘴巴”和”耳朵”。TTS（文字→语音）让 Agent 能说话，ASR（语音→文字）让 Agent 能听懂用户说话。两者结合就是语音 Agent。

1. 为什么 Agent 需要嘴和耳朵

1.1 纯文字 Agent 的局限

场景 文字 Agent 语音 Agent
用户在开车时提问 无法使用（不能打字） 语音输入→语音回答
朗读长文档 只能看屏幕 语音播报，解放眼睛
电话客服 需要人转述 直接语音对话
语音笔记转文字 无法处理 ASR 识别 + 整理

1.2 语音交互的核心流程

核心就两件事：听（ASR） 和 说（TTS）。Agent 的大脑仍然是 Text LLM，语音只是输入/输出通道。

1.3 智谱 GLM 语音 API 体系

API 模型 功能 调用方式
GLM-TTS glm-tts 文字→语音（纯 TTS） POST /audio/speech
GLM-4-Voice glm-4-voice 语音↔文字（端到端） chat/completions

5. GLM-TTS vs GLM-4-Voice 选型

5.1 对比

维度 GLM-TTS GLM-4-Voice
定位 纯 TTS（文字→语音） 端到端语音模型
输入 文字 文字 + 语音
输出 音频二进制 文字 + 音频
API POST /audio/speech chat/completions
ASR 能力 无 有（能理解语音）
TTS 质量 更好（专用模型） 可以（多任务模型）
成本 较低 较高
延迟 较低 较高

5.2 选型建议

  • 只需要朗读 → GLM-TTS（音质好、成本低）
  • 需要语音对话 → GLM-4-Voice（能听能说）
  • Agent 场景 → 两者结合（ASR 用 GLM-4-Voice，TTS 用 GLM-TTS）
  • 实时对话 → 两者都用流式模式，降低延迟

5.3 实际生产架构

6. 最佳实践与局限

6.1 最佳实践

实践 说明
流式优先 TTS/ASR 都用流式模式，降低感知延迟
VAD 检测 语音输入前加 VAD（Voice Activity Detection），避免静音段浪费计算
缓存 TTS 相同文字不重复合成，缓存音频文件
降级策略 TTS 失败时降级为纯文字回答
音频格式统一 统一使用 44100Hz 单声道 16bit WAV

6.2 当前局限

局限 表现 应对
延迟 端到端语音链路延迟 2-5 秒 流式 + 缓存 + VAD
情感表达 TTS 语音情感较平淡 选择更好的音色、调整 speed/volume
噪声 嘈杂环境 ASR 准确率下降 前端降噪、WebRTC AEC
打断 Agent 说话时用户插话 实现全双工对话、VAD 检测打断
方言 非普通话识别不准 模型微调、多语言模型

7. 总结

关键概念

  • 语音 Agent = Text LLM（决策）+ TTS Tool（说）+ ASR Tool（听）
  • TTS 和 ASR 都封装为 @tool，Agent 自动决定何时调用
  • GLM-TTS 适合纯朗读，GLM-4-Voice 适合语音对话
  • 流式模式是生产环境的必选项

与 Vision Agent 的统一架构

Agent 的大脑始终是 Text LLM，多模态能力通过 Tool 扩展。

实战练习

  3. 挑战：实现流式语音对话（TTS + ASR 都用 stream 模式）
