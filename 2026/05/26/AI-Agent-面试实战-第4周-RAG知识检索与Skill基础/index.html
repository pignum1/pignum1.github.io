AI Agent 面试实战 — 第4周：RAG 知识检索与 Skill 基础

第4周完成 L2 反思层的 RAG 知识检索——混合检索 + Reranker 重排序 + Token 预算联合裁剪 + 层级知识库 + Skill 注册基础。让 Agent 能从知识库中精准检索信息，并和记忆系统协作注入上下文。

一、RAG 整体设计

Q1: 你们的 RAG 方案整体架构是怎样的？

难度：⭐⭐⭐（进阶）
频率：★★★★★（必问）

回答结构：

“我们的 RAG 方案是基于混合检索 + 重排序 + 层级知识库的架构。

完整流程：

  1. 文档注入阶段：用户上传文档到知识库 → 语义分块（512 token / 重叠 64 token）→ 调用 bge-m3 Embedding API 生成向量 → 存入 pgvector，同时生成 tsvector 全文索引。
  2. 检索阶段：
    • Dense 检索：将查询向量化，在 pgvector 中做余弦相似度搜索，返回 Top-K
    • Sparse 检索：用 PostgreSQL tsvector 做 BM25 全文检索，返回 Top-K
    • 两路结果通过 RRF（Reciprocal Rank Fusion） 算法融合排序，k=60
  3. 重排序阶段：用 bge-reranker-v2-m3 做 Cross-Encoder 精排，对融合结果重新打分，返回最终 Top-K
  4. 注入阶段：Context Manager 按知识库层级（Agent 专属 → 团队 → 全局）补充检索结果，按 Token 预算裁剪 RAG 文档数量，最终注入 Prompt。”

追问：为什么选择 RRF 融合而不是线性加权？

“RRF 不需要调权，也不依赖分数的绝对大小（因为 Dense 和 Sparse 的分数分布差异很大，Dense 是 0~1 的相似度，BM25 可能 >10），它只关心排序位置。公式：score(d) = Σ 1/(k + rank_i(d))，k=60。这个值是通过实验确定的——在代码检索和文档检索的混合场景中，k=60 时 NDCG@10 最高。”

二、Dense vs Sparse 检索

Q2: Dense 检索和 Sparse 检索各自适用什么场景？为什么要同时用？

难度：⭐⭐⭐（进阶）
频率：★★★★★（必问）

回答结构：

“Dense 和 Sparse 是互补的。

Dense 检索（语义向量搜索）：

  • 原理：用 Embedding 模型把文本和查询转成向量，在 pgvector 中用余弦相似度找最近的 chunk
  • 适用场景：自然语言描述（’怎么优化数据库性能’）、同义词匹配（’登录’也能匹配到’sign in’）
  • 局限：对精确关键词匹配弱（findUser 可能返回 fetchUser，语义接近但不是同一个函数）

Sparse 检索（BM25 全文搜索）：

  • 原理：基于 PostgreSQL tsvector 倒排索引 + BM25 评分，精确匹配关键词
  • 适用场景：代码函数名搜索（findUser）、API 端点搜索（POST /api/users）
  • 局限：不理解语义，无法处理同义词

混合后效果对比（在我们的开发文档测试集上）：

方案 Recall@10
纯 Dense 0.82
纯 Sparse 0.68
混合 + RRF 0.91
混合 + Reranker 0.94

所以混合检索不是为了炫技，而是确实带来了约 9% 的召回提升。”

三、分块（Chunking）策略

Q3: 文档分块（Chunking）是怎么做的？有哪些策略考虑？

难度：⭐⭐⭐（进阶）
频率：★★★★（常见）

回答结构：

“我们采用语义分块作为默认策略，配合递归分块作为兜底。

具体实现：

  1. 优先按 Markdown 标题层级切分。 以 ##、### 为边界，保证每个 chunk 是一个完整的语义单元（如一个函数说明、一个章节）。
  2. 若文档无结构（纯文本），则按段落边界切分，保证 max_chunk_size=512，overlap=64。Overlap 保证了在被切开的句子中，跨 chunk 的上下文不会丢失。
  3. 递归兜底：如果某个段落仍然超过 512 token，递归按句子、短语切分，直到满足限制。

关键发现：元数据参与 Embedding 能显著提升检索质量。 我们把标题、章节名、文档类型拼接到 chunk 内容的前面一起嵌入，准确率提升了约 15%。”

四、混合检索与重排序

Q4: 为什么需要 Reranker？混合检索的结果不够好吗？

难度：⭐⭐⭐（进阶）
频率：★★★（偶尔）

回答结构：

“混合检索（RRF）做的是粗排——它把两路结果按位置融合，但没有深度计算 query 和每篇文档的语义匹配度。

Reranker（Cross-Encoder）做的是精排——把 query 和每个候选文档拼成一对，送入模型计算精细的相关性分数。这个过程更慢，但更准。

效果对比：

方案 NDCG@10
混合检索 0.87
混合 + Reranker 0.93

Reranker 在混合检索的基础上又提升了 6%。但 Reranker 的计算成本高（每个候选都要和 query 对算一次），我们只在 Top-20 的候选上做 Reranker，然后取最终 Top-5。这样既保证了精度，又控制了延迟。”

五、Token 预算与裁剪策略

Q5: RAG 结果的 Token 预算是怎么分配的？和 Memory 是什么关系？

难度：⭐⭐⭐⭐（深入）
频率：★★★★（常见）

回答结构：

“RAG 和 Memory 共享同一个上下文窗口，因此必须联合分配 Token 预算。

我们的分配模型（以 32000 token 窗口为例）：

RAG 的 9600 tokens 具体分配：每个 chunk 平均约 200 tokens，默认 Top-K = 10 约 2000 tokens。如果结果不足，剩余预算留给 Memory。

联合裁剪顺序（当总预算不足时）：

  1. 首先减少 RAG 文档数（Top-K 从 10 降到 5 → 3）
  2. 若仍不足，裁剪 L4 上下文记忆中 importance 最低的条目
  3. 再裁剪 L3 个人记忆中低 importance 的条目
  4. 裁 L2 早期条目
  5. 最后裁 L1（极少发生）

RAG 优先被裁的原因是：外部知识通常可以通过更精准的 query 浓缩，而内部记忆（尤其是 L2 决策）丢失后 Agent 容易产生幻觉或违反团队规范。”

Q6: 裁剪的具体实现细节？怎么确认裁剪后的效果？

难度：⭐⭐⭐⭐⭐（深入）
频率：★★★（偶尔）

回答结构：

“裁剪的触发条件和实现流程分三步：

1. 计算当前 Token 总量

每次组装 Prompt 前，用 tiktoken 计算 System Prompt + Task State + Memory + RAG 的总 Token 数。

2. 判断触发级别

  • 若总量 > 窗口 × 0.7：触发摘要压缩——把多条记忆合并为一段结构化摘要（调用轻量模型）
  • 若总量 > 窗口 × 0.9：触发硬裁剪——直接丢弃低重要性条目

3. 硬裁剪算法

对每层记忆，按 importance 升序排序，逐个移除直到总量回到 0.9 阈值以下。对 RAG，减少 Top-K 数量。

效果评估方法：我们在内部测试集上做了 A/B 测试——给定 20 个标准问答任务（涵盖代码生成、架构设计、规范咨询），分别用无裁剪、仅压缩、压缩+裁剪三种模式运行，用 LLM-as-Judge 评估回答的完整性、准确性、相关性。

模式 平均得分 截断率
无裁剪 0.89 15%
仅压缩 0.87 3%
压缩+裁剪 0.85 0%

在 0.04 的得分损失下，系统稳定性提升了 15%，这是可接受的取舍。”

Q7: 压缩和裁剪的区别是什么？什么时候用哪一个？

难度：⭐⭐⭐⭐（深入）
频率：★★★（偶尔）

回答结构：

“压缩和裁剪是两个不同层级的应对策略：

摘要压缩：当 Memory 总量达到分配预算的 70% 时触发。不是丢弃记忆，而是调用轻量 LLM（如 GPT-4o-mini）把多条记忆合并成一段摘要。优点是信息不会完全丢失，缺点是会丢失细节。适合中等问题。

硬裁剪：当总量达到窗口的 90% 时触发。直接按 importance 排序，丢弃最低分的条目。优点是快速、确定性高；缺点是信息可能永久丢失。

两者配合：轻度超标先压缩（保留概要），严重超标再裁剪（保最重要）。

压缩效果示例：

裁剪优先级：RAG 文档数 → L4 低 importance → L3 低 importance → L2 早期 → L1（极少触发）。”

六、效果评测与质量保证

Q8: RAG 检索质量怎么评测？有什么指标？

难度：⭐⭐⭐⭐（深入）
频率：★★★（偶尔）

回答结构：

“我们使用检索和生成两个阶段的评测指标：

检索阶段：

指标 含义 目标
Recall@K 相关文档被检索到的比例 > 0.90
MRR 第一个相关文档的平均排名倒数 —
NDCG@K 考虑位置权重的排序质量 —

评测数据集：我们自己构建了一个包含 50 个查询、200 个标注文档对的测试集，覆盖代码检索、架构文档、API 规范等场景。

生成阶段：

指标 含义
Faithfulness 生成内容是否基于检索到的文档（用 TruLens 评测）
Answer Relevance 回答是否直接回应了问题
Context Precision 检索到的内容中相关部分的比例

我们会定期（每周）跑一次评测。如果连续两周 Recall@10 下降超过 3%，就会触发告警，检查 Embedding 模型或 chunk 策略是否需要更新。”

Q9: RAG 系统上线后，发现检索结果不理想，你会怎么优化？

难度：⭐⭐⭐⭐（深入）
频率：★★★（偶尔）

回答结构：

“这是典型的 RAG 系统调优问题，我会从以下几个方向排查：

  1. 诊断问题的性质：是检索不回、检索不准、还是检索太多无关内容？
  2. Chunk 维度：是不是分块太大导致语义稀释，或者太小导致上下文丢失？尝试调整大小或 overlap。
  3. Query 端：用户的原始问题是不是不适合直接向量化？尝试 Query Rewrite——用 LLM 把问题改写为更结构化的查询语句。
  4. 索引维度：Embedding 模型是否适合当前领域？可能需要在领域数据上微调 bge-m3。
  5. 后端维度：连续多次交互中，上一轮检索到的文档可能是下一轮的高质量上下文。可以加入多跳检索或查询扩展。
  6. 评估闭环：每次调整后，用固定的评测集跑一次分数，对比优化前后的效果。

在我的项目中，解决常见的不理想情况都是优先调整 Chunk 大小和 Query Rewrite，这两招能解决 80% 的问题。”

七、知识库层级与检索策略

Q10: 知识库为什么要分层？层级检索的逻辑是什么？

难度：⭐⭐⭐（进阶）
频率：★★★★（常见）

回答结构：

“分层是为了解决检索结果的相关性和安全性问题。

做 RAG 最怕的是：Agent 从一个不相关的全局知识库里检索到错误信息，然后按照这个信息执行，导致任务失败甚至系统损坏。

我们的三层是：

层级 内容 示例
Agent 专属库 角色级专业知识 架构师的架构设计模式文档
团队公共库 团队项目文档、会议结论、规范 Sprint 目标、编码规范
全局库 通用编码规范、API 文档 Python Best Practices

检索策略是逐层补充：

  1. 先在 Agent 专属库中检索目标数量的 1.5 倍结果
  2. 如果不足目标数量，再在团队公共库中检索剩余部分
  3. 仍不够，最后检索全局库

这样保证了最相关、最权威的知识优先被注入。同时在返回结果的 metadata 中标记 source_level，方便调试和可视化。”

八、Memory 与 RAG 的协作

Q11: 当 Memory 和 RAG 信息冲突时怎么办？

难度：⭐⭐⭐⭐（深入）
频率：★★（偶见但高价值）

回答结构：

“这是一个信息冲突问题，也是 RAG+Memory 联合工作的一个核心设计决策。

我们的处理逻辑是：内部决策优先于外部知识。因为 L2 团队记忆代表了团队的显式决策，而全局知识只是一份通用文档，不能自动覆盖团队决议。

具体实现：

  • 当 Agent 组装 Prompt 时，Memory 放在 RAG 内容的前面，并且加了提示：以下信息来自团队内部决策，优先级高于外部知识库：...
  • 后续会开发基础的事实冲突检测模块，检查 Memory 和 RAG 的一致性问题，标记给用户知晓，但最终仍以记忆为准。

举个例子：团队 L2 记忆有 决策：JWT 签名用 HS256，RAG 检索到 OAuth2 使用 RS256。Agent 会看到这两者，但 System Prompt 会提示’遵从内部决策’。最终 Agent 会使用 JWT HS256，并在回答中可能提到’根据团队决策，我们使用 JWT+HS256，不同于全局文档中的 OAuth2 建议’。这样既保持了内部一致性，也让用户清楚存在外部参考。”

九、Skill 基础

Q12: Skill 文件的结构是怎样的？系统是如何解析和存储的？

难度：⭐⭐（基础）
频率：★★★（偶尔）

回答结构：

“Skill 是一个自包含的 ZIP 目录，其中至少包含两个文件：

  • SKILL.md：包含 YAML frontmatter（name, description, version）和 markdown body（instructions, output_format）
  • config.yaml：定义触发条件（如 sop_node: review）、参数定义（如 depth: quick/standard/deep）

上传后，系统：

  1. 解压 ZIP，做安全检查（文件大小、内容扫描）
  2. 解析 SKILL.md 的 frontmatter 和 body，提取指令结构
  3. 解析 config.yaml 获取参数和触发规则
  4. 存储到 skills 表中，skill_md 存原始内容，config_yaml 存配置

此时 Skill 还不会被真正执行（那是第5周的任务），只是完成注册和元数据管理。用户可以在资源中心看到所有已上传的 Skill，并查看解析状态。”

一句话总结第4周：

第4周完成了 Agent 的”知识库”——混合检索（Dense+Sparse+RRF+Reranker）让召回率达到 0.94，层级知识库保证最相关的知识优先注入，Token 预算联合裁剪让 Memory 和 RAG 在有限窗口内高效协作，为后续的多 Agent SOP 编排提供精准的知识支撑。
