AI Agent 面试实战 — 第3周：四层记忆系统与上下文管理

第3周完成 L2 反思层的核心——四层记忆系统（全局/团队/个人/上下文）、Token 预算管理、摘要压缩与硬裁剪策略。让 Agent 能”记住”团队决策和个人经验，同时不会浪费 Token。

一、记忆系统设计

Q1: 四层记忆系统是如何设计的？每层的作用是什么？

难度：⭐⭐⭐（进阶）
频率：★★★★★（必问）

回答结构：

“我设计了一个四层记忆系统，解决 Agent 协作中的记忆隔离、经验积累和上下文管理问题。

L1 系统记忆：全局通用知识，所有团队和 Agent 共享。比如 API 设计规范、安全基线。由管理员维护，任何 Agent 创建后自动继承。

L2 团队记忆：团队级共享基座。同一个 Team 内的所有 Agent 都能看到。比如架构决策、Sprint 目标、团队编码规范。关键设计是 Leader 决策自动写入 L2，所有 Member 下次对话自动同步，不需要手动传播。

L3 Agent 全局记忆：跟随 Agent 跨团队的个人经验。比如语言偏好（Python）、工具熟练度。不管 Agent 加入哪个团队，这层记忆都随身携带，解决的是’Agent 个人成长’问题。

L4 上下文记忆：Agent 在特定团队内的上下文。同一个 Agent 在不同团队，L4 是完全隔离的。比如在 DevTeam 里是后端 Coder 角色，在 DataTeam 里是数据清洗角色，互不影响。

Agent 实际可见的记忆 = L1（全局）+ L2（当前团队）+ L3（个人）+ L4（当前团队上下文）。这种分层的核心价值在于：既让团队共享关键信息（L2），又保护个人经验和隐私（L4 隔离），同时还能让 Agent 的个人能力跨团队复用（L3）。“

Q2: 为什么要设计四层，而不是三层或两层？

难度：⭐⭐⭐（进阶）
频率：★★★★（常见）

回答结构：

“如果只用两层（全局 + 会话），会出现两个问题：

  1. 团队知识污染：比如 DevTeam 的技术规范会被 HobbyTeam 读到，而 HobbyTeam 的实验性代码也可能污染 DevTeam 的上下文。
  2. Agent 无法跨团队成长：一个 Agent 在 A 团队学会的技巧，到了 B 团队全部忘光，每次都是零起点。

四层设计把’团队共享’和’个人成长’解耦了：

层级 解决什么 如果没有会怎样
L2 团队记忆 团队内部协同一致 每个 Member 都要重复学习团队规范
L3 Agent 全局记忆 个人经验全域可用 Agent 换团队后从零开始
L4 上下文记忆 团队内个人上下文隔离 A 团队的上下文污染 B 团队

同时，L2 的’Leader 写入自动广播’机制让决策传递变得高效，不需要像 LangChain 那样手动把 Memory 传来传去。”

Q3: L2 和 L4 的读写权限怎么控制？会不会出现团队记忆污染？

难度：⭐⭐⭐⭐（深入）
频率：★★★（偶尔）

回答结构：

“关键规则是：Leader 输出进 L2，Member 输出进 L4。

实现上，我在 SOP 节点配置里定义了一个 memory_write_level 字段。当 SOP Engine 执行一个节点时：

  • 如果节点角色是 leader，Agent 的输出标记为 write_level=L2
  • 如果是 member，标记为 write_level=L4

MemoryManager 的 save_memory 方法会检查这个标记：

这样，即使 Member Agent 输出了大量信息，也不会污染团队共享知识库。而 Leader 做的每个决策都会自动’广播’，所有 Member 下次对话都能看到，不需要手动同步。”

二、记忆类型与重要性

Q4: 为什么还要给记忆分类型（decision/standard/context/conclusion/warning）？

难度：⭐⭐（基础）
频率：★★★（偶尔）

回答结构：

“分类是为了精细化管理和检索。举个例子：

  • 当 Agent 被问到’这个项目的技术栈是什么’，应该优先检索 decision 类型的记忆（如’后端选 FastAPI’），而不是 context 类型的短期上下文（如’昨天改了 auth.py’）。
  • 当 Token 预算紧张需要裁剪时，warning 类型的记忆（如’禁止 root 权限’）应该被保留到最后，因为它可能关乎安全；而 context 类型的日常上下文可以优先裁剪。

五种类型覆盖了团队协作中最常见的记忆形态：

类型 内容 裁剪优先级
decision 技术选型、架构决策 保留（高）
standard 编码规范、流程标准 保留（高）
warning 安全警告、风险提示 保留到最后
conclusion 会议结论、方案共识 保留（中）
context 当前 Sprint、临时状态 优先裁剪

每种类型有一个颜色标记，在 UI 或 CLI 中呈现时一目了然。”

Q5: 记忆的重要性分数（importance）是怎么计算的？

难度：⭐⭐⭐（进阶）
频率：★★★（偶尔）

回答结构：

“重要性分数是一个加权公式：

维度 含义 计算方式
freshness（新鲜度） 越新越重要 指数衰减模型，7 天后衰减一半
relevance（相关性） 与当前任务越相关越重要 关键词匹配度
authority（权威性） 写入者越权威越重要 Leader 权重 1.0，Member 权重 0.5
frequency（引用次数） 被引用越多越重要 被 Agent 调用的次数

当 Token 预算紧张需要裁剪时，importance 最低的条目会先被丢弃。这样保证了最重要的团队决策和个人经验能被保留，临时性、低相关的内容被优先舍弃。”

三、上下文管理与 Token 预算

Q6: Context Manager 是怎么把记忆注入到 Agent 对话中的？

难度：⭐⭐⭐（进阶）
频率：★★★★（常见）

回答结构：

“Context Manager 是记忆和 Agent 之间的桥梁。在 Agent 每次对话前，它会执行一个 build_prompt_context 方法：

  1. 从 MemoryManager 拉取当前 Agent 的四层记忆视图
  2. 按配置的 Token 预算分配各部分的额度：
部分 预算占比 说明
System Prompt 10% 角色定义
Task State 10% 当前任务状态
Memory 40% L2 占一半，其它三层平分
RAG 文档 30% 知识库检索结果
输出预留 10% 给 LLM 回复留空间
  1. 如果 Memory 总量超出分配预算，按优先级裁剪：先剪 L4 中 importance 最低的 → 再剪 L3 → 然后剪 L2 早期条目 → 最后才动 L1
  2. 如果 Memory 使用率超过 70%，触发摘要压缩——把多条记忆合并为一段简短的结构化摘要
  3. 组装最终的 messages 列表，传给 Model Adapter

整个设计保证 Agent 在有限 Token 窗口内始终看到最重要、最相关的信息。”

Q7: Token 预算紧张时，压缩和裁剪的区别是什么？什么时候用哪一个？

难度：⭐⭐⭐⭐（深入）
频率：★★★（偶尔）

回答结构：

“压缩和裁剪是两个不同层级的应对策略：

摘要压缩：当 Memory 总量达到分配预算的 70% 时触发。不是丢弃记忆，而是调用一个轻量 LLM（如 GPT-4o-mini）把多条记忆合并成一段摘要。优点是信息不会完全丢失，缺点是会丢失细节。适合中等问题。

硬裁剪：当总量达到窗口的 90% 时触发。直接按 importance 排序，丢弃最低分的条目。优点是快速、确定性高；缺点是信息可能永久丢失。

两者配合：轻度超标先压缩（保留概要），严重超标再裁剪（保最重要）。

举个例子：

压缩和裁剪都记录在日志中，方便调试和优化。”

四、工程实践

Q8: 短期记忆和长期记忆怎么划分？为什么用 Redis + PostgreSQL？

难度：⭐⭐（基础）
频率：★★★（偶尔）

回答结构：

“划分的标准是生命周期。

短期记忆：当前活跃 Session 的 L4 上下文，存在 Redis 中。会话结束后，重要的条目才持久化到 PostgreSQL。读写极快，但数据可丢失。

长期记忆：L1、L2、L3 以及已持久化的 L4，存在 PostgreSQL 中。永久保留，跨 Session 可查询。

Redis + PostgreSQL 的分层方案照顾了性能和持久化：

  • 高并发下，大部分查询走 Redis，毫秒级响应
  • 长期有价值的数据写入 PostgreSQL，支持复杂 SQL 查询（如按 team_id 聚合、按时间范围搜索）
  • 如果 Redis 挂了，只丢失当前未持久化的会话上下文，核心的团队决策和系统规范都在 PostgreSQL 中安全存储”

Q9: 如果同一个 Agent 在两个团队，L2 和 L4 的记忆会互相串吗？

难度：⭐⭐（基础）
频率：★★★★（常见）

回答结构：

“不会。 这是四层记忆系统最核心的隔离性保证。

每个记忆条目在存储时都会带上 team_id 和 agent_id。MemoryManager 的 get_agent_view 方法会根据当前 team_id 过滤：

  • L2：只返回当前 team_id 的团队记忆，不会把 A 团队的技术栈决策泄露给 B 团队
  • L4：返回 agent_id + team_id 的组合键，A 团队的上下文和 B 团队的上下文完全独立
  • L3：只带 agent_id，不依赖 team_id，所以可以跨团队共享

举个例子：Coder-01 在 DevTeam 的 L4 里记了’已完成 auth 模块’，在 DataTeam 的 L4 里记了’今天做了数据清洗’。当他在 DevTeam 环境中对话时，只能看到 DevTeam 的 L4 内容，不会看到 DataTeam 的。”

五、面试应对策略

Q10: 你们的记忆系统和 LangChain 的 Memory 有什么区别？

“LangChain 的 Memory 主要是会话级、单 Agent 维度的短期记忆，默认没有多 Agent 隔离和团队共享机制。我们的四层系统在 LangChain 的基础上增加了：

  1. 团队维度的共享记忆（L2），并且自动由 Leader 决策同步
  2. 跨团队的 Agent 个人记忆（L3），让 Agent 经验可迁移
  3. 结合 Token 预算的智能裁剪和摘要压缩
  4. 与 RAG 联合调度，共享上下文窗口

LangChain 是一个通用框架，我们是针对多 Agent 协作场景做的深度定制。”

第3周面试总结

问题 核心话术 关键词
四层记忆设计？ 解耦团队共享与个人成长，L2 广播决策，L4 隔离上下文 分层架构
为什么四层不是两层？ 团队知识污染 + Agent 无法跨团队成长 解耦
L2/L4 读写权限？ Leader 输出进 L2，Member 输出进 L4 权限隔离
记忆类型系统？ 五种类型方便精细化管理和裁剪，warning 优先保留 类型标注
重要性计算？ 加权公式：新鲜度 + 相关度 + 权威 + 引用次数 算法
上下文组装？ Memory 40% 预算，裁剪顺序 L4→L3→L2→L1 Token 预算
压缩 vs 裁剪？ 70% 触发压缩（保留概要），90% 触发硬裁剪 分层策略
Redis + PG？ 短期 Redis，长期 PG，兼顾性能与持久化 存储分层
跨团队隔离？ team_id + agent_id 组合键，L4 严格隔离 多租户
vs LangChain Memory？ 团队共享 + 跨团队成长 + Token 预算 + RAG 联合 深度定制

一句话总结第3周：

第3周完成了 Agent 的”记忆”——四层记忆系统让团队决策自动同步、个人经验跨团队复用、上下文严格隔离。Token 预算管理保证在有限窗口内始终注入最重要的信息，为后续的多 Agent 编排提供上下文保障。
