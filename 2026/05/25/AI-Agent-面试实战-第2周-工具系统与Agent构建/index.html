AI Agent 面试实战 — 第2周：工具系统与 Agent 构建

第2周完成 L1 能力层核心——Tool Registry 三层权限模型、有状态工具生命周期管理、Agent Factory 组合构建。让 Agent 有了”手”和”边界”。

一、工具系统设计

Q1: 工具注册表（Tool Registry）是怎么设计的？

难度：⭐⭐（基础+）
频率：★★★★★（必问）

回答结构：

“Tool Registry 是整个工具系统的中枢，负责工具的注册、发现、权限校验。

设计上分三层：

第一层：注册层。 所有工具在 Registry 中注册，包括 name、description、参数 Schema（JSON Schema 格式）、工具类型（function / mcp / api）、是否是有状态工具。这个是全局唯一的，所有 Agent 都能看到这个注册表。

第二层：声明层。 每个 Persona 在创建时，声明自己需要哪些工具。比如架构师声明自己需要 file-ops-readonly（只读文件操作），程序员声明自己需要 file-ops + terminal（完整文件操作+终端执行）。

第三层：过滤层。 每个 Team 在创建时，配置一个 allowedTools 白名单。运行时，Agent 实际可用工具 = Persona 声明 ∩ Team 白名单。

举个例子：架构师声明了 file-ops，但 Team 白名单只允许 file-ops-readonly 和 git。那么架构师在这个 Team 中只能使用 file-ops-readonly，即使他想写文件也做不到——因为工具层就不支持。”

追问：为什么不在数据库层做外键约束？

“工具和 Persona 的关系是多对多，按规范应该建个 persona_tools 关联表。但我们选择在 Persona 表中存 tools_declared 的 TEXT 数组，是因为：

  • 一个 Persona 通常只声明 3-8 个工具，数组完全够用
  • 运行时权限计算是在内存中取交集（set(persona_tools) & set(team_tools)），不需要数据库做 JOIN
  • 这种设计更灵活——Persona 的声明改了，不需要迁移关联表”

Q2: Function Calling 的工具 Schema 怎么设计的？

难度：⭐⭐（基础）
频率：★★★★（常见）

回答结构：

“工具 Schema 遵循 OpenAI Function Calling 规范，也是 MCP 协议的基础格式：

这里的关键设计点是：

  1. description 字段是给 LLM 看的，不是给人看的。LLM 通过这个字段理解工具什么时候该调用。所以 description 必须清晰、具体——‘文件读写操作’比’操作文件’好，因为它暗示了不止一个操作。
  2. enum 限制操作范围。 不让 LLM 自由发挥，所有操作都预定义好，避免 LLM 产生幻觉（比如自己发明一个 delete 操作）。
  3. required 字段告诉 LLM 哪些参数必须提供，减少无效调用。

工具执行器（Tool Executor）会根据这个 Schema 做参数校验，验证通过后才真正执行工具。”

追问：怎么防止 LLM 调用不该调用的工具？

“三层防护：

  1. Prompt 层——System Prompt 中说明 Agent 的能力边界
  2. Schema 层——没有在 Function Calling 中传给 LLM 的工具，LLM 根本不知道存在
  3. 工具层——即使 LLM 构造了调用请求，Tool Executor 会根据当前 Team 白名单做最终校验，越权直接拒绝”

Q3: 有状态工具和无状态工具有什么区别？生命周期怎么管理？

难度：⭐⭐⭐（进阶）
频率：★★★★（常见）

回答结构：

“这是工具系统设计中最重要的区分。

无状态工具（如 file-ops）每次调用都是独立的，不保留调用之间的任何状态。读文件就是读文件，读完就结束，下次读还是从磁盘重新读。这类工具随用随调，不需要特殊管理。

有状态工具（如 Terminal）持有长连接、进程、环境变量等持久化状态。比如 Terminal 工具会启动一个 bash 进程，Agent 第一次执行 cd /project 后，第二次执行 ls，是在 /project 目录下列文件。这个工作目录的状态是持久的。

生命周期管理：

关键设计点：

  1. 绑定 Session。 有状态工具实例不跟着 Agent 走，是跟着 Session 走的。Agent 可以被释放回 Pool，但它的 Session 需要保留着 Terminal 的状态。
  2. GC 回收。 Session 正常销毁时，同步清理所有有状态工具资源。Session 异常退出时，通过心跳检测超时未活跃的 Session，自动触发 GC 清理（关闭进程、释放端口、删除临时文件）。
  3. 资源泄露防护。 每次创建有状态工具实例时，注册到资源表（Resource Table）。定期扫描孤儿资源——比如 Session 已销毁但 Terminal 进程还在跑的，自动回收。”

追问：为什么不让 Agent 绑定有状态工具？

“如果 Agent 绑定 Terminal，那这个 Agent 就不能被释放回 Pool 了——它的 Terminal 会话还存着，如果被另一个 Team 拿去用，就会串状态。所以有状态工具绑定的是 Session，不是 Agent。Agent 用完了可以释放，但 Session 保留着当前的工具状态，下一个分配到同一个 Session 的 Agent（可能是同一个）可以接着用。”

二、Agent 系统设计

Q4: Agent 是怎么创建的？Agent 和 Model 的关系？

难度：⭐⭐（基础）
频率：★★★★★（必问）

回答结构：

“Agent 的创建过程就是一个组合过程，公式是：Agent = Default Model + Persona + Tools。

具体流程：

  1. 加载 Persona——读取角色定义（system_prompt、capabilities、tools_declared、output_prefs）
  2. 加载 Model——读取 Model 配置（provider、model_name、api_key），创建 LLMConfig
  3. 解析 Tools——根据 Persona 声明的工具和创建时传入的 tool_ids，取交集，确定 Agent 的可用工具列表
  4. 组合为 Agent 实例——将以上三者组合，创建 Agent 对象，状态为 idle
  5. 注册到 Agent Pool——进入资源池，等待分配

关于 Agent 和 Model 的关系：Agent ≠ Model，Model 可以随时切换。

  • Agent 是一个角色实例——架构师、程序员、审查员
  • Model 是一个推理引擎——GPT-4o、Claude Sonnet、GLM-5.1
  • 同一个 Agent（架构师）可以用 GPT-4o 驱动，也可以切换到 Claude
  • 同一个 Model（GPT-4o）可以驱动不同 Agent（架构师、程序员）

这种解耦的好处是：当 OpenAI API 限流时，可以把某个 Agent 的 Model 临时切换到 GLM，业务逻辑完全不变。Model 只是一个配置项。”

追问：运行时要怎么切换 Model？

“Agent 对象上有一个 override_model(new_model_id) 方法。调用后，Agent 的 llm_config 立即更新，下一次对话就使用新模型。这个过程不需要重新创建 Agent，不影响 Agent 的 Persona 和 Tools。并且切换动作会记录到 Observer 日志，方便后续成本归因——比如’架构师从 GPT-4o 切换到 GLM-5.1 是因为费用超预算’。”

Q5: Agent Pool 的分配策略是什么？

难度：⭐⭐⭐（进阶）
频率：★★★★（常见）

回答结构：

“Agent Pool 是一个能力匹配的资源池，核心方法是 acquire(required_capabilities)。

分配流程：

  1. 定义需求——SOP 节点指定需要什么能力的 Agent（如 {"coding": 0.8, "python": 0.6}）
  2. 过滤 idle Agent——从 Pool 中找出所有空闲状态的 Agent
  3. 能力匹配——逐一检查 Agent 的 Persona 中定义的 capabilities 是否覆盖需求的每一项
  4. 最优选择——如果有多个符合条件的 Agent，优先选择有该 Team 上下文记忆的 Agent，因为可以减少学习成本
  5. 分配锁定——将 Agent 状态置为 busy，加入 busy 集合

关键约束：

  • 独占性——一个 Agent 同一时刻最多归属一个 Team
  • 不可抢占（MVP 阶段）——一旦分配，不会中途抢走
  • 能力超配允许——capabilities = {"coding": 0.9} 可以匹配 {"coding": 0.7} 的需求

回收流程：任务完成后，调用 release(agent_id) 将 Agent 状态置回 idle，释放到 Pool 中供下一个任务使用。”

追问：如果能力不够怎么办？

“如果 Pool 中所有 Agent 都不满足能力要求，系统返回 None 并通知 SOP Engine。SOP Engine 会暂停当前任务，等待人工介入——用户可以手动指定一个 Agent 或者降低能力要求。这是 HITL 的一个早期应用。”

Q6: Agent 是怎么执行对话的？System Prompt 怎么组装？

难度：⭐⭐（基础）
频率：★★★★（常见）

回答结构：

“Agent 对话的执行流程分三步：

第一步：组装 System Prompt。 把 Persona 的 system_prompt（角色定义）、Tools 的描述（告诉模型有什么工具可用）、当前上下文组装成一个完整的 System Prompt。

第二步：构建 Function Calling 定义。 把 Tools 的参数 Schema 转换成 OpenAI Function Calling 格式，传给 Model Adapter。这告诉 LLM 哪些函数可以调用、需要什么参数。

第三步：调用 LLM 并处理返回。 LLM 返回后，如果包含 tool_calls，Tool Executor 根据 tool_name 路由到对应的工具实例，执行工具，拿到结果，再作为新的对话上下文传给 LLM，最终输出回答。

流式模式下，LLM 逐字返回内容，Tool Executor 拦截到 Function Call 后暂停流式输出，执行工具，然后再继续流式。”

追问：如果 LLM 返回了无效的 Function Call 怎么办？

“Tool Executor 会做参数校验——用 JSON Schema 验证参数格式、检查必填字段、校验枚举值。校验失败时，返回错误信息给 LLM，让 LLM 修正。如果连续 2 次失败，取消 Function Call，改用文本模式回答。这是 Feedback Loop 的雏形。”

三、安全与设计权衡

Q7: 怎么防止 Agent 越权调用工具？

难度：⭐⭐（基础）
频率：★★★（偶尔）

回答结构：

“三层防护：

第一层：Prompt 边界。 System Prompt 中明确 Agent 的能力边界——‘架构师只出方案不写代码’。但这只是软约束，LLM 可能不遵守。

第二层：可用工具最小化。 Agent 在创建时，只会加载 Persona 声明 ∩ Team 白名单的工具。架构师根本不会注册 file-ops 的 write 功能，所以即使 LLM 想写文件，它也不知道有这个工具存在。

第三层：工具执行前校验。 即使 LLM 构造了一个调用请求，Tool Executor 在真正执行前会检查这个 Agent 的可用工具列表。不在列表中直接拒绝，返回 ‘don’t have permission to use tool X’。

举个例子：架构师想修改代码——第一步，Prompt 说’你不应该写代码’；第二步，架构师的 Persona 声明只包含 file-ops-readonly；第三步，即使架构师强行尝试调用 write_file，Tool Executor 检查权限后拒绝。

这种方法的特点是不依赖单一防线。任何一层被绕过（比如 LLM 不遵守 Prompt），后面还有两层硬性检测。而且越权检测是通过代码实现的，不是 Prompt 实现的，确定性更高。”

追问：怎么检测 Prompt Injection 攻击？

“当前的简单防护是路径安全检测——file-ops 工具禁止访问 .env、id_rsa、.ssh/ 等敏感文件。完整的 Prompt Injection 三层防线会在后续 Phase 实现：输入清洗 → 行为边界检测 → 敏感操作确认。目前的 MVP 阶段先做基础防护。”

四、面试高频追问应对

追问：为什么不直接用 LangChain Agent？

“LangChain Agent 是单 Agent 全能模式——一个 Agent 拥有所有工具、在单个上下文中完成所有工作。这类似于给一个人所有公司的权限，让他自己决定做什么。

我们的场景是多 Agent 协作——三个不同角色的 Agent 按 SOP 分工：架构师只做设计、程序员只写代码、审查员只做审查。如果用 LangChain Agent，得把所有工具都装在一个 Agent 里，然后靠 Prompt 约束它’不要越界’——这很不靠谱。

所以我们的做法是：底层用 LiteLLM 做模型统一调用（类似 LangChain 的 Model 层），上层用 LangGraph 做多 Agent 协作的 SOP 编排（类似 LangChain 的 Agent 层，但更灵活可控）。技术上没有严格的好坏，是场景驱动的选择。”

追问：为什么 Agent 不用 gRPC 通信？

“先说 MVP 是什么——MVP = Minimum Viable Product（最小可行产品），就是先做最核心的功能跑通，再逐步扩展。The Boy Assistant 最终要支持 4 个团队、16 个模块，但 MVP 阶段只做开发助手团队跑通一个 SOP。如果一开始就搞 gRPC、微服务、分布式，可能 2 个月还在搭基础设施，一行业务逻辑没写。

MVP 阶段用内存调用和 REST API 通信。gRPC 的优势（性能、强类型、流式）在 Agent 通信场景中不是第一需求，原因是瓶颈不在通信，而在 LLM 推理：

Agent 之间的通信通过 Blackboard（Redis Pub/Sub）和直接调用完成。REST 传输 1ms，gRPC 0.5ms，省了 0.5ms 但总共还是 3 秒多，完全没有意义。

只有在 Agent 数量超过 100 个、每秒通信超过 1000 次时，才需要考虑 gRPC。那是生产级扩展的问题，不是 MVP 阶段该考虑的。遵循原则：先把业务跑通，性能问题等真遇到再说。“

追问：三层权限模型和有状态工具是什么关系？

“这两个是完全不同的维度，经常被混淆：

  • 三层权限模型解决的是’谁能用什么工具’——是权限问题
  • 有状态/无状态解决的是’工具用完要不要清理’——是生命周期问题

生活类比：

概念 解决什么 类比
三层权限 谁能用什么工具 公司门禁卡（你能刷哪些门）
有状态/无状态 工具本身有没有记忆 一次性纸杯 vs 要洗的杯子
生命周期管理 用完必须回收，不然资源泄露 用完关灯、锁门

再看无状态工具和有状态工具的区别：

无状态工具（file-ops）——每次调用完全独立：

有状态工具（Terminal）——维护持续的状态：

四次调用共享同一个 bash 进程。这就是’有状态’——工具内部维护了持续存在的状态。

为什么必须管理生命周期？ 因为有状态工具持有资源（进程、端口、连接），用完不关就会泄露：

所以绑定 Session 生命周期：Session 创建时启动，Session 销毁时 GC 回收关闭。面试时这两个问题经常连在一起问，但回答时一定要分开——权限控制是’能不能用’，生命周期是’用完怎么回收’。“

第2周面试总结

问题 核心话术 关键词
Tool Registry 设计？ 三层权限模型：注册→声明→过滤 RBAC、Function Calling Schema
Function Calling Schema？ description 给 LLM 看 + enum 限制 + required 校验 参数校验、幻觉防护
有状态 vs 无状态？ 绑定 Session 生命周期，GC 回收 资源泄露、心跳检测
Agent 创建？ Agent = Model + Persona + Tools 解耦、运行时切换
Agent Pool 分配？ 能力匹配 + 独占性 + 回收机制 资源池、状态机
Agent 对话执行？ System Prompt 组装 + Function Calling + 工具执行 三层防护
越权防护？ Prompt + Schema + 工具执行前校验 多层防护
MVP vs gRPC？ LLM 推理 2-10s，通信 <1ms，优化通信无意义 先跑通再优化
权限 vs 有状态？ 权限=能不能用，有状态=用完怎么回收 两个独立维度
