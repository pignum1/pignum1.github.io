AI Agent 面试实战 — 第1周：架构设计与工程基础

边做 The Boy Assistant 项目，边整理高频面试题。第1周完成系统地基——从零搭建 FastAPI 项目、Model Adapter、API Key 管理、数据表设计。以下是这一周遇到的高频面试问题。

一、架构设计类

Q1: 为什么选 FastAPI 而不是 Django 或 Flask？

难度：⭐（基础）
频率：★★★★★（几乎必问）

回答结构：

“选 FastAPI 是基于三个核心考量：**

第一：async 原生支持。 Agent 系统需要大量并发 API 调用——同时调用多个 LLM Provider、同时处理多个 Agent 的请求、同时推送 WebSocket 流式输出。FastAPI 原生支持 async/await，而 Django 的 async 支持是后来加的，稳定性不够；Flask 根本没有原生 async。

第二：WebSocket 内置支持。 我们的聊天室需要实时推送 SOP 执行进度、Agent 对话流。FastAPI 的 WebSocket 是内置的，同一个框架同时处理 REST API 和 WebSocket。如果用 Django，需要额外配置 Django Channels。

第三：OpenAPI 自动生成。 前后端分离开发，前端的 React 页面需要清晰的 API 文档。FastAPI 自动生成 OpenAPI 规范，前端可以直接用 Swagger UI 调试接口，减少沟通成本。

追问：性能上差距有多大？

“我做过一个简单的压测对比：在同样 100 并发下，FastAPI 的吞吐量是 Flask 的 3 倍左右，和 Django 差距不大。但 Django 的 async 支持是后来加入的（Django 3.1+），很多第三方库还是同步的，容易出现调用链上的阻塞。FastAPI 从第一天就是 async 设计的，生态更纯粹。”

Q2: Model Adapter 为什么用适配器模式？

难度：⭐⭐（基础+）
频率：★★★★★（必问）

回答结构：

“适配器模式解决的是多 Provider API 差异的问题。每个 LLM Provider 的 API 格式、鉴权方式、错误码、参数名都不一样——OpenAI 叫 max_tokens，Anthropic 叫 max_tokens_to_sample；OpenAI 用 HTTP 状态码表示错误，Anthropic 用自定义错误类型。

适配器模式的做法是：定义一个抽象基类 BaseLLMAdapter，声明统一接口（chat、chat_stream、check_health）。每个 Provider 实现这个接口，把自家 API 的差异封装在实现类内部。上游代码只依赖抽象基类，不依赖具体实现。

好处有三点：

  1. 新增 Provider 成本低——添加新的 Model，只需要写一个新的 Adapter 类实现，不需要修改任何现有代码
  2. 切换模型无感——同一个 Agent 从 GPT-4o 切换到 Claude，只是改一个配置项，不需要改业务逻辑
  3. 测试方便——可以写一个 MockLLMAdapter，开发时用 Mock 模式，不消耗真实 API 费用

追问：LiteLLM 已经是统一封装了，为什么还要自己写 Adapter？

“LiteLLM 解决了 80% 的问题——统一了调用接口、支持 100+ Provider、内置 Retry 和 Fallback。但我需要的是更加可控的抽象：

  • 自己的 Response 数据结构。 LiteLLM 返回的是它自己的格式，我需要在 LLMResponse 里统一 latency、usage、model 等字段，方便后续的成本统计和链路追踪。
  • 自定义 Fallback 策略。 LiteLLM 的 Fallback 是按顺序重试，我需要在 Fallback 之间插入日志记录、告警通知、成本计算。
  • Mock 模式。 开发环境我需要一个完全不调用真实 API 的 Adapter，LiteLLM 没有这个功能。

所以我们是在 LiteLLM 外面再包一层自己的 Adapter——LiteLLM 负责’能调通 100 个 Provider’，我们的 Adapter 负责’统一数据结构和扩展能力’。”

Q3: 表结构设计的时候，为什么用 JSONB 存能力和工具列表？

难度：⭐⭐（基础）
频率：★★★★（常见）

回答结构：

“这是 Schema-on-Read vs Schema-on-Write 的权衡。Persona 的 capabilities 和能力评分是动态的——今天可能只有 coding 和 analysis，下周可能增加 design 和 communication。如果用关系模型（建一个 persona_capabilities 表），每次新增能力维度都要改表结构，加 migration。

JSONB 的好处是：

  1. 灵活——Persona A 可以只有 2 个能力维度，Persona B 可以有 10 个，不需要统一
  2. 支持索引——PostgreSQL 的 JSONB 支持 GIN 索引，可以查询 capabilities @> '{"coding": 0.9}'
  3. 与代码模型天然对应——Python 的 dict 直接序列化到 JSONB，不需要 ORM 映射

同样道理，tools_declared 用 TEXT[] 数组而不是关联表，是因为 Persona 和 Tool 的关系是多对多，但在运行时我们会在内存中做完整校验（Persona 声明 ∩ Team 白名单），不需要在数据库层面做引用完整性约束。

追问：JSONB 性能有问题吗？

“在我们预期的数据量下（不超过 10 万条 Persona），JSONB 的查询性能完全够用。capabilities @> '{"coding": 0.9}' 的查询走 GIN 索引，毫秒级返回。只有到了百万级且需要复杂 JSON 路径查询时，才需要考虑拆表。这属于过早优化的范畴，我们会等到真正遇到性能瓶颈时再优化。”

二、工程实践类

Q4: API Key 怎么保证安全？

难度：⭐⭐（基础）
频率：★★★★（常见）

回答结构：

“API Key 安全做了四层防护：

第一层：加密存储。 使用 cryptography.fernet 对称加密。加密密钥从 .env 的环境变量读取，不硬编码在代码里。每个 Provider 的 Key 独立加密，解密密钥是唯一的。

第二层：日志脱敏。 所有日志输出时，自动屏蔽 Key 的中间部分。比如 sk-proj-abc123...xyz789，只保留前 8 位和后 4 位，中间用 ... 代替。防止日志泄露。

第三层：不回传明文。 创建 API Key 的 API 只返回是否创建成功，不回传明文。用户在 UI 上输入 Key 后，系统立即加密存储，之后再也看不到明文。

第四层：连通性测试。 每个 Key 创建时都调用一次 check_health 接口，验证 Key 确实有效。无效 Key 直接拒绝存储，不给用户留下’好像配了又好像没配’的疑惑。

追问：如果加密密钥泄露了怎么办？

“密钥安全是防御的最后一环。我们的做法是：加密密钥使用环境变量管理，不在任何配置文件或代码仓库中存储。生产环境的密钥通过密钥管理服务（如 AWS KMS 或 HashiCorp Vault）分发，不在开发人员的本地环境中。

如果真泄露了，应对流程是：立即轮换加密密钥 → 重新加密所有存储的 API Key → 通知用户重新配置 Key。但因为 Key 本身也加密了，攻击者即使拿到加密密钥，还需要访问数据库才能解密，增加了攻击成本。”

Q5: 开发时怎么避免消耗真实的 API 费用？

难度：⭐（基础）
频率：★★★（偶尔）

回答结构：

“开发环境用 MockLLMAdapter。它继承自同一个抽象基类 BaseLLMAdapter，返回固定格式的模拟输出——[Mock] 你说了: {message}，当前模型：{model}。不调用任何真实 API，零成本。

切换方式很简单：在配置中 MOCK_MODE=true 时，依赖注入 MockLLMAdapter；MOCK_MODE=false 时注入 LiteLLMAdapter。代码里只有一个配置项的变化，业务逻辑完全不变。

而且这种设计还有一个额外好处：单元测试稳定。用 MockLLMAdapter 写测试，不依赖外部 API，测试不会因为网络波动或 API 限流而失败。只有集成测试才切换到真实 Provider。

追问：Mock 和真实环境差异大怎么办？

“这是一个真实存在的风险——Mock 返回的是固定格式的理想输出，真实模型可能返回不同的格式。我们的应对是两层测试：

  • 单元测试用 Mock——验证业务逻辑正确性（比如条件路由、状态转移）
  • 集成测试用真实 Provider——每天跑一次，验证真实效果差异

而且我们的 Validation Gate 会在真实环境中做输出格式校验，如果真实模型返回的格式不符合预期，会自动触发 Feedback Loop 重试。所以即使有差异，系统也能自愈。”

三、决策类

Q6: Agent 的 tools 字段为什么存 UUID 数组而不是关联表？

难度：⭐⭐（基础）
频率：★★★（偶尔）

回答结构：

“这是性能考量的结果。Agent 和 Tool 是多对多关系，按规范做法应该建一个 agent_tools 关联表。但我查了典型的使用场景——一个 Agent 通常只绑定 3-8 个工具（Coder 用 file-ops + git + terminal，Reviewer 用 file-ops-readonly），不会超过 20 个。

用 UUID[] 数组的好处是：

  1. 查询少一次 JOIN。 取 Agent 时直接拿到工具 ID 列表，不需要 JOIN agent_tools
  2. 更新快。 给 Agent 添加/移除工具，只需要更新一个字段，不用操作关联表
  3. 序列化方便。 ORM 映射后直接是 Python list，不需要额外处理

只有当一个 Agent 的工具数量可能超过 50 个，或者工具关联的元数据（如权限级别、使用频率）足够复杂时，才有必要拆关联表。

追问：数组的引用完整性怎么保证？

“PostgreSQL 的 UUID[] 没有外键约束，不能自动级联删除。所以我们在删除 Tool 时，手动检查所有 Agent 的 tools 字段是否引用了这个 Tool。如果有，拒绝删除或自动从 Agent 移除。

这是在业务层保证的，不是在数据库层。选择数据库层灵活（数组），业务层做校验。这是权衡后的选择——为了性能牺牲一点数据层的严谨性，但在业务层补回来。”

Q7: 预置的 System Prompt 怎么设计的？

难度：⭐⭐（进阶）
频率：★★★★（常见）

回答结构：

“以’高级架构师’这个 Persona 为例，System Prompt 的结构是：角色定义 + 能力边界 + 行为规则 + 输出格式。

核心设计原则是先说角色，再说边界——先告诉模型它是什么角色（高级架构师），再告诉它不能做什么（不写具体代码，只出设计方案），最后告诉它怎么输出（Markdown 格式，包含方案选型理由）。

具体内容上，我们强调三个点：

  1. 明确角色——‘你是一个有 15 年经验的软件架构师，擅长系统设计和技术选型’
  2. 设定边界——‘你只负责架构方案设计，不生成具体代码实现。如果被问到编码问题，礼貌地拒绝并建议咨询程序员 Agent’
  3. 输出规范——‘你的输出必须包含：问题分析、方案对比、推荐方案及理由、实施计划，格式为 Markdown’

追问：Prompt 怎么保证 Agent 不越界？

“一个 Prompt 不可能 100% 保证 Agent 不越界，所以我们有三层防护：

  1. Prompt 层——System Prompt 中明确角色边界
  2. Tool 层——Tool Registry 的三层权限模型（Persona 声明 ∩ Team 白名单），架构师没有 write_file 工具，所以即使想写代码也写不了
  3. Validation 层——Validation Gate 会校验 Agent 输出是否符合期望格式

System Prompt 是第一道防线，但不是唯一防线。越界检测靠的是多层防护，不依赖单一手段。”

四、追问应对

追问：Adapter 模式的性能开销？

“适配器模式有一个间接调用层的开销，但这个开销可以忽略不计——每次 LLM 调用本身耗时 2-10 秒，Adapter 的函数调用开销是纳秒级的，完全不是瓶颈。

真正影响性能的是网络 I/O 和模型推理时间，不是代码架构的抽象层级。为了代码的可维护性（新增一个 Provider 只需要写一个类），这点性能损失完全值得。”

追问：数据库为什么选 PostgreSQL 而不是 MySQL？

“选 PostgreSQL 有三个原因：

  1. pgvector 扩展——RAG 需要向量检索，PostgreSQL 有 pgvector 可以直接在库里做，不需要额外部署 Milvus
  2. JSONB 支持——Persona 的 capabilities、SOP 的 nodes/edges 都需要灵活的结构化存储
  3. 异步驱动成熟——asyncpg 性能稳定，配合 SQLAlchemy 2.0 async 模式，与 FastAPI 的 async 原生匹配

如果用 MySQL，需要额外部署一个向量数据库来处理 RAG，增加运维复杂度。”

第1周面试总结

问题 核心话术 关键词
为什么选 FastAPI？ async 原生 + WebSocket + OpenAPI 并发、流式、文档
适配器模式的好处？ 统一接口 + 新增成本低 + Mock 方便 抽象、扩展、测试
JSONB 为什么？ 灵活 + 索引 + 代码对应 Schema-on-Read
API Key 安全？ 加密存储 + 日志脱敏 + 不回传 + 验证 四层防护
Mock 模式？ 配置切换 + 零成本 + 测试稳定 依赖注入
UUID 数组 vs 关联表？ 少 JOIN + 快更新 + 序列化方便 性能权衡
System Prompt 设计？ 角色定义 + 能力边界 + 行为规则 + 输出格式 三层防护
PostgreSQL vs MySQL？ pgvector + JSONB + asyncpg 一站式存储

一句话总结第1周：

第1周完成了系统地基——从零搭建了 FastAPI 项目、LiteLLM 适配器模式封装的 Model Adapter、加密安全的 API Key 管理、灵活的数据表设计。让 Agent 能说话、能调用工具，为后续的多 Agent 协作打下基础。
