AI-Python-08.1: Agent 设计模式全景

构建 Agent 不是选一个框架然后开始写代码——而是先理解设计模式，再决定架构。本文系统梳理 Agent 的 推理、规划、反思、协作 四大类设计模式，建立完整的模式选型思维框架。

1. 为什么需要设计模式？

1.1 同一个任务，不同模式天差地别

同样的任务，LLM 调用次数可以从 2 次到 10+ 次，成本差 5 倍。

1.2 Andrew Ng 的四大模式

Andrew Ng（吴恩达）将 Agent 设计模式归纳为四大类：

    end

    A --> |"做完了检查"| C
    B --> |"有工具可用"| C
    C --> |"复杂到需要分工"| D

这四类不是孤立的——生产系统是它们的组合。

2. 从四大类到五层架构

吴恩达的四大类是宏观概念框架，本文在其基础上做实现层面的细化拆分——把 Ng 的 “Planning” 拆成”推理”（怎么想）和”规划”（怎么安排步骤），因为 CoT/ToT 和 Plan-and-Execute 解决的是不同维度的问题。

为什么要拆？”Planning” 包含了两种截然不同的能力：

推理（怎么想） 规划（怎么安排）
关注点 单步内的思维过程 多步之间的编排策略
代表模式 CoT、ToT、Self-Ask Plan-and-Execute、ReWOO、LATS
核心区别 不涉及工具，纯思维链 决定工具调用的顺序和策略

2.2 五层总览

层级 对应 Ng 的类别 职责 核心问题 典型模式
Layer 1 推理 Planning 怎么想 如何分解问题、组织思路？ CoT、Self-Ask、ToT
Layer 2 行动 Tool Use 怎么做 如何与外部世界交互？ Function Calling、ReAct、CodeAct
Layer 3 规划 Planning 怎么安排 多步任务如何编排？ Plan-and-Execute、ReWOO、LATS
Layer 4 反思 Reflection 怎么改进 做完了如何验证和修正？ Reflexion、Self-Critic
Layer 5 协作 Multi-Agent 怎么分工 多个 Agent 如何配合？ Supervisor、Swarm、Debate

原则：从低层到高层组合使用，不是越复杂越好。

接下来的第 3-7 节，按 Layer 1 → Layer 5 逐层展开每种模式的详细设计。

3. Layer 1 — 推理模式

3.1 Chain of Thought（思维链）

让 LLM 分步推理，而不是直接给答案。

两种触发方式：

方式 做法 适用场景
Zero-shot CoT 追加 “请一步步思考” 通用场景
Few-shot CoT 给出推理示例 高精度场景
Self-Consistency 采样多条推理路径，投票选多数 数学、逻辑推理

效果：在复杂推理任务上可提升高达 61 个百分点。

3.2 Self-Ask（自问自答）

模型自己提出子问题，逐个回答，最终合成答案。

适用场景： 多跳推理（Multi-hop Reasoning），需要串联多个信息源。

3.3 Tree of Thoughts（思维树）

CoT 是单条路径，ToT 是多分支探索——像下棋一样，同时考虑多种走法。

    flowchart TB
    Q[问题: 5个数字排最大的数] --> T1[思路1: 从大到小排列]
    Q --> T2[思路2: 贪心策略]
    Q --> T3[思路3: 全排列比较]

    T1 --> S1["85321 ✅ 评分:9"]
    T2 --> S2["85321 ✅ 评分:9"]
    T3 --> S3["计算量大\n但一定正确 评分:7"]

    S1 --> BEST[最优路径: 思路1]
    S2 --> BEST

    style Q fill:#ffd1d1,stroke:#e57373,stroke-width:2px,color:#333
    style BEST fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333

三种推理模式对比：

维度 CoT Self-Ask ToT
路径 单条线性 单条阶梯式 多分支树状
探索性 低 中 高
Token 消耗 低 中 很高
适用场景 常规推理 多跳问答 规划、创意、博弈

4. Layer 2 — 行动模式

4.1 Function Calling（函数调用）

最基础的行动模式——LLM 输出结构化的函数调用指令，由代码执行。

关键：LLM 自己不执行任何操作，它只告诉你”该调什么、传什么参数”。

4.2 ReAct（推理 + 行动）

将 CoT 推理与工具调用结合，形成 “思考→行动→观察” 循环。

    flowchart TB
    S[用户提问] --> T1["Thought\n用户想知道腾讯股价\n我需要搜索"]
    T1 --> A1["Action\nsearch('腾讯股价')"]
    A1 --> O1["Observation\n380.5 港元"]
    O1 --> T2["Thought\n得到了答案\n可以回复了"]
    T2 --> F["Final Answer\n腾讯当前股价 380.5 港元"]

    style T1 fill:#d1e8ff,stroke:#42a5f5,stroke-width:2px,color:#333
    style T2 fill:#d1e8ff,stroke:#42a5f5,stroke-width:2px,color:#333
    style A1 fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333
    style O1 fill:#fff3d1,stroke:#ffb74d,stroke-width:2px,color:#333
    style F fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333

ReAct 的优缺点：

维度 评价
灵活性 最高——每步根据上一步结果调整
调试友好 每一步 Thought 都可观察
Token 成本 每步一次 LLM 调用，线性增长
风险 超过 5-7 步后成本和延迟显著上升；可能陷入循环

4.3 CodeAct（代码即行动）

最新模式——Agent 通过编写和执行代码来完成任务，而不是调用预定义工具。

    flowchart TB
    S[用户: 分析这份 CSV 数据] --> R["Reason\n理解需求"]
    R --> C["Code\n生成 Python 分析代码"]
    C --> E["Execute\n沙箱中执行代码"]
    E --> O["Observe\n查看执行结果"]
    O --> |"成功"| F["输出结果"]
    O --> |"报错"| D["Debug\n修改代码"]
    D --> E

    style C fill:#d1e8ff,stroke:#42a5f5,stroke-width:2px,color:#333
    style E fill:#fff3d1,stroke:#ffb74d,stroke-width:2px,color:#333
    style D fill:#ffd1d1,stroke:#e57373,stroke-width:2px,color:#333

ReAct vs CodeAct：

维度 ReAct CodeAct
行动方式 调用预定义工具函数 编写任意代码执行
灵活性 受限于工具集 理论无限
安全性 高（工具可控） 需要沙箱隔离
适用场景 明确的 API 调用 数据分析、复杂计算
代表产品 LangChain Agent Claude Code、Devin

5. Layer 3 — 规划模式

5.1 Plan-and-Execute（先规划再执行）

将 “规划” 和 “执行” 分离为两个角色。

    flowchart TB
    S[START] --> P["Planner\n规划全部步骤"]
    P --> |"步骤列表"| E1["Executor\n执行步骤1"]
    E1 --> C1{"还有步骤?"}
    C1 -->|"是"| E2["Executor\n执行步骤2"]
    E2 --> C2{"还有步骤?"}
    C2 -->|"否"| SUM["Summarizer\n汇总报告"]
    C1 -->|"否"| SUM
    SUM --> E[END]

    style P fill:#ffd1d1,stroke:#e57373,stroke-width:2px,color:#333
    style E1 fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333
    style E2 fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333

优势： 10 步任务 ReAct 需要 10+ 次 LLM 调用，Plan-and-Execute 只需 1-3 次。

风险： “计划漂移”（Plan Drift）——执行过程中现实变化，原计划失效。需要 Replan 机制。

5.2 ReWOO（无观察推理）

一次规划所有工具调用，并行执行，只调 2 次 LLM。

    flowchart TB
    S[START] --> P["Planner\n一次规划全部调用"]
    P --> |"#E1 = Search['澳网冠军']\n#E2 = Search['#E1 的家乡']"| W["Worker\n并行执行所有工具"]
    W --> |"结果1, 结果2, ..."| SOL["Solver\n一次汇总答案"]
    SOL --> E[END]

    style P fill:#ffd1d1,stroke:#e57373,stroke-width:2px,color:#333
    style W fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333
    style SOL fill:#d1e8ff,stroke:#42a5f5,stroke-width:2px,color:#333

三种规划模式 Token 效率对比：

模式 10步任务的 LLM 调用 能中途调整吗 并行性
ReAct 10+ 次 能（每步灵活） 不能
Plan-and-Execute 3-5 次 能（可 Replan） 部分
ReWOO 2 次 不能（计划固定） 完全并行

ReWOO 的致命弱点： 零中途适应能力。如果步骤 2 的结果意味着步骤 4 应该改变，ReWOO 无法感知。

5.3 LATS（语言智能体树搜索）

将蒙特卡洛树搜索（MCTS）应用于 Agent——像 AlphaGo 一样思考。

    flowchart TB
    S["Select\n选择最有潜力的节点"] --> E["Expand\n展开子节点"]
    E --> V["Evaluate\n评估子节点价值"]
    V --> B["Backpropagate\n向上更新祖先价值"]
    B --> |"重复N次"| S
    S --> |"预算用完"| F["Extract\n沿最高价值路径提取结果"]

    style S fill:#d1e8ff,stroke:#42a5f5,stroke-width:2px,color:#333
    style E fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333
    style V fill:#fff3d1,stroke:#ffb74d,stroke-width:2px,color:#333
    style B fill:#ffd1d1,stroke:#e57373,stroke-width:2px,color:#333
    style F fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333

LATS 的代价与回报：

维度 评价
质量 最高——系统性地探索多种可能
成本 极高——10 次迭代 × 3 展开 = 40+ 次 LLM 调用
适用场景 博弈、复杂规划、多方案比选

6. Layer 4 — 反思模式

6.1 Reflexion（自我反思）

做完之后自我检查，不合格就重做。

    flowchart TB
    S[START] --> G["Generator\n生成代码"]
    G --> R["Reviewer\n审查代码"]
    R --> C{"审查通过?"}
    C -->|"FAIL"| RE["记录反思\n存入记忆"]
    RE --> G
    C -->|"PASS"| F["Final\n输出结果"]
    F --> E[END]

    style G fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333
    style R fill:#ffd1d1,stroke:#e57373,stroke-width:2px,color:#333
    style RE fill:#fff3d1,stroke:#ffb74d,stroke-width:2px,color:#333

关键数据： Reflexion 在 HumanEval 上将 GPT-4 的通过率从 80% 提升到 91%。但大多数改进集中在前 2-3 次迭代，超过 5 次后模型会强化自身盲点。

6.2 Self-Critic（自我批评）

与 Reflexion 类似，但引入第二个模型做审查——避免”自己审自己”的盲区。

    flowchart TB
    S[START] --> G["Generator\n(模型A) 生成"]
    G --> C["Critic\n(模型B) 批评"]
    C --> |"改进意见"| G
    C --> |"通过"| F["Final 输出"]
    F --> E[END]

    style G fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333
    style C fill:#ffd1d1,stroke:#e57373,stroke-width:2px,color:#333

Reflexion vs Self-Critic：

维度 Reflexion Self-Critic
审查者 自己审自己 独立模型审查
盲区风险 高（强化自身偏见） 低（视角不同）
成本 中（一个模型） 高（两个模型）
改进幅度 前 2-3 次显著 更持续稳定

7. Layer 5 — 多智能体协作模式

7.1 Supervisor（主管模式）

一个”主管”负责分发任务，多个”专家”负责执行。

    flowchart TB
    U[用户] --> SUP["Supervisor\n主管决策"]
    SUP --> |"调研任务"| R["Researcher\n调研专家"]
    SUP --> |"编程任务"| C["Coder\n编程专家"]
    SUP --> |"分析任务"| A["Analyst\n分析专家"]
    R --> SUP
    C --> SUP
    A --> SUP
    SUP --> |"全部完成"| F["汇总回复"]
    F --> OUT[输出]

    style SUP fill:#ffd1d1,stroke:#e57373,stroke-width:3px,color:#333
    style R fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333
    style C fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333
    style A fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333

已在 06.6 LangGraph 状态机编排 中详细实现，此处不再重复代码。

7.2 Swarm / Handoff（群体交接模式）

没有主管，Agent 之间平等交接任务。

    flowchart LR
    A["客服 Agent\n处理咨询"] --> |"转技术支持"| B["技术 Agent\n处理故障"]
    B --> |"需要回访"| C["回访 Agent\n满意度调查"]
    C --> |"新问题"| A

    style A fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333
    style B fill:#d1e8ff,stroke:#42a5f5,stroke-width:2px,color:#333
    style C fill:#fff3d1,stroke:#ffb74d,stroke-width:2px,color:#333

Supervisor vs Swarm：

维度 Supervisor Swarm
控制力 强（主管统一决策） 弱（Agent 自主交接）
调试难度 低（集中路由） 高（分散决策）
扩展性 主管可能成瓶颈 天然水平扩展
生产推荐 ✅ 首选 仅探索/研究场景

行业共识（2026）：”生产环境几乎总是选 Supervisor 或 Graph 编排，Swarm 仅用于流程无法预设的探索性场景。”

7.3 Map-Reduce（并行分治）

分而治之——拆分任务，并行执行，汇总结果。

    flowchart TB
    S[START] --> A["Researcher\n调研"]
    S --> B["Coder\n编码"]
    A --> C["Summarizer\n汇总"]
    B --> C
    C --> E[END]

    style A fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333
    style B fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333
    style C fill:#ffd1d1,stroke:#e57373,stroke-width:2px,color:#333

适用场景： 独立子任务，如”同时调研 3 个竞品并汇总对比”。

7.4 Debate（辩论模式）

多个 Agent 对同一问题独立作答，互相质疑，择优输出。

    flowchart TB
    Q[问题] --> A["Agent A\n主张 X"]
    Q --> B["Agent B\n主张 Y"]
    A --> |"质疑 Y"| B
    B --> |"质疑 X"| A
    A --> |"反驳"| B
    B --> |"反驳"| A
    A --> J["Judge\n裁判"]
    B --> J
    J --> F["最终答案"]

    style A fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333
    style B fill:#d1e8ff,stroke:#42a5f5,stroke-width:2px,color:#333
    style J fill:#ffd1d1,stroke:#e57373,stroke-width:2px,color:#333

适用场景： 需要高质量决策、对抗性推理（如安全审查、方案比选）。

7.5 Hierarchical（层级模式）

多层 Supervisor 嵌套——像企业组织架构一样分层管理。

    flowchart TB
    U[用户] --> CEO["CEO Agent\n战略决策"]
    CEO --> |"技术线"| CTO["CTO Agent\n技术管理"]
    CEO --> |"产品线"| PM["PM Agent\n产品管理"]
    CTO --> |"后端"| BE["Backend Agent\n后端开发"]
    CTO --> |"前端"| FE["Frontend Agent\n前端开发"]
    PM --> |"设计"| DES["Design Agent\nUI设计"]
    PM --> |"测试"| QA["QA Agent\n质量测试"]

    BE --> CTO
    FE --> CTO
    DES --> PM
    QA --> PM
    CTO --> CEO
    PM --> CEO
    CEO --> OUT[输出]

    style CTO fill:#d1e8ff,stroke:#42a5f5,stroke-width:2px,color:#333
    style PM fill:#d1e8ff,stroke:#42a5f5,stroke-width:2px,color:#333
    style BE fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333
    style FE fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333
    style DES fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333
    style QA fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333

适用场景： 大型企业工作流、复杂项目管理、组织架构映射。

7.6 Pipeline（流水线模式）

Agent 按固定顺序依次处理，每个 Agent 处理前一个的输出。

    flowchart LR
    A["收集 Agent\n采集数据"] --> B["分析 Agent\n提取洞察"]
    B --> C["写作 Agent\n生成报告"]
    C --> D["审核 Agent\n质量检查"]
    D --> E["发布"]

    style A fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333
    style B fill:#d1e8ff,stroke:#42a5f5,stroke-width:2px,color:#333
    style C fill:#fff3d1,stroke:#ffb74d,stroke-width:2px,color:#333
    style D fill:#ffd1d1,stroke:#e57373,stroke-width:2px,color:#333

适用场景： 流程固定的任务链，如数据处理管道、内容生成流水线。

10. 模式组合最佳实践

生产系统不是选一个模式，而是组合使用：

10.1 常见组合

组合 模式 适用场景
基础组合 ReAct + Function Calling 通用 Agent
自愈组合 ReAct + Reflexion 容错 Agent
高效组合 Plan-and-Execute + ReWOO 可预测任务
高可靠组合 ReAct + Self-Critic 代码生成、关键决策
团队组合 Supervisor + ReAct（每个 Worker） 多角色协作
企业组合 Hierarchical + Plan-and-Execute + Reflexion 大型工作流

10.2 渐进式架构升级路径

    flowchart LR
    L1["Step 1\nReAct 基线"] --> L2["Step 2\n+ Reflexion"]
    L2 --> L3["Step 3\n+ ReWOO 快路径"]
    L3 --> L4["Step 4\n+ Supervisor"]
    L4 --> L5["Step 5\n+ Hierarchical"]

    L1 -.- N1["1 Agent\n2-5 步"]
    L2 -.- N2["自愈能力\n2-3 次重试"]
    L3 -.- N3["简单任务\n2 次 LLM"]
    L4 -.- N4["3-5 Agent\n角色分工"]
    L5 -.- N5["N Agent\n企业级"]

关键原则：

  1. 从 ReAct 基线开始 — 先让单 Agent 能跑通
  2. 测量瓶颈再加 — 不要在 Agent 还没到上限时就加多 Agent
  3. 工具 > 模式 — 大多数 Agent 失败是工具定义不好，不是模式选错
  4. 先建评估再选架构 — 如果无法衡量成功，任何模式都救不了你
