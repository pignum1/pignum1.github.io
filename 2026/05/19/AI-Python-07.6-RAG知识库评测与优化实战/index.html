RAG 知识库评测与优化实战

核心问题：构建了一个 124 篇博客的 Wiki 知识库（基于 LLM Wiki 模式），但如何验证它真的好用？回答质量如何量化？出了问题怎么定位和修复？

本文价值：完整记录从”初始评分 3.1/5”到”稳定 4.3-4.5/5”的优化全过程，涵盖 7 个真实 bug 的定位修复、LLM-as-Judge 评测体系搭建、以及 Wiki 内容质量提升策略。这是一次真实的 RAG 优化实战经验，可作为面试中展示 RAG 工程能力的核心案例。

一、知识库架构回顾

1.1 整体架构

关键参数：

  • 124 篇博客 → 124 个 Wiki 页面（经过 LLM 编译压缩）
  • 路由：LLM 从 index.md 选择 1-5 个相关 Wiki
  • 上下文：Top 2 Wiki 全文（≤4000字），其余摘要
  • 评测：LLM-as-Judge，四维度评分（路由/覆盖/忠实/质量）

1.2 技术栈

组件 技术选型 原因
向量数据库 无（LLM Wiki 模式） 124 篇可直接塞入上下文窗口
路由 LLM 读 index.md 选择 利用 LLM 的语义理解能力
评测 LLM-as-Judge (双次评分取平均) 自动化评测，减少人工标注
编译 LLM 将博客编译为 Wiki 结构化 + 信息压缩
模型 glm-5.1 (Anthropic 兼容 API) 中文能力好，成本可控

三、七个真实 Bug 的定位与修复

Bug 1: 回答生成 Prompt 缺失用户问题（最关键）

现象：多个问题得到完全跑题的回答（问 RAG 和 Agent 关系，回答 @tool 装饰器）

根因：ANSWER_PROMPT 模板中没有 {question} 占位符

效果：T07 (RAG+Agent) 从 ~1.0 → 4.5，T08 (学习路线) 从 ~1.0 → 4.5

Bug 2: Index 截断导致尾部领域不可见

现象：Java Spring 相关查询总是触发降级（fallback），无法正常路由

根因：index_content[:10000] 截断在设计模式区域（约第 110 行），Spring 条目从第 153 行开始

效果：T12 (Spring) 从 fallback_hit → hit

Bug 3: T02 ReAct 路由 100% 漏选

现象：ReAct 相关查询约 50% 概率路由不中，走 fallback

根因：Indexer 只取前 5 个 headings，而 “ReAct” 关键词排在第 8 位

效果：T02 ReAct 路由 100% 命中

Bug 4: 评测温度过高导致评分波动

现象：同一问题连续评测分数差异大（3.5 vs 5.0）

修复：eval temperature 从 0.3 降到 0 + 双次评分取平均

Bug 5: 回答生成温度过高

现象：相同 Wiki 内容，不同轮次回答差异大

修复：query temperature 从 0.5 降到 0.3

Bug 6: 编译 Prompt 过度压缩

现象：33 篇 Wiki 缺少速查表，多篇代码示例被截断到只剩骨架

修复：优化编译 Prompt

效果：26 篇重编译后平均内容量 +78%

Bug 7: 来源真实性校验阈值过严

现象：正确回答被判为”来源存疑”（关键词匹配率 0.55 < 0.6 阈值）

分析：这是 evaluator 关键词匹配的固有限制，回答用不同措辞表达同一概念会导致匹配率偏低

决策：保持 0.6 阈值，接受少量误判。未来可升级为 embedding 相似度

六、面试高频问题

Q1: 你的 RAG 系统遇到的最大挑战是什么？

A: 最具挑战的是回答生成 LLM 完全忽略用户问题。根因是 ANSWER_PROMPT 模板中缺少 {question} 占位符 —— Python 的 str.format() 会静默忽略未使用的参数，所以问题被悄悄丢弃了。LLM 只看到 Wiki 内容，无法知道用户在问什么，只能随机总结上下文。这个 bug 隐藏得非常深，因为路由正确（选了正确的 Wiki），但回答完全跑题。最终通过 LLM-as-Judge 的忠实度评分发现端倪，逐条检查回答内容后定位到根因。

Q2: LLM-as-Judge 评测怎么做的？如何保证稳定性？

A: 三个关键技术：

  1. temperature=0 — 确定性输出，同样的输入同样的评分
  2. 双次评分取平均 — 自洽性校验，两次评分差异大说明 prompt 设计有问题
  3. 给 evaluator 看实际 Wiki 内容 — 而非仅给 slug，让评分有据可依

另外评分标准从模糊的 3 级改为精确的 5 级锚点（每级有具体描述），减少主观性。

Q3: 如何处理知识库中后部内容的检索问题？

A: 经典的长文档截断问题。Index 有 124 个条目约 19000 字，但路由器只能看前 10000 字，导致 Spring（第 153 行起）等尾部领域完全不可见。

解决方案不是简单增大窗口，而是在 Index 顶部添加领域概览行——一行文字覆盖所有 16 个领域的名称和篇数，始终在截断窗口内。路由器先通过概览定位领域，再从详细条目中找具体 Wiki。这本质上是”摘要 + 详情”的两级检索思路。

Q4: 你的 Wiki 编译策略是什么？如何保证内容质量？

A: 编译策略的核心是信息密度最大化。初始 Prompt 要求”精简到 5-10 行”，导致大量细节丢失。优化后的 Prompt 要求：

  1. 保留完整代码逻辑（10-20 行而非 5-10 行）
  2. 必须包含速查表（从内容中自行构建）
  3. 术语必须中英对照（如 “自注意力机制(Self-Attention)”）

重编译 26 篇后平均内容量 +78%，回答质量显著提升。

Q5: 如果让你重新设计这个系统，会怎么做？

A: 三个改进方向：

  1. 评测先行：先搭建稳定的评测体系，再做任何优化。避免”优化了但不知道效果”
  2. 语义化忠实度检查：用 embedding 相似度替代关键词匹配，避免语义等价但措辞不同被判为”不忠实”
  3. 分层路由：先用领域概览做粗筛（一级），再从领域内 Wiki 列表做精选（二级），减少 index 截断影响
