AI-Python-08.2: Agent 实战 — 从零构建智能体

核心认知：Agent = LLM（大脑推理）+ Tool（手脚交互）+ Memory（记忆上下文）+ Planning（规划策略），通过 ReAct 循环（思考→行动→观察）协同工作。

本文路径：先拆解 Agent 的四大组件，理解每个组件的职责；再用 50 行纯 Python 手搓一个 Agent；最后用 LangChain/LangGraph 框架重构。先懂原理，再用框架。

1. Agent 的本质与组成

1.1 什么是 Agent？

“An AI agent is a system that combines LLMs for reasoning and decision-making with tools for real-world interaction.” — Andrew Ng

“AI agents are autonomous systems that can understand queries, plan workflows, and execute complex tasks by combining LLMs with external tools.” — IBM Research

普通 LLM 是一问一答 — 你问它答，它只能用训练数据里的知识。遇到需要实时数据、外部操作的问题就束手无策。

Agent 是一个自主决策系统 — LLM 不再只回答一次，而是可以反复思考、调用工具、观察结果，直到完成任务。

维度 普通 LLM 对话 Agent
交互模式 一问一答 自主多轮决策
外界能力 无（只能用训练数据） 可调工具（API、数据库、搜索引擎）
决策权 人类决定每步做什么 LLM 自己决定下一步
适用场景 简单问答/写作 复杂任务/多步骤推理

1.2 Agent 的四大组件

Agent 不是单一的东西，而是由四个组件协同工作的系统：

每个组件的功能和职责：

🧠 LLM（大脑）— 推理与决策

职责：理解问题、分析信息、决定下一步行动。

LLM 是 Agent 的核心引擎。在循环的每一轮，LLM 做三件事：

  1. 理解当前状态 — 用户问了什么？工具返回了什么结果？
  2. 推理下一步 — 信息够了吗？还需要查什么？用什么工具？
  3. 生成输出 — 要么调用工具（Action），要么生成最终回答（Answer）

关键能力：LLM 不只生成文字，它能输出结构化的工具调用指令（Function Calling），告诉代码”请帮我调 get_weather(city='北京')“。

🔧 Tool（工具）— 外部交互

职责：执行具体操作，获取外部数据，让 LLM 能和真实世界交互。

LLM 本身被”关在笼子里” — 不知道今天的天气、不能查数据库、不能调 API。Tool 就是把 LLM 和外部世界连接起来的桥梁。

Tool 的三要素：

  • name — 工具名称（如 get_weather）
  • description — 功能描述（LLM 靠它判断何时调用）
  • parameters schema — 参数格式（LLM 按此格式输出 JSON）

本质：LLM 不直接执行任何代码。它只输出 JSON 指令说”我想调这个工具”，是你的代码负责真正执行函数并把结果喂回 LLM。

💾 Memory（记忆）— 上下文管理

职责：维护对话历史，让 LLM 在每一轮都能看到完整上下文。

LLM 是无状态的 — 每次调用都是独立的，不记得上一轮说了什么。Memory 就是把所有历史信息拼接起来，每轮都传给 LLM：

Memory 的两个层级：

类型 存什么 生命周期 示例
短期记忆 当前对话历史 一次会话 本轮的问答、工具调用结果
长期记忆 跨会话知识 持久化 向量数据库存储的历史经验

Agent 的消息列表（messages）就是最简单的短期记忆 — 每轮循环往里面追加新信息，LLM 每次都能看到完整历史。

📋 Planning（规划）— 任务分解与策略

职责：面对复杂任务，先制定计划，再逐步执行。

简单的任务（查个天气）LLM 不需要规划。但复杂任务需要先拆解：

Planning 在 Agent 中的体现：

模式 规划方式 适用场景
ReAct 边想边做，每轮即时决策 简单任务（2-5 步）
Plan-and-Execute 先出完整计划，再逐步执行 复杂任务（5+ 步）
Reflexion 执行后自我反思，修正计划 需要高质量输出

注意：在手搓 50 行 Agent 时，Planning 隐含在 LLM 的推理能力中（LLM 自己决定先查什么再查什么）。在高级 Agent 框架（如 LangGraph）中，Planning 才会显式编码为独立节点。

1.3 组件如何协作：ReAct 循环

四个组件不是独立工作的，它们通过 ReAct 循环 协同运行。

ReAct（Reasoning + Acting） 是 2022 年 Yao et al. 提出的经典 Agent 模式。核心思想：思考 → 行动 → 观察，反复迭代直到完成任务。

每个组件在循环中的角色：

ReAct 循环的本质：

  • while 循环提供了机会（可以反复尝试）
  • LLM 提供了判断力（决定每步做什么）
  • Tool 提供了能力（能和外界交互）
  • Memory 提供了记忆（记住每步发生了什么）

没有 while 循环 → Agent 只能做一步（就是普通 LLM）
没有 LLM → 代码不知道该调什么工具（就是普通程序）
没有 Tool → Agent 无法和外界交互（LLM 只能编造答案）
没有 Memory → Agent 记不住上一步做了什么（每轮都从零开始）

1.4 Agent 架构全景图

1.5 核心概念：Tool / MCP / Skill

理解了 Agent 的四大组件后，再看几个和 Tool 相关的重要概念。

MCP（Model Context Protocol）

MCP 是 Anthropic 发布的开放标准协议，为 AI 模型提供连接外部数据源和工具的统一方式。 — Anthropic 官方

如果把 AI 比作电脑，MCP 就是 AI 的 USB 接口：

角色 类比 职责
Host 电脑 AI 应用本体（Claude Desktop、Cursor）
Client USB 控制器 协议翻译，连接 Host 和 Server
Server USB 设备 提供具体的工具和数据（文件系统、数据库、GitHub）

MCP 的核心价值：一次接入，处处可用。工具开发者只需按 MCP 标准写一个 Server，所有支持 MCP 的 AI 应用都能直接使用。

Skill（技能）

Skill 是 Anthropic Agent SDK 中的概念：一组组织好的指令、脚本和资源，Agent 可以动态发现和加载。 — Anthropic Agent Skills

Skill 比 Tool 更高维 — 它不是单个函数，而是一个能力包：

层级 概念 粒度 示例
Tool 单个函数 get_weather(city)
Skill 能力包 SKILL.md + scripts/ + resources/
Agent 自主系统 LLM + Skills + Memory

Skill 的渐进式加载机制：

  1. Agent 先看 Skill 的元数据（名字、描述）— 决定是否需要
  2. 如果需要，加载 SKILL.md（完整指令）— 了解怎么做
  3. 执行时再加载 bundled 的脚本和资源 — 拿到具体工具

概念关系与演进

一句话总结：Agent 包含 Skills，Skills 通过 MCP 协议连接 Tools。Function Calling 是底层机制，MCP 是通信标准，Skill 是上层能力封装。

5. Agent 调试与可观测性

5.1 常见问题

问题 现象 原因
无限循环 Agent 一直调工具不停 工具描述不清，LLM 不知道何时停止
选错工具 问天气却调了计算器 工具 description 写得不好
幻觉参数 get_weather(city="今天") 参数 description 不够明确
卡死 一直 Step 1 没响应 API 超时或 prompt 太长

5.2 调试策略

核心：打印每一步的 Thought/Action/Observation

三个必查项：

  1. LLM 的 thought 是什么？→ 看它理解了问题没
  2. LLM 选了什么 action？→ 看它选对工具没
  3. 工具的 observation 是什么？→ 看结果是否有效
