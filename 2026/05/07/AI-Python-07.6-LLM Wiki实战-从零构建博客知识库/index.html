AI-Python-07.6: LLM Wiki 实战 — 从零构建博客知识库

07.5 章全景对比了六大 RAG 架构后，选定 LLM Wiki 模式（预编译知识库）作为最适合个人博客的方案。本章完整记录从架构设计到代码实现、全量编译、查询优化的全过程，将 124 篇 Hexo 博客编译为结构化 Wiki 知识库，实现零基础设施的智能问答。

1. 项目背景

1.1 知识资产盘点

经过一年多的 AI 学习，积累了 124 篇技术博客，涵盖多个领域：

系列 篇数 内容范围
AI-Python-01.x 7 Python 语法精髓（类型、切片、推导式、装饰器、魔法方法、dataclass）
AI-Python-02.x 5 Python OOP/元编程（面向对象、继承、元类、描述符、ABC）
AI-Python-03.x 3 Python 并发（GIL/多线程、Asyncio、多进程）
AI-Python-04.x 3 数据科学（NumPy、Pandas、Matplotlib）
AI-Python-05.x 3 深度学习（PyTorch、Transformer、推理优化）
AI-Python-06.x 6 LangChain/LangGraph（基础、Prompt、LCEL、Memory、Tool/Agent、LangGraph）
AI-Python-07.x 5 RAG 检索（向量检索、文档解析、混合检索、GraphRAG、架构对比）
AI-Agent 1 学习路线图
设计模式 23 GoF 23 种设计模式
计算机基础 16 组成原理(12)、网络(3)、操作系统(1)
Java 相关 ~52 JavaSE、JVM、并发、Spring、分布式、消息队列、数据库等
合计 124 约 40-50 万字

1.2 为什么选 LLM Wiki？

在 07.5 的架构对比中，LLM Wiki 模式最适合这个场景：

考量 LLM Wiki 的优势
124 篇 < 500 篇上限 完全在 LLM 上下文窗口范围内
纯 Markdown 博客 天然适配文本处理，无需 OCR
个人知识库 不需要企业级基础设施
想要可读的知识 编译后的 Wiki 人可阅读，向量 DB 不可读
零运维成本 只需 Git + Markdown，无需数据库服务

4.1 完整查询链路

    flowchart TB
    Q[用户提问] --> R[Step 1: 智能路由\nLLM 读取 index.md\n定位相关 Wiki]
    R --> |命中| E[Step 2: 关联扩展\n基于 links.json\n扩展相关页面]
    R --> |未命中| F[Step 5: 降级兜底]
    E --> C[Step 3: 分层上下文构建\nTop 2 全文 + 其余摘要]
    C --> G[Step 4: LLM 生成回答\n基于上下文回答]

    EV --> |评分 < 3.5| RETRY[Step C: 二次扩展检索\n补充更多上下文]
    RETRY --> OUT

    F --> |Level 1| K[关键词搜索]
    K --> |有结果| G
    K --> |无结果| T[Level 2: Tag模糊匹配]

4.2 上下文分层策略

层级 内容 预算
Top 1 Wiki 完整内容（截断到 4000 字） ~2000 tokens
Top 2 Wiki 完整内容（截断到 4000 字） ~2000 tokens
其余主 Wiki 摘要 + 核心概念 ~500 tokens/篇
扩展 Wiki 仅摘要 ~200 tokens/篇

核心思路： 装得下就用全文，装不下才压缩。Top 2 Wiki 保留完整内容确保回答质量，其余只取精华控制延迟。

12. 设计心得与踩坑记录

12.1 踩坑：API 认证方式

智谱 AI 提供 OpenAI 和 Anthropic 两种兼容接口，认证方式不同：

接口 端点 认证方式 结果
OpenAI 兼容 /api/paas/v4 Bearer <api_key> 401 失败
Anthropic 兼容 /api/anthropic/v1 x-api-key: <token> 成功

GLM Coding Plan 的 API key 通过 Anthropic 兼容接口调用，模型名为 glm-5.1。最终实现了一个轻量 LLM 客户端，直接用 urllib 调用，无需额外 SDK。

12.2 设计心得：上下文管理的平衡

查询质量与速度的核心矛盾在于上下文长度：

  • 全文加载 → 质量高（35s）→ 慢
  • 全压缩 → 速度快（10s）→ 质量差
  • Top 2 全文 + 其余摘要 → 平衡（15-25s）→ 质量好

最佳策略不是”一刀切”，而是根据相关性分层处理。

12.3 设计心得：降级兜底必不可少

LLM 路由不是万能的。当用户问题模糊或超出知识库范围时，三级降级确保用户始终有反馈，而不是冷冰冰的”未找到结果”。
