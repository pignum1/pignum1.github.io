AI Agent 面试实战 — 第7周：Agent Pool、Session 隔离、调度与通信

第7周完善 L4 编排层——Agent Pool 资源管理（状态机 + 并发安全 + 能力匹配）、Session 全维度隔离（上下文/Memory/Workspace/有状态工具）、资源感知调度器、Blackboard 事件驱动通信。

一、Agent Pool 资源管理

Q1: Agent Pool 的资源分配机制是怎样的？如何保证并发安全和能力匹配？

难度：⭐⭐⭐⭐（深入）
频率：★★★★（常见）

回答结构：

“Agent Pool 是整个系统的’人力资源部’，负责管理所有 Agent 的状态（空闲、忙碌、故障）和按需分配。核心实现有四个关键点：

状态机

Agent 只有三种状态：idle（可分配）、busy（正在执行任务）、error（例如模型不可用，需人工恢复）。所有状态变更通过 asyncio.Lock 保护的原子操作完成。

能力匹配

调用 acquire(required_capabilities) 时，会遍历所有 idle 的 Agent，检查其 Persona.capabilities 是否覆盖请求的每一项能力。公式：agent.caps[c] >= required[c] 对所有 key 成立。如果有多个匹配，优先选择在当前团队已有 L4 上下文记忆的 Agent，减少学习成本。

并发安全

由于 SOP Engine 是异步的，多个协程可能同时请求同一个 Agent。通过在 acquire 内部使用 asyncio.Lock，保证’检查状态 → 标记 busy → 返回 Agent’是一个原子操作。后来的请求会收到 AgentBusyError，节点执行器会按配置重试（指数退避），超过重试次数则任务失败。

资源回收

节点执行结束后，在 finally 块中调用 release，将 Agent 状态置回 idle，并清除该 agent 的当前 session 引用。如果 Agent 处于 error 状态，不会自动回收，需要人工排查并调用 mark_error 恢复。”

追问：如果长时间没有空闲 Agent，任务会无限排队吗？

“不会。创建任务时可以配置 timeout（默认 10 分钟）。如果超时仍未获得 Agent，任务自动标记为 failed，并通知用户。另外 Scheduler 有一个防饥饿机制，记录了每个任务的等待时间，可以动态提升优先级，防止低优先级任务永远得不到资源。”

二、Session 隔离与有状态工具管理

Q2: Agent 同时参与多个团队，如何保证记忆和工作环境不互相污染？

难度：⭐⭐⭐⭐（深入）
频率：★★★★（常见）

回答结构：

“这就是 Session Manager 的核心职责。Session 是 Agent 在特定团队内的隔离运行环境。关键隔离维度包括：

上下文隔离

每个 Session 拥有独立的对话历史，Redis 中用 session_id 作为 key 存储，不同 Session 无法互相读取。

记忆视图隔离

Session 创建时，MemoryManager 会生成该 Agent 在当前团队的记忆快照：L1 全局 + L2 本团队 + L3 个人 + L4 本团队上下文。Agent 换团队后，L2 和 L4 完全切换，L3 随身携带。

工作空间隔离

每个 Session 绑定一个 Workspace（文件目录）。对于 Terminal/Docker 等有状态工具，实例存储在 session.tool_instances 字典中，生命周期与 Session 绑定。

有状态工具 GC

Session 关闭时，close_session 方法会遍历所有 tool_instances，调用各自的 cleanup() 方法（如关闭 Terminal 进程、停止 Docker 容器）。如果 Session 异常退出，后台有一个定时任务扫描 last_active > TTL 的僵尸 Session，强制回收资源，防止文件描述符泄漏。”

追问：Session 在 Redis 和 PostgreSQL 中分别存了什么？

“Redis 存储当前活跃 Session 的对话上下文、短期记忆和临时工具状态，主要用于高速读写。PostgreSQL 存储 Session 的元数据（status、创建时间、绑定的 team/agent）以及已持久化的 L4 记忆条目。Session 关闭时，Redis 中的关键记忆会异步刷入 PostgreSQL，保证长期可查。”

三、任务调度与优先级

Q3: 你们的任务调度器如何平衡优先级和资源约束？

难度：⭐⭐⭐⭐（深入）
频率：★★★（偶见）

回答结构：

“调度器需要同时考虑三个维度：任务优先级、Agent 可用性、LLM API 限流余量。

优先级队列

基于 Redis ZSET 实现。Score = priority_weight * 10^6 + timestamp_sequence，保证同优先级内 FIFO。级别：CRITICAL（HITL 恢复）、HIGH（开发 SOP）、NORMAL（学习任务）、LOW（批量分析）。

资源感知调度

主循环从队列头部取任务，但在真正执行前会检查：

  1. AgentPool 中是否有空闲 Agent 满足任务能力要求？
  2. ModelAdapter 的 RateLimiter 中所有涉及 Provider 的 Token Bucket 是否有足够令牌？

如果任一条件不满足，任务重新放回队列头部，调度器短暂休眠后重试。

防饥饿

每个优先级维护一个 max_wait_time。当低优先级任务等待超过该时间（如 LOW 默认 30 秒），自动提升一级。避免高优先级任务持续涌入时低优先级任务永远得不到执行。

集成到 SOP Engine

POST /api/v1/tasks 不再直接执行 SOP，而是调用 scheduler.enqueue。调度器后台协程 run_scheduler_loop 持续拉取并执行。这使得系统能同时处理大量任务，而不会立即耗尽资源。”

追问：如果 Agent 正在执行低优先级任务，突然来了一个 CRITICAL 任务，能抢占吗？

“MVP 阶段采用非抢占模式——不会强行中止正在运行的 Agent，避免 Workspace 状态混乱。如果 CRITICAL 任务必须立即执行，可以通过配置保留一个预留槽位（Reserved Slot），始终空闲一个 Agent 专供紧急任务使用。后续我们会结合 Workspace 快照实现安全抢占，即保存当前进度、回收 Agent、执行紧急任务、再恢复原任务。”

四、Blackboard 通信模型

Q4: 团队内的 Agent 如何实时感知任务状态变化？跨团队通信怎么实现？

难度：⭐⭐⭐（进阶）
频率：★★★（常见）

回答结构：

“我们设计了一个基于 Redis Pub/Sub 的 Blackboard 事件总线。

频道结构

  • team:{team_id}:events 用于团队内广播（task_update、agent_status 等）
  • global:events 用于跨团队通知和系统事件（rate_limit_warning、cross_team_request）

消息模型

每条消息包含 event_type、payload、timestamp。SOP Engine 在每个节点开始/结束时发布 task_update；Agent Pool 在 Agent 状态变化时发布 agent_status；HITL 节点发布 hitl_notification。

前端实时更新

后端 WebSocket 服务订阅相关团队频道，收到消息后推送给已连接的前端。前端据此实时更新节点状态颜色和对话流。

跨团队请求

如 DevTeam 需要 DataTeam 的帮助，SOP 可通过 Blackboard 的 cross_team_request 事件发布需求。DataTeam 的 Leader Agent 订阅该频道，收到通知后可决定是否协助，并发送响应事件。这类似于微服务间的异步协作模式。”

追问：消息丢失怎么办？

“Redis Pub/Sub 是 fire-and-forget，没有持久化。但我们同时在 PostgreSQL 的 event_log 表中存储了关键事件（如任务状态变更），前端可随时通过 REST API 获取历史状态，作为 Pub/Sub 的补偿。对于跨团队请求，如果响应超时未收到，SOP 会重试或降级为 HITL 人工处理。”

五、并发场景下的边界处理

Q5: 多个 SOP 并行执行，如果两个任务同时请求同一个 Agent，会发生什么？

难度：⭐⭐⭐（进阶）
频率：★★★（场景题）

回答结构：

“典型并发冲突场景，我们用三步解决：

原子获取

AgentPool.acquire 内部用 asyncio.Lock 串行化所有请求。第一个到达的协程获得 Agent 并标记 busy，后续协程会收到 AgentBusyError。

节点级重试

SOP Engine 的 agent_node_executor 捕获该异常后，按节点配置的 retryDelay 等待（首次 1 秒，后续加倍），然后重新调用 acquire。重试次数由节点 maxRetries 控制。

超时处理

如果最终仍未获得，任务标记为 failed，并记录 error_reason = "Agent unavailable"。同时调度器会收到通知，可根据策略决定是否重新加入队列或转人工。”

追问：如果一个 Agent 执行到一半自己崩了（error），正在执行的任务怎么办？

“Agent 从 busy 变为 error 时，AgentPool 会通过 Blackboard 或回调通知 SOP Engine。SOP Engine 会将当前 Workspace 做一个紧急快照，然后尝试将节点重新分配给同角色的备用 Agent（如果有），从快照状态继续执行。如果没有备用 Agent，任务暂停并升级为人工恢复。”

第7周面试总结

亮点 解析 面试价值
Agent Pool 能力匹配+并发安全 状态机+原子锁，避免超配和冲突，支持记忆关联 ⭐⭐⭐⭐
Session 全维度隔离 上下文/Memory/Workspace/有状态工具四重隔离，GC 防泄漏 ⭐⭐⭐⭐
资源感知调度 优先级队列 × Agent 可用 × API 限流，三维决策 ⭐⭐⭐⭐
Blackboard 事件驱动 基于 Redis Pub/Sub 的实时通信，解耦团队间协作 ⭐⭐⭐
优雅降级与冲突处理 并发冲突、Agent 故障等异常场景均有明确恢复路径 ⭐⭐⭐

一句话总结第7周：

第7周完善了 L4 编排层的”基础设施”——Agent Pool 管理多 Agent 生命周期，Session 实现全维度隔离防止数据串扰，调度器在有限资源下平衡效率与公平，Blackboard 让 Agent 实时协同而不强耦合。四者共同构成了多 Agent 系统可靠运行的底座。
