AI Agent 面试实战 — 第8周：SOP 设计器、实时同步、可观测性与测试

第8周 L4 编排层收尾 + Harness 启动——SOP 可视化设计器（React Flow + YAML 双向同步）、聊天室 WebSocket 实时同步、Observer 轻量级 Agent Trace、Workspace Manager 隔离与快照、四层测试体系。

一、SOP 设计器前端（React Flow）

Q1: 你们的 SOP 设计器是怎么实现的？为什么选择 React Flow？

难度：⭐⭐⭐（进阶）
频率：★★★（常见）

回答结构：

“我们基于 React Flow 构建了可视化 SOP 编辑器。React Flow 是 LangGraph Studio、Flowise、Dify 等主流 Agent 平台的共同选择，它提供了开箱即用的节点拖拽、连线、缩放平移、自定义节点渲染等能力，让我们把精力集中在业务逻辑上。

核心功能模块：

  • 节点面板：左侧有五种节点类型（Start、Agent Action、HITL、Validation、End），支持拖拽到画布。
  • 属性编辑：点击节点，右侧面板动态显示该节点类型的配置项。例如 Agent Action 节点可以下拉选择 Role Slot 和 Skill，配置 retry 参数；HITL 节点可以设置人工审批还是自动通过，甚至配置条件自动审批。
  • 连线与条件路由：从一个节点的 handle 拖出连线到另一个节点，点击连线可以编辑 condition 表达式（如 hitl_result == 'approve'），支持配置 max_loop。
  • YAML 双向同步：画布操作会实时生成 YAML 预览在底部面板；用户也可以直接编辑 YAML，画布自动重新渲染。这得益于我们设计的一套 YAML ↔ Graph 双向绑定的状态管理。
  • 校验：保存前自动检查孤立节点、死循环、必填字段、Role Slot 是否有对应 Agent 等，给出错误提示。

技术决策：用 dagre 做自动布局，js-yaml 做解析生成，节点颜色/图标与后端执行引擎保持一致，降低认知负担。

亮点：YAML 和画布的双向同步不是简单的字符串替换，而是通过一个中间数据结构（SOP Graph Model）进行转换，确保画布的图形信息和 YAML 的文本信息完全等价。这是很多同类工具容易出 Bug 的地方。”

追问：为什么不是纯 YAML 编辑，而是加了一套可视化画布？

“纯 YAML 适合开发者快速定义简单流程，但面对复杂分支、循环、并行节点时，文本的嵌套结构容易出错，可读性也差。可视化画布直观展示了流程的拓扑和状态流转，同时我们保留了 YAML 预览和直接编辑能力，两者共存。未来我们可以在节点上直接显示上次执行耗时、成功率等指标，把编辑器升级为’流程观测与优化控制台’。”

二、聊天室 WebSocket 实时同步

Q2: 前端聊天室如何实时展示 SOP 执行进度和 Agent 对话？WebSocket 断开怎么处理？

难度：⭐⭐⭐（进阶）
频率：★★★★（常见）

回答结构：

“聊天室是用户观察 Agent 协作的核心窗口。我们通过 WebSocket 实现实时同步，配合 REST 兜底保证数据完整性。

数据流：

  1. SOP Engine 在每个节点开始/结束时，通过 Blackboard（Redis Pub/Sub）发布 task_update 事件。
  2. 后端 WebSocket 服务订阅相应团队频道，将事件推送给前端。
  3. 前端维护一个节点状态映射表，根据事件更新每个节点的颜色/动画（pending 灰、running 蓝旋转、done 绿、failed 红、paused 黄）。

Agent 对话流：

当 Agent 流式输出时，每收到一个 token，后端推送一个 agent_message 事件片段，前端模拟打字机效果。内容支持 Markdown 渲染（用 react-markdown），代码块、表格都能正确展示，提升可读性。

HITL 审批 UI：

到达 HITL 节点时，前端弹出审批对话框，显示消息内容、超时倒计时，并提供’通过’和’驳回’按钮。用户操作后调用 REST API 恢复任务。

断线重连与状态补偿：

WebSocket 断开是分布式系统的常见问题。我们设计了三层保障：

  1. 前端自动重连，指数退避（1s, 2s, 4s，最多 5 次）。
  2. 重连成功后，立即调用任务状态 REST API（GET /api/v1/tasks/{task_id}）获取当前完整状态，全量刷新 UI，确保不丢任何进度。
  3. 如果短时间无法重连，前端显示友好提示，用户可手动刷新。

亮点：WebSocket 做增量推送，REST 做全量补偿。这避免了单纯依赖 WebSocket 导致断线期间状态丢失的经典问题，同时保持了极低的网络开销。”

追问：为什么不直接用 WebSocket 做所有的状态同步，还要保留 REST？

“WebSocket 推送的本质是’事件流’，适合通知变化，但无法提供当前’完整态’。当页面初次加载或断线重连时，必须有一个全量数据源来恢复现场。REST 就是这个数据源。两者职责清晰：WebSocket 负责实时增量，REST 负责全量快照，配合起来才能应对复杂网络环境。”

三、Observer 可观测性基础

Q3: Observer 模块是如何追踪整个 SOP 调用链的？和分布式追踪有什么异同？

难度：⭐⭐⭐⭐（深入）
频率：★★★★（常见）

回答结构：

“Observer 是系统的’黑匣子’，核心能力包括 Trace 追踪、结构化日志和 Token 统计。

Trace 追踪（简版 OpenTelemetry）：

我们设计了一个 Span 树模型。每个任务创建一个 Trace，每个执行步骤（SOP 工作流、节点执行、LLM 调用、工具调用）创建一个 Span，形成父子关系。

例如一次开发 SOP 的完整 Trace：

数据持久化到 PostgreSQL，可通过 API 查询完整调用树。这让我们能精确回答：“这次任务哪个节点最慢？哪个模型成本最高？Agent 调用了哪些工具？”

与分布式追踪的区别：

传统分布式追踪（Jaeger/Zipkin）关注跨服务 RPC 调用，我们需要追踪的是单进程内的异步调用链，且关注 LLM 特化属性（模型名、Token 数、温度参数）。我们参考了 OpenTelemetry 的 Span 概念，但做了轻量化实现，无需 sidecar 或 collector。

集成方式：

通过 Python 装饰器 @observer.trace(name) 无侵入地注入到 ModelAdapter、ToolExecutor 等关键方法。装饰器自动创建 Span、捕获异常、记录耗时和属性。在 FastAPI 中间件层，自动为每个请求创建 Trace。

Token 统计：

每次 LLM 调用后，在 Span 中附加 prompt_tokens、completion_tokens，同时实时聚合到 Redis（按小时），异步持久化到 PG。提供 API 查询成本趋势、团队费用排名。

亮点：我们在没有引入完整 OpenTelemetry 栈的情况下，用约 300 行 Python 代码实现了 Agent 领域的定制化追踪，开销极低（每个 Span < 1ms）。这对于理解 Agent 行为、优化 Prompt 和成本分配至关重要。”

四、Workspace Manager 基础

Q4: Agent 操作文件时，如何防止多个 Agent 并发修改冲突？Snapshot 快照是怎么做的？

难度：⭐⭐⭐⭐（深入）
频率：★★★（偶见但高价值）

回答结构：

“Workspace Manager 是文件系统的’隔离沙箱’和’回滚装置’。

隔离策略：

每个 Session 拥有独立的工作目录。所有 Agent 的文件操作（读写、执行）都通过 WorkspaceManager 代理，确保操作限定在 Session 的工作空间内，不同 Agent 的临时文件不会互相污染。

并发控制：

使用 asyncio.Lock 实现文件级别的读写锁。读操作共享锁（允许多个 Agent 同时读取），写操作排他锁（写文件时其他 Agent 等待）。锁的 key 是规范化后的文件路径，避免死锁通过超时释放机制解决。

Snapshot 快照：

基础版本使用文件哈希 + 备份目录方式实现：

  • create_snapshot(workspace) 会计算当前所有文件的 SHA256 哈希，并将文件复制到一个 backup 子目录。
  • restore_snapshot(workspace, snapshot_id) 读取备份目录中的记录，将文件原样覆盖回工作空间。
  • 快照列表及元数据存储在 SQLite 或 JSON 文件中，简单可靠。

应用场景：

Snapshot 是 Feedback Loop 的基础设施。当 Agent 的输出在 Validation Gate 校验失败时，我们将工作空间回滚到变更前的快照，再注入错误信息让 Agent 重试，确保每次重试都基于干净状态。

进阶设计：

后续会升级为基于 Git 的增量快照，利用 Git 的 three-way merge 能力处理并行分支合并，并可通过 git diff 直接查看 Agent 的代码改动，方便 Code Review。”

五、集成测试与端到端验证

Q5: 整个多 Agent 系统的端到端测试你们是怎么设计的？如何覆盖异步和并发场景？

难度：⭐⭐⭐（进阶）
频率：★★★（常见）

回答结构：

“端到端测试是保证系统可靠性的关键。我们设计了四层测试策略：

单元测试（pytest + asyncio）：

Mock 外部 LLM 调用和数据库，验证每个 Service 层的核心逻辑，如 AgentPool 的 acquire/release 并发安全、ConditionRouter 的各种表达式。

集成测试：

真实数据库（测试 PostgreSQL）+ Mock 的 LLM 适配器。布置完整的 SOP，模拟各类输出，验证状态流转、HITL 流程、错误恢复路径。

并发压力测试：

用 asyncio.gather 同时发起多个 SOP 执行任务，检查 Agent Pool 的独占性、Session 隔离、Scheduler 优先级排序。用 pytest-timeout 防止死锁。

E2E 验收测试：

启动完整 Docker 环境，用 httpx 执行任务 API，用 playwright 打开前端，检查聊天室节点变色、WebSocket 消息推送。模拟真实用户操作（审批、驳回）。

异步测试技巧：

由于整个系统大量使用 async/await，所有测试必须用 @pytest.mark.asyncio，并使用 pytest-asyncio 插件。对于需要等待真实时间的场景（如重试延迟），我们用 asyncio.sleep(0.1) 加速，避免测试太慢。

测试数据管理：每个测试类开始时，通过 fixture 创建独立的 Team、Agent、SOP，测试结束后自动清理，保证互不干扰。”
