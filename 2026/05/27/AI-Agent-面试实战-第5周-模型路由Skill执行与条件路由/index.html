AI Agent 面试实战 — 第5周：模型路由、Skill 执行与条件路由

第5周完成 L3 策略层——模型智能路由（成本降低 80%）、Skill 可执行指令系统、条件路由 AST 求值器。让系统从”能做”变为”会做”。

一、模型路由与成本优化

Q1: 什么是模型路由（Model Router）？你们是怎么实现的？

难度：⭐⭐⭐（进阶）
频率：★★★★★（必问）

回答结构：

“模型路由的核心思路是：不同任务用不同模型，在成本和质量间自动平衡。

我们实现了一个自动路由层：

  1. 任务复杂度分类：对用户输入进行快速评估。消息长度 < 50 字且无特殊关键词 → SIMPLE；包含’架构/设计/选型’或长度 > 300 字 → COMPLEX；其余为 STANDARD。分类器用规则引擎实现，轻量且无额外 API 消耗。
  2. 模型选择：复杂度映射到模型池：
复杂度 模型 特点
SIMPLE Haiku / GLM-4-Flash 最快、最便宜
STANDARD Sonnet / GPT-4o / GLM-5.1 性价比
COMPLEX Opus / Claude-Sonnet-4-6 最高质量
  1. Fallback 链：如果主模型不可用（429、500、超时），自动按顺序尝试下一个。每失败一次增加退避，最多 3 次，最终降级到 HITL。

效果：在日常开发中，约 60% 的简单对话自动使用 Haiku，单次成本降低了 80%，而复杂任务仍保证用最强模型完成。”

追问：复杂度分类只用规则？有没有考虑过用 LLM 分类？

“规则分类在初期已覆盖 95% 的场景，且零延迟、零成本。但我们在代码里预留了 classify_by_llm 的扩展点，当规则判断不确定时（如长度中等的模糊需求），可降级调用轻量模型做补充分类。这是典型的快速启动 + 渐进式增强策略。”

Q2: Fallback 链是怎么设计的？怎样避免雪崩？

难度：⭐⭐⭐（进阶）
频率：★★★★（常见）

回答结构：

“Fallback 链是模型路由的’保险丝’。

设计上遵循指数退避 + 最大尝试：

  1. 主模型失败时，立即尝试下一个候选模型（首次不退避）
  2. 第二次失败后等待 2 秒，第三次等待 4 秒
  3. 超过 3 次全部失败，抛出异常并告警，此时 Orchestrator 会降级到 HITL，由人工兜底

为了防止雪崩，我们做了两个关键控制：

  • 全局限流：所有调用共享 Token Bucket，单个模型流量暴增会被限流，不会把压力传递到下一个模型。
  • 熔断器：实时监控模型错误率，连续失败超过阈值直接熔断，避免向已不可用的模型发送请求。

这样即使在某个 Provider 全面故障时，系统仍能秒级平滑切换。”

二、Skill 执行与指令注入

Q3: Skill 是怎么被 Agent 真正执行的？和直接写 Prompt 有什么区别？

难度：⭐⭐⭐（进阶）
频率：★★★★（常见）

回答结构：

“Skill 不是简单的 Prompt 复制，而是一个自包含的执行单元。

执行流程：

  1. 加载 Skill 定义：从 Skill Registry 读取 SKILL.md 中的指令（Instructions）和输出格式（Output Format），以及 config.yaml 中的参数列表。
  2. 参数注入：SOP 节点将任务上下文（TaskState）中的相关变量按 config.yaml 定义的参数映射注入 Skill。
  3. 组装 Skill Prompt：使用专门的 Skill 模板，把指令和当前任务组装成独立上下文块，注入到 Agent 的 System Prompt 中。
  4. Agent 执行：Agent 基于这个 System Prompt + Skill Instructions + Tools 去调用 LLM，按照 Skill 规定的步骤完成任务。
  5. 输出校验：AI 回答后，校验其格式是否符合 SKILL.md 中定义的 Output Format（如 Markdown 表格、JSON 结构），若不符合则触发自动重试。

与直接写 Prompt 的区别：

维度 直接写 Prompt Skill
复用性 一次性 可复用、可版本管理
参数化 无，纯文本 有明确输入参数定义
质量保证 无输出校验 自带 Output Format 校验
调度方式 手动触发 SOP 节点可像函数一样调度

追问：如果一个 Skill 执行失败（比如输出了错误格式），会怎样？

“Skill 执行器内置了输出校验器。如果格式与 SKILL.md 中定义的 Output Format 不匹配，会触发两件事：

  1. 自动重试：将错误信息（格式不符的具体部分）反馈给 Agent，要求它重新输出符合格式的内容。最多重试 2 次。
  2. 升级 HITL：重试依然失败，将控制权交给人工，并携带当前的 Skill 上下文和失败记录，让用户快速定位是 Skill 指令不清晰还是 Agent 能力不足。”

Q4: Skill 和 Persona 有什么区别？为什么要分开？

难度：⭐⭐⭐（进阶）
频率：★★★★（常见）

回答结构：

“Persona 定义的是’是谁’——角色身份、核心能力、行为边界。如架构师，他的长期性格是’擅长系统设计，不写代码’。

Skill 定义的是’会什么技能’——一个具体的任务能力，比如’代码审查’、’需求分析’。Skill 可以像一个工具包一样被任何合适的 Agent 临时加载和使用。

分开的好处：

  1. 复用：一个’代码审查’ Skill 可以被架构师、程序员、Reviewer 共同使用，不需要重复定义。
  2. 解耦：Persona 可以独立调整（比如架构师换用更强模型），Skill 可以独立更新（如审查标准变得更严格），互不影响。
  3. 按需加载：Agent 不带着所有技能，只在 SOP 节点需要时才动态加载，避免 Prompt 过长而稀释注意力。

在实际执行中，SOP 节点的绑定是 Agent + Skill，这就是’人选工具’的模式。”

三、Prompt 组装公式

Q5: Agent 的 Prompt 最终是怎么组装的？你们有统一的模板吗？

难度：⭐⭐⭐（进阶）
频率：★★★★（常见）

回答结构：

“我们设计了一个统一的 Prompt 组装公式，保证所有 Agent 调用的一致性：

具体实现上，PromptBuilder 维护了三个核心模板：

模板 注入内容 来源
SYSTEM_TEMPLATE Persona 的身份、可用工具、输出偏好 Persona Registry
CONTEXT_TEMPLATE Memory（内部经验）+ RAG 检索结果（外部知识）+ 任务描述 Memory + RAG + TaskState
SKILL_TEMPLATE 技能指令和输出格式 Skill Registry

这三个模板由统一的 build_full_prompt 方法一次性组装，同时 Context Manager 在前面已经做了 Token 预算裁剪，保证不超过窗口。

统一模板的好处是：所有的 Agent 调用都遵循相同的结构，方便调试（verbose 模式可以看到完整 Prompt 结构），也为后续的 A/B 测试和版本监控打下了基础。”

四、条件路由与循环

Q6: 条件路由的实现细节？表达式求值如何保证安全？

难度：⭐⭐⭐⭐（深入）
频率：★★★（偶见）

回答结构：

“条件路由是 SOP 引擎的核心组件之一，它让工作流能根据 Agent 的输出动态决定下一步。

实现上，ConditionRouter 接收一个表达式和当前的 TaskState，返回下一节点 ID。

表达式解析：表达式格式为 result.score >= 80 或 status == 'approved'。我们没有用 Python 的 eval（太危险），而是实现了一个受限的表达式求值器：

  1. 解析表达式为 AST（抽象语法树）
  2. 只允许白名单中的操作符（>=, <=, >, <, ==, !=, in, contains）
  3. 变量值从 TaskState 的安全字典中获取，禁止任何函数调用或属性访问

路由决策：在 edges 定义中，每个分支有一个 condition 和一个 max_loop 控制循环回退。求值器遍历条件，返回第一个为真的目标节点。如果都不匹配，走默认路径（无 condition 的分支）。

安全层面：AST 解析只允许比较操作和布尔运算，拒绝任何导入、函数执行等操作。测试用 50 个注入攻击用例全部被拦截。”

追问：循环回退怎么防止死循环？

“通过 max_loop 硬限制。TaskState 中有一个 retry_count，每次经过循环边时自增。当 retry_count >= max_loop 时，路由器强制走 failed 或 HITL 分支，并记录告警日志。”

Q7: 条件路由在一个真实的开发 SOP 中是怎么跑的？

难度：⭐⭐（基础）
频率：★★★（常见）

回答结构：

“以代码审查环节为例：

  1. 节点定义：review 节点绑定 Reviewer-01 Agent 和 code-review Skill
  2. 边定义：
    • 如果 review.score >= 80 → 进入下一节点（通过）
    • 如果 review.score < 80 且 retry_count < 3 → 回退到编码节点重新修改
    • 如果 retry_count >= 3 → 进入 HITL 人工审批节点

执行时，审查节点完成后，Router 从 TaskState 中读取 score 字段，求值并跳转。整个过程在聊天室的可视化界面上可实时看到节点亮起和路径变化。

这种设计将工作流逻辑从 Agent 的 Prompt 中抽出来，变成了可配置、可追踪的显式规则，大大提升系统的确定性。”

五、整体策略层评价

Q8: L3 策略层给你带来的最大价值是什么？

难度：⭐（开放）
频率：★★（偶见）

回答结构：

“最大价值是让系统从’能做’变为’会做’。

之前 L1 和 L2 给 Agent 提供了能力和知识，但所有任务一视同仁——简单对话也用最强模型，成本高；任务需要审查时，Agent 还是得靠冗长的 Prompt 指导。

L3 带来的三个关键改变：

  1. 智能路由：省下 60% 的成本，同时不损害质量。
  2. Skill 可执行：Agent 可以像使用工具一样使用 Skill，指令清晰、可复用。
  3. 条件路由：流程从此有了真正的决策能力，不再是线性的。

对于面试来说，这是展示工程深度的地方——不是简单调 API，而是在上面构建了智能决策层。”

第5周面试总结

问题 核心话术 关键词
模型路由？ 任务分类 → 模型选择 → Fallback，成本降低 80% 智能选型
Fallback 链？ 指数退避 + 全局限流 + 熔断器，防止雪崩 鲁棒性
Skill 执行？ 加载定义 → 参数注入 → 组装 Prompt → 校验输出 可复用指令
Skill vs Persona？ Persona 是谁，Skill 会什么，解耦按需组合 角色-技能分离
Prompt 组装？ 统一公式：System + Context + Skill，标准化 模板化
条件路由？ AST 受限求值，拒绝 eval，max_loop 防死循环 安全决策
SOP 中的条件路由？ score >= 80 通过，< 80 回退，>= 3 次 HITL 可视化流程
策略层价值？ 从”能做”到”会做”，成本、质量、决策三维提升 工程深度

一句话总结第5周：

第5周完成了 L3 策略层——模型智能路由让 60% 的简单对话自动用廉价模型（成本降 80%），Skill 系统让指令可复用可校验，条件路由让 SOP 有了真正的决策能力。系统从”所有任务一视同仁”进化为”聪明地选择怎么做”。
