AI-Python-06.1: LangChain 基础

LangChain 是当前最流行的 LLM 应用开发框架，让开发者快速搭建智能体、问答系统、对话机器人、知识库等复杂功能。

1. 什么是 LangChain？

1.1 为什么需要 LangChain？

众所周知，OpenAI 的 API 是无法联网的。如果想实现：

  • 联网搜索并给出回答
  • 总结 PDF 文档
  • 基于某个视频进行问答
  • 连接企业数据库

直接调用 API 是无法实现的。LangChain 就是为了解决这些问题而诞生的。

1.3 直接调用 vs LangChain

对比维度 直接调用API 使用LangChain
开发模式 大量自定义代码 模块化组装
多模型支持 需单独适配 统一接口
对话记忆 手动维护 内置Memory
工具调用 逻辑复杂 Agent自动决策
RAG支持 自行实现 内置检索组件

2. 必知概念

在开始实战之前，先理解几个核心概念。

2.1 Model I/O（模型交互）

Model I/O = 输入格式化 → 模型调用 → 输出解析

    flowchart LR
    A[Prompt\n提示词模板] --> B[Model\n模型调用]
    B --> C[Output Parser\n输出解析]

    A -->|格式化输入| B
    B -->|原始响应| C
    C -->|结构化数据| D[应用]

2.1.1 三种模型类型

LangChain 支持三种不同的模型类型，设计理念各不相同：

1. LLMs（纯文本模型）

设计理念： 最基础的模型接口，模拟传统的文本补全行为。

    flowchart LR
    A[字符串输入] --> B[LLM]
    B --> C[字符串输出]

工作方式：

  • 输入一段文本，模型预测”接下来最可能出现的文本”
  • 类似 GPT-3 的原始行为（instruct 系列模型）
  • 没有对话上下文概念，每次调用独立

适用场景： 文本补全、简单生成任务、翻译

2. Chat Models（对话模型）

设计理念： 专为对话交互设计，理解消息的”角色”概念。

    flowchart TD
    subgraph 输入
        S[SystemMessage: 设定角色]
        H[HumanMessage: 用户问题]
        A[AIMessage: 历史回复]
    end

    输入 --> M[Chat Model]
    M --> R[AIMessage: 新回复]

三种消息角色：

角色 职责 示例
System 设定 AI 的行为边界和人设 “你是Python专家，回答简洁”
Human 用户的输入内容 “什么是装饰器？”
AI 模型回复（含历史） “装饰器是一种设计模式…”

核心优势：

  • 支持多轮对话上下文
  • 可精确控制 AI 行为（通过 System Prompt）
  • 返回丰富的元数据（Token 统计、模型信息等）

3. Embedding Models（嵌入模型）

设计理念： 将文本转换为数学向量，让计算机”理解”语义。

    flowchart LR
    A[文本: 机器学习] --> B[Embedding Model]
    B --> C[向量: 0.12, -0.34, ...]

    D[文本: 人工智能] --> E[Embedding Model]
    E --> F[向量: 0.11, -0.32, ...]

    C --> G[相似度计算]
    F --> G
    G --> H[语义相近]

核心用途：

  • 语义搜索：找到意思相近的文档，而非关键词匹配
  • RAG 检索：从知识库中找出与问题最相关的内容
  • 聚类分析：自动将相似文本分组

2.1.2 三种模型对比

维度 LLMs Chat Models Embedding Models
输入格式 纯字符串 消息列表（带角色） 字符串
输出格式 纯字符串 消息对象 + 元数据 向量（浮点数组）
上下文 无 支持多轮对话历史 无
核心能力 文本补全 对话交互 语义表示
典型用途 补全、翻译 问答、Agent 检索、相似度
代表模型 GPT-3 instruct GPT-4、Claude text-embedding-3

2.1.3 如何保证输出格式稳定？

LLM 的输出本质是不确定的，但生产环境需要稳定的结构化数据。

    flowchart TD
    A[LLM 输出] --> B{格式是否正确?}
    B -->|是| C[解析成功]
    B -->|否| D[解析失败]

    D --> E[重试/修正]
    E --> A

    subgraph 解决方案
        F[PydanticOutputParser]
        G[with_structured_output]
        H[JsonOutputParser]
    end

方法1：PydanticOutputParser（推荐生产使用）

原理： 通过 Pydantic 定义数据结构，自动生成”格式指令”注入到 Prompt 中。

    sequenceDiagram
    participant U as 用户
    participant P as Parser
    participant L as LLM

    U->>P: 定义 Person name age skills
    P->>P: 生成格式指令
    P->>L: Prompt + 格式指令
    L->>P: JSON 字符串
    P->>P: 验证 + 解析
    P->>U: Person 对象

特点：

  • ✅ 类型安全，自动验证
  • ✅ 字段可添加描述，引导模型输出
  • ✅ 解析失败可自动重试
  • ❌ 需要学习 Pydantic 语法

方法2：with_structured_output（最简单）

原理： 利用模型原生的 Function Calling / Tool Use 能力，强制输出符合 Schema。

    flowchart LR
    A[Pydantic Schema] --> B[LLM.with_structured_output]
    B --> C[Function Calling]
    C --> D[结构化对象]

特点：

  • ✅ 代码最简洁，一行搞定
  • ✅ 准确率最高（原生支持）
  • ❌ 依赖模型支持（OpenAI、Claude 支持，部分模型不支持）

方法3：JsonOutputParser（轻量级）

原理： 在 Prompt 中手动说明格式要求，解析 JSON 字符串。

特点：

  • ✅ 轻量级，无额外依赖
  • ✅ 适用于不支持 Function Calling 的模型
  • ❌ 无自动验证，可能解析失败
  • ❌ 需要手动编写格式说明

2.1.4 三种格式化方法对比

方法 准确率 易用性 兼容性 推荐场景
PydanticOutputParser ⭐⭐⭐⭐ ⭐⭐⭐ ⭐⭐⭐⭐⭐ 生产环境、复杂数据
with_structured_output ⭐⭐⭐⭐⭐ ⭐⭐⭐⭐⭐ ⭐⭐⭐ 新项目、简单结构
JsonOutputParser ⭐⭐⭐ ⭐⭐⭐⭐ ⭐⭐⭐⭐⭐ 本地模型、快速原型

选择建议：

  • OpenAI/Claude → 用 with_structured_output
  • 本地模型/老模型 → 用 PydanticOutputParser
  • 快速测试 → 用 JsonOutputParser

2.2 Prompt Template（提示词模板）

核心作用： 将提示词模板化，实现复用、变量填充和统一管理。

2.2.1 为什么需要模板？

    flowchart LR
    subgraph 硬编码方式
        A1[写死提示词] --> B1[每次都要改代码]
        B1 --> C1[维护困难]
    end

    subgraph 模板方式
        A2[定义模板] --> B2[传入变量]
        B2 --> C2[复用 易维护]
    end

模板的核心价值：

  • 复用性：同一模板可用于不同场景
  • 可维护性：修改模板无需改业务代码
  • 一致性：确保团队使用统一的提示词风格
  • 变量化：动态填充内容，灵活适配

2.2.2 四种模板类型

模板类型 适用场景 特点
PromptTemplate 简单文本提示 字符串模板，单轮输入
ChatPromptTemplate 对话模型 支持多角色消息（System/Human/AI）
FewShotPromptTemplate 少样本学习 包含示例，引导模型输出风格
PipelinePromptTemplate 复杂组合 多个模板组合成完整提示

2.2.3 PromptTemplate（基础模板）

设计理念： 最简单的字符串模板，通过 {变量名} 占位符实现变量替换。

    flowchart LR
    A[模板: 你是role 请action] --> B[变量: role=专家 action=解释]
    B --> C[结果: 你是专家 请解释]

适用场景：

  • 单轮文本输入
  • 简单的问答场景
  • 不需要区分消息角色

2.2.4 ChatPromptTemplate（对话模板）

设计理念： 为 Chat Model 设计，支持多角色消息结构。

    flowchart TD
    subgraph ChatPromptTemplate
        S[SystemMessage: 设定角色]
        H[HumanMessage: 用户问题]
        A[AIMessage: 历史回复]
        P[Placeholder: 动态插入]
    end

    ChatPromptTemplate --> M[Chat Model]
    M --> R[AIMessage: 新回复]

消息类型说明：

消息类型 作用 使用时机
("system", "...") 设定 AI 行为和人设 每次调用必传
("human", "...") 用户输入 每次调用必传
("ai", "...") 历史回复（上下文） 多轮对话时传入
MessagesPlaceholder 动态消息列表 对话历史、Few-shot 示例

核心优势：

  • 可精确控制 System Prompt（影响模型行为最关键的部分）
  • 支持对话历史注入（实现多轮对话）
  • 与 Chat Model 原生接口完美匹配

2.2.5 FewShotPromptTemplate（少样本模板）

设计理念： 通过提供示例，让模型”模仿”期望的输出格式和风格。

    flowchart TD
    subgraph FewShot结构
        E1[示例1: 输入-输出]
        E2[示例2: 输入-输出]
        E3[示例3: 输入-输出]
    end

    FewShot结构 --> P[实际问题]
    P --> M[LLM 模仿输出]

工作原理：

阶段 内容
示例 3-5个典型的输入输出对
前缀 任务描述（可选）
后缀 实际问题 + 格式提示
分隔符 区分示例和实际问题

适用场景：

  • 需要特定输出格式
  • 模型难以理解的任务
  • 风格迁移（如模仿某种写作风格）

Few-Shot vs Zero-Shot：

方式 示例数量 准确率 Token 消耗
Zero-Shot 0 较低 最少
Few-Shot 3-5 较高 较多
Many-Shot 10+ 最高 最多

2.2.6 模板选择决策

    flowchart TD
    A[需要提示词模板] --> B{使用什么模型?}

    B -->|LLM 文本模型| C[PromptTemplate]
    B -->|Chat 对话模型| D{需要示例吗?}

    D -->|是| E[FewShotPromptTemplate]
    D -->|否| F[ChatPromptTemplate]

    C --> G{需要组合多个模板?}
    F --> G
    E --> G

    G -->|是| H[PipelinePromptTemplate]
    G -->|否| I[单一模板即可]

选择建议：

场景 推荐模板
简单问答 PromptTemplate
多轮对话 ChatPromptTemplate
格式化输出 FewShotPromptTemplate
复杂工作流 PipelinePromptTemplate
