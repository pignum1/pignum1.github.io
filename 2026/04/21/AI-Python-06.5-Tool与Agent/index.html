AI-Python-06.5: Tool 与 Agent

LLM 只会”说话”，Tool 让它能”做事”，Agent 让它自主决定”用什么工具做事”。本章深入理解 Tool 定义、Function Calling 机制和 Agent 的决策原理。

1. 为什么需要 Tool？

1.1 LLM 的局限

LLM 本质上只会生成文本，它不能：

  • 搜索网络获取实时信息
  • 执行精确的数学计算
  • 查询数据库
  • 调用外部 API
  • 读写文件

1.2 Tool 的作用

Tool = 给 AI 装上”手脚”

2. Function Calling 机制

在学 Tool 之前，必须理解底层的 Function Calling（也叫 Tool Use）。

2.3 完整流程

    sequenceDiagram
    participant U as 用户
    participant A as Agent
    participant L as LLM
    participant T as Tool

    U->>A: 135×246等于多少？
    A->>L: 用户消息 + 可用工具列表
    L->>A: tool_calls: multiply(135, 246)
    A->>T: 执行 multiply(135, 246)
    T->>A: 33210
    A->>L: 工具结果: 33210
    L->>A: 135 × 246 等于 33210
    A->>U: 135 × 246 等于 33210

关键：LLM 不是自己算，而是告诉你的代码”该调什么函数、传什么参数”。

3. 定义 Tool 的三种方式

3.1 @tool 装饰器（最常用）

LLM 看到的工具描述：

三个关键要素：

要素 作用 来自哪里
工具名 LLM 用名字调用工具 函数名 multiply
工具描述 LLM 判断”什么时候”该用 docstring 文档字符串
参数签名 LLM 知道传什么参数 类型注解 a: int, b: int

3.2 StructuredTool（更灵活）

适用场景： 参数有默认值、需要复杂验证。

3.3 三种方式对比

方式 代码量 灵活性 适用场景
@tool 最少 中等 大多数场景首选
StructuredTool 较多 最高 复杂参数、默认值

    TC2 -->|是| EXE
    TC2 -->|否| ANS[生成最终回答]
    TC -->|否| ANS

6.1 docstring 是关键

6.2 参数命名要清晰

6.3 不抛异常，返回错误描述

原因： LLM 看到错误信息后可以调整策略（换参数重试、换工具等），而异常会导致整个流程中断。
