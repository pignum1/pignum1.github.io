AI-Python-07.1: RAG 检索增强与向量检索

RAG（Retrieval-Augmented Generation）让 LLM 在回答问题前先”查资料”，解决知识过时、幻觉和缺乏私有数据三大痛点。本章从 RAG 架构全流程到 Embedding 原理、向量数据库实战，构建完整的 RAG Pipeline。

1. 什么是 RAG？

1.1 为什么需要 RAG？

LLM 有三大固有局限：

局限 问题 RAG 如何解决
知识过时 训练数据有截止日期，不知道最新信息 检索实时文档/网页
幻觉 编造看似合理但错误的信息 基于真实文档生成回答
缺乏私有数据 不知道企业内部知识 连接内部知识库

1.2 RAG vs Fine-tuning

维度 RAG Fine-tuning
知识更新 实时更新文档即可 需要重新训练
成本 低（检索 + 推理） 高（GPU 训练）
可解释性 高（可以展示来源文档） 低（知识融入权重）
私有数据 天然支持 需构建训练集
适用场景 问答、知识库、客服 风格迁移、领域适应

选择建议： 需要”查资料再回答”用 RAG，需要”改变模型行为风格”用 Fine-tuning，两者可组合使用。

3. Embedding 原理

3.1 什么是 Embedding？

Embedding = 将文本转换为一组数字（向量），使得语义相近的文本在向量空间中距离更近。

8. 构建RAG常见问题清单

实际构建 RAG 系统时最常踩的 7 个坑，以及对应的解决方案。

8.1 文档解析质量差

现象： PDF 表格变成 姓名部门工资张三技术25000 一坨文本，语义被破坏。

解决： 用 pdfplumber/Docling 保留表格结构，转成 Markdown 表格再入库。（详见 07.2）

8.2 切分把完整意思切断

现象： “公司规定报销需要满足以下条件：第一，必须是” 在一个 Chunk，”入职满三个月的正式员工” 在下一个 Chunk，检索到前者但缺少关键信息。

解决：

  • chunk_overlap 设 10%，让相邻块有重叠
  • 按段落/句子边界切分，别在句中间切
  • 使用语义切分器（SemanticChunker），按意思变化点切

8.3 检索出来不相关

现象： 用户问”如何部署模型”，检索出来的是”模型训练方法”——向量距离近但答非所问。

根因： 纯向量检索只看”语义相似”，不看”是否真能回答问题”。

解决：

  • 混合检索：向量 + BM25 关键词，关键词保证精确匹配
  • Rerank 重排序：粗检索 Top-20，Cross-Encoder 精排到 Top-3
  • 加 metadata 过滤：先按类型/部门/时间过滤，再在子集中检索

8.4 用户问题太短/太模糊

现象： 用户问”怎么配？”，向量模型完全不知道在问什么，检索结果随机。

解决： 用 LLM 做 Query 改写，把模糊问题改写成明确的检索 query。

8.5 LLM 不听话，仍然幻觉

现象： 明明给了正确文档，LLM 还是编造文档里没有的信息，或者把多篇文档的内容错误拼接。

解决：

  • 强约束 Prompt：”只基于以下参考资料回答，不要添加任何资料中没有的信息。如果无法回答，请说’根据已有资料无法回答’”
  • 引用溯源：要求回答时标注来源，如 [文档A-第3页]
  • 置信度检查：用第二个 LLM 检查回答是否与原文一致

8.6 知识库更新了但检索到旧内容

现象： 文档已经更新，但向量库里还有旧版本的向量，检索到过时信息。

解决：

  • 给每个 chunk 加 metadata={"doc_id": "xxx", "version": "v2"}
  • 更新时先删除旧版本再添加：delete(filter={"doc_id": "xxx"}) → add_documents()
  • 定期清理过期数据

8.7 分不清”该不该用知识库”

现象： 用户问”你好”或”写一首诗”，系统还去知识库检索，浪费资源。

解决： 意图路由——先用小模型/分类器判断是否需要检索，闲聊类直接 LLM 回答，知识类走检索流程。
