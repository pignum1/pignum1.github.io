AI-Python-06.2: Prompt 模板与输出解析

LangChain 与模型交互的核心流程：输入格式化（Prompt Template）→ 模型调用 → 输出解析（Output Parser）。本章从”输入”到”输出”完整讲透。

1. 完整数据流：从输入到输出

    flowchart LR
    A[Prompt Template\n输入模板] --> B[Chat Model\n模型调用]
    B --> C[Output Parser\n输出解析]

    A -->|结构化输入| B
    B -->|原始 AIMessage| C
    C -->|业务数据| D[应用]
阶段 组件 做什么 输入 输出
输入格式化 PromptTemplate / ChatPromptTemplate 把变量填入模板，生成结构化消息 {"role": "专家", "question": "xxx"} [SystemMessage, HumanMessage]
模型调用 ChatOpenAI 发给 LLM，获取回复 消息列表 AIMessage 对象
输出解析 StrOutputParser / JsonOutputParser / PydanticOutputParser 从 AIMessage 提取业务数据 AIMessage 字符串 / JSON / Pydantic 对象

2. 输入端：Prompt Template 详解

2.1 PromptTemplate（纯文本模板）

适用场景： 简单的单轮文本生成，不需要区分消息角色。

定义方式一：from_template（自动推断变量）

定义方式二：手动声明变量（更严格）

定义方式三：partial_variables 设默认值

适用场景： language、style、analysis_type 这类”基本不变”的参数，用默认值减少重复代码。

模板变量的约束规则

规则 正确 错误
变量名不能有空格 {product} {product name}
变量名必须合法标识符 {user_input} {user-input}
字面花括号用 {{` `}} 转义 "输出JSON {{key}}" "输出JSON {key}"
必填变量不传会报错 全部传入 漏传 → ValidationError

模板格式：f-string vs jinja2

语法 适用场景 占位符写法
f-string 简单变量替换（默认） {var}
jinja2 循环、条件、复杂逻辑 {{ var }}, {% for %}

2.2 ChatPromptTemplate（对话模板，最常用）

适用场景： 多轮对话、需要区分 System/Human/AI 角色。

基本用法

三种消息角色的控制力

角色 控制什么 示例
("system", ...) AI 的行为边界、人设、约束 “你只能回答Python相关问题”
("human", ...) 用户的输入内容 “什么是装饰器？”
("ai", ...) 历史回复（注入对话记忆） “装饰器是一种设计模式…”

MessagesPlaceholder：动态插入消息列表

生成的结果等价于：

MessagesPlaceholder 的典型用途：

用途 variable_name 说明
对话历史 "history" 多轮对话时注入之前的消息
Few-shot 示例 "examples" 注入示例引导模型输出格式
Agent 思考记录 "agent_scratchpad" Agent 的 Thought/Action/Observation 循环记录

2.3 FewShotPromptTemplate（少样本模板）

核心思想： 给模型看几个”输入→输出”的示例，让它模仿你的格式。

输出：

模型看到前面的示例，就会模仿 {input} → {output} 的格式来回答。

方式 示例数量 准确率 Token 消耗
Zero-Shot 0 较低 最少
Few-Shot 3-5 较高 较多
Many-Shot 10+ 最高 最多

2.4 四种模板选择决策

    flowchart TD
    A[需要提示词模板] --> B{使用什么模型?}

    B -->|LLM 纯文本模型| C[PromptTemplate]
    B -->|Chat 对话模型| D{需要示例引导吗?}

    D -->|是| E[FewShotPromptTemplate]
    D -->|否| F[ChatPromptTemplate]

    C --> G{需要组合多个模板?}
    F --> G
    E --> G

    G -->|是| H[PipelinePromptTemplate]
    G -->|否| I[单一模板即可]
场景 推荐模板
简单问答、单轮文本 PromptTemplate
多轮对话、需要控制 AI 行为 ChatPromptTemplate
需要特定输出格式、风格模仿 FewShotPromptTemplate
复杂工作流、多模板组合 PipelinePromptTemplate

3. 输出端：Output Parser 详解

3.1 为什么需要 Output Parser？

LLM 返回的是 AIMessage 对象，不是直接可用的数据：

而且 LLM 的输出是不确定的，生产环境需要稳定的结构化数据（JSON、对象）。

Output Parser 解决的问题：

问题 Parser 的作用
AIMessage 对象太复杂 提取 .content 得到纯文本
输出格式不稳定 强制解析为 JSON / Pydantic 对象
解析失败怎么办 自动重试或报清晰错误

3.2 StrOutputParser（最基础，提取纯文本）

作用： 从 AIMessage 中提取 .content，返回纯字符串。

数据流：

    flowchart LR
    A[ChatPromptTemplate] --> B[ChatOpenAI]
    B --> C[AIMessage\ncontent=GIL是...\nresponse_metadata=...]
    C --> D[StrOutputParser]
    D --> E[GIL是全局解释器锁...]

3.3 JsonOutputParser（解析 JSON）

作用： 要求模型输出 JSON 格式，并自动解析为 Python 字典。

关键方法 get_format_instructions()： 自动生成一段”格式说明”文本，注入到 Prompt 中告诉模型应该输出什么格式的 JSON。

3.4 PydanticOutputParser（推荐生产使用）

作用： 用 Pydantic 模型定义输出结构，自动生成格式指令 + 自动验证 + 自动解析。

3.5 with_structured_output（最简单的方式）

作用： 利用模型原生的 Function Calling 能力，一行代码实现结构化输出。

不需要手动写 Parser，不需要手动注入格式指令！ 模型原生支持。

3.6 三种输出解析方式对比

    flowchart TD
    A[需要结构化输出] --> B{模型支持 Function Calling?}

    B -->|是 OpenAI Claude等| C[with_structured_output\n最简单，准确率最高]
    B -->|否 不确定| D{需要自动验证?}

    D -->|是| E[PydanticOutputParser\n类型安全，自动校验]
    D -->|否| F[JsonOutputParser\n轻量级，快速原型]

    G[只需要纯文本] --> H[StrOutputParser\n提取 .content]
方式 准确率 代码量 兼容性 推荐场景
StrOutputParser - 最少 全部 纯文本问答
with_structured_output 最高 最少 OpenAI/Claude 新项目首选
PydanticOutputParser 高 中等 全部模型 生产环境、复杂结构
JsonOutputParser 中等 少 全部模型 快速测试、本地模型

6. 总结

输入端速查

模板 用法 适用场景
PromptTemplate.from_template("...") 自动推断变量 简单文本
PromptTemplate(template=..., input_variables=[...]) 手动声明变量 团队协作、严格校验
partial_variables={"key": "value"} 设默认值 减少重复传参
ChatPromptTemplate.from_messages([...]) 多角色消息 对话场景（最常用）
MessagesPlaceholder("history") 动态消息列表 多轮对话历史
FewShotPromptTemplate 带示例引导 格式化输出

输出端速查

Parser 输出类型 适用场景
StrOutputParser str 纯文本问答
JsonOutputParser dict 快速获取 JSON
PydanticOutputParser Pydantic 对象 生产环境、类型安全
with_structured_output Pydantic 对象 最简单（需模型支持）
