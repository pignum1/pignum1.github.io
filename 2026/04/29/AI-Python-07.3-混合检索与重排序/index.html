AI-Python-07.3: 混合检索与重排序

纯向量检索精度有限，本章讲解 BM25 关键词检索 + 向量检索的混合策略、Cross-Encoder 重排序、Query 改写，以及 RAGAS 评测体系，把 RAG 检索精度从 60 分提升到 90 分。

1. 为什么纯向量检索不够？

1.1 纯向量检索的三个盲区

盲区 例子 原因
专有名词匹配差 搜”BERT模型”漏掉含”BERT”的文档 向量模型把”BERT”和”Transformer”看成近义词，反而匹配到不相关内容
精确匹配弱 搜”订单号 ORD-20260428”找不到 向量语义理解把精确编号”理解”成了别的意思
领域术语不准 搜”KV Cache”找不到对应段落 通用 Embedding 模型对技术术语理解有限

1.2 解法：混合检索（Hybrid Search）

    flowchart TB
    Q[用户问题] --> V[向量检索\n语义匹配]
    Q --> K[BM25 检索\n关键词匹配]
    V --> M[结果融合\nReciprocal Rank Fusion]
    K --> M
    M --> R[Rerank 重排序]
    R --> TOP[Top-K 最终结果]

    style Q fill:#fff3cd,stroke:#ffc107,stroke-width:2px,color:#333
    style V fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333
    style K fill:#d1ecf1,stroke:#17a2b8,stroke-width:2px,color:#333
    style R fill:#ffd1d1,stroke:#e57373,stroke-width:2px,color:#333
    style TOP fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333

核心思想： 向量检索擅长”理解意思”，BM25 擅长”精确匹配”。两者互补，融合后精度大幅提升。

2. BM25 关键词检索

2.1 BM25 原理

BM25 是经典的基于词频的检索算法，核心思想：一个词在文档中出现越多、在整个语料库越少见，这个词对这篇文档越重要。

因子 含义 效果
TF（词频） 词在文档中出现几次 出现越多越相关
IDF（逆文档频率） 词在所有文档中多罕见 “的””是”这种常见词权重低，”BERT””KV Cache”这种罕见词权重高
长度归一化 文档越长，TF 越打折 防止长文档靠字数优势占便宜

2.2 LangChain 使用 BM25

BM25 的优势： 精确匹配关键词，不会像向量检索那样被”语义相近”误导。

BM25 的劣势： 不理解同义词——搜”缓存”找不到只写了”Cache”的文档。

3. 混合检索：向量 + BM25 融合

3.1 融合策略：RRF（Reciprocal Rank Fusion）

两种检索各返回一个列表，如何合并？最常用的是 RRF：

举例：

文档 向量检索排名 BM25 排名 RRF 得分
Doc A 1 3 1/(60+1) + 1/(60+3) = 0.0164 + 0.0159 = 0.0323
Doc B 2 1 1/(60+2) + 1/(60+1) = 0.0161 + 0.0164 = 0.0325
Doc C 3 2 1/(60+3) + 1/(60+2) = 0.0159 + 0.0161 = 0.0320

排序：Doc B > Doc A > Doc C（两种检索都认可的文档排名最高）

3.2 混合检索实现

3.3 向量 vs BM25 vs 混合 对比

特性 向量检索 BM25 混合检索
同义词理解 强 弱 强
精确匹配 弱 强 强
领域术语 一般 强 强
长尾查询 较好 差 较好
实现复杂度 低 低 中

4. 重排序（Rerank）

4.1 为什么需要 Rerank？

混合检索解决了”召回”问题，但排序精度还不够。Rerank 用 Cross-Encoder 对少量候选做精排。

    flowchart LR
    Q[用户问题] --> H[混合检索\n粗召回]
    H --> |Top-20| R[Cross-Encoder\n精排]
    R --> |Top-3| L[LLM 生成]

    style H fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333
    style R fill:#ffd1d1,stroke:#e57373,stroke-width:2px,color:#333
    style L fill:#e6f7e6,stroke:#4caf50,stroke-width:2px,color:#333

为什么不一上来就用 Cross-Encoder？ 太慢——每个候选文档都要和问题拼起来过一遍 BERT。100 万文档跑 100 万次不现实。但 20 个候选只需跑 20 次，完全可接受。

5.1 为什么需要改写？

原始问题 问题 改写后

“那个参数” 不知道哪个参数 “FlashAttention 的 block_size 参数设置”

6. RAGAS 评测体系

6.1 为什么需要评测？

没有评测就无法知道 RAG 效果好不好，更不知道该优化哪个环节。

6.2 核心指标

指标 衡量什么 怎么算
Faithfulness（忠实度） 回答是否忠于检索到的文档 检查回答中的每个声明是否都能在文档中找到依据
Answer Relevancy（答案相关性） 回答是否切题 回答与问题的语义相似度
Context Precision（上下文精确度） 检索到的文档中有多少是相关的 相关文档在检索结果中的排名
Context Recall（上下文召回率） 应该检索到的文档是否都检索到了 标准答案中的信息是否都能在检索结果中找到

6.3 评测示例

6.4 评测驱动的优化闭环

    flowchart LR
    A[构建 RAG] --> B[RAGAS 评测]

    C -->|Faithfulness 低| D[优化 Prompt\n加强约束]
    C -->|Context Precision 低| E[加 Rerank\n优化检索]
    C -->|Context Recall 低| F[调整 Chunk 大小\n加混合检索]
    C -->|Answer Relevancy 低| G[Query 改写\n优化 Prompt]
    D --> A
    E --> A
    F --> A
    G --> A

指标 低了怎么办
