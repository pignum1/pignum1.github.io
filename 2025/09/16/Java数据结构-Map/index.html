Java 集合框架：Map 接口详解

在 Java 集合框架中，Map 接口代表了一种键值对 (key-value) 的映射关系。它与 Collection 接口不同，Map 存储的是两组成员：一组是唯一的键 (Key)，另一组是可重复的值 (Value)。每个键最多只能映射到一个值。

1. Map 接口的核心特性

  • 键值对存储: Map 中的每个元素都包含一个键和一个值。
  • 键的唯一性: Map 不允许有重复的键。如果你用一个已经存在的键去存入一个新的值，新的值会覆盖旧的值，put 方法会返回旧的值。
  • 值的可重复性: Map 中的值可以重复。
  • 非 Collection 子接口: Map 自身是一个顶层接口，它不继承自 Collection 接口。

2. HashMap：最常用的 Map 实现

HashMap 是 Map 接口最核心和最常用的实现类，它提供了 O(1) 的平均时间复杂度的性能。

内部原理 (JDK 8+)

HashMap 的成功关键在于其高效的哈希算法和精妙的存储结构。其底层结构可以概括为数组 + 链表 + 红黑树。

哈希算法与散列分布

HashMap 如何为任意的 key 计算出它在数组中的存储位置？这个过程分为三步，目的是让键值对尽可能均匀地散列在数组中，以减少哈希冲突。

    graph LR

    B -- "2. 高位扰动 (Disturbance Function)<br/>h ^ (h >>> 16)" --> C["最终哈希值<br/>混合高16位和低16位"] 
    C -- "3. 计算索引 (Bitwise AND)<br/>(n - 1) & hash" --> D["数组索引<br/>n 为数组长度 (必须是2的幂)"]

    style B fill:#E0F2F1,stroke:#009688
    style C fill:#FFFDE7,stroke:#FBC02D
    style D fill:#FBE9E7,stroke:#FF5722

  1. 获取原始哈希值: 首先，调用 key 对象的 hashCode() 方法，获得一个32位的 int 型哈希值。

  2. 高位扰动/混合: HashMap 并不直接使用这个原始哈希值。它将原始哈希值 h 右移16位，然后将原始哈希值与右移后的值进行异或(XOR)运算。即 (h ^ (h >>> 16))。这样做的目的是将哈希值的高16位信息混合到低16位中，使得即使 hashCode() 实现得不够好，也能在后续计算中产生更均匀的散列效果，减少冲突。

  3. 计算数组索引: HashMap 的数组长度 n 始终是2的幂。因此，它可以通过 (n - 1) & hash 这个位运算来代替取模运算（hash % n），效率极高。因为 n 是2的幂，所以 n-1 的二进制表示就是一串连续的1（例如 n=16, n-1=15, 二进制为 0000...1111），这确保了 & 运算的结果范围总是在 [0, n-1] 之内。

深入理解：为什么 (n - 1) & hash 等价于 hash % n？

我们的目标是做取模运算（求余数），比如 hash % 16，结果范围是 0 到 15。计算机做 % 运算的效率较低，而 & 位运算是计算机最快的运算之一。HashMap 的设计者发现，当 n 是2的幂时，hash % n 和 (n - 1) & hash 的结果是完全等价的。

关键在于二进制：

  1. 当 n 是 2 的幂，比如 n = 16，其二进制是 10000。
  2. 那么 n - 1 (也就是 15) 的二进制就是 01111。这个 01111 就成了一个完美的**“低位掩码”**。

案例：hash = 25, n = 16

  • 取模运算 (%): 25 % 16 = 9
  • 位运算 (&):
    • 25 的二进制是 0001 1001
    • 15 的二进制是 0000 1111
    • 进行 & 运算：

    两者结果完全相同！& 15 这个操作，就像一把“剪刀”，把 25 的二进制 00011001 中，所有超出后四位的部分（高位）都“剪掉”了，只保留了后四位 1001，也就是 9。这和取余的本质是一样的。

什么是哈希碰撞？

理想情况下，每个键都映射到数组中一个唯一的索引。但实际上，两个不同的键（如 "Aa" 和 "BB"，它们的 hashCode 往往相同）通过哈希算法计算后，可能会得到完全相同的数组索引。这种情况就叫做 哈希碰撞 (Hash Collision)。HashMap 使用 “链地址法” 来解决这个问题。

  • 数组 (Buckets): HashMap 内部维护一个 Node[] 类型的数组，是其主体结构。
  • 链表: 如果多个不同的键计算出的索引位置相同（哈希冲突），这些键值对会以链表的形式存储在该哈希桶中。
  • 红黑树: 为了优化极端情况下的性能，从 JDK 8 开始，当某个哈希桶中的链表长度超过一个阈值（默认为 8）并且数组的总容量大于 64 时，这个链表会转化为一个红黑树。这样做可以将该位置的查找时间复杂度从 O(n) 优化到 O(log n)。

性能特点

  • 增删查改快: 对于 put, get, remove 等操作，平均时间复杂度都是 O(1)。
  • 无序: 它不保证键值对的任何特定顺序。
  • 线程不安全: 在多线程环境下使用需要手动同步或使用 ConcurrentHashMap。
源码解析：put 方法核心流程

HashMap 的 put(K key, V value) 方法其实是一个“门面”，它内部直接调用了核心的 putVal(...) 方法。下面我们来逐行拆解 putVal 的实现逻辑。

第 1 步：初始化哈希表

  • 逻辑： 检查内部的数组 table (哈希桶) 是否为空。如果是，说明这是第一次 put 操作，此时会调用 resize() 方法进行初始化，创建一个默认容量为 16 的数组。

第 2 步：计算索引，处理无冲突情况

  • 逻辑： 计算出键值对应当存放在数组中的哪个索引位置 i，并检查这个桶是否为空。如果是空的（无哈希冲突），则直接创建一个新节点 Node 并放入。这是最理想、最快的情况。

第 3 步：处理哈希冲突
如果上一步的 p != null，说明该位置已被占用，代码进入 else 块。

  • 3a： 检查桶里第一个节点 p 的 key 是否和当前要放入的 key 完全相同。如果是，直接用 e = p 记录下这个节点，后续直接更新它的 value。

  • 3b： 如果不是，则检查它是不是一个 TreeNode。如果是，说明这个桶已经是一个红黑树了，就调用红黑树专属的 putTreeVal 方法把新节点放进去。

  • 3c： 如果还不是红黑树，那它就是一个普通的链表。遍历这个链表，如果在末尾还没找到相同的 key，就把新节点链接到末尾，并检查是否需要“树化”；如果在中途找到了，就记录下来并跳出循环。

    将链表转化为红黑树，需要同时满足以下 两个 条件：

    1. 链表长度达到阈值： 链表中的节点数必须大于等于 TREEIFY_THRESHOLD（默认为 8）。
    2. 哈希表容量达到阈值： HashMap 内部的数组 table 的长度必须大于等于 MIN_TREEIFY_CAPACITY（默认为 64）。

    为什么需要第二个条件？
    如果只满足了条件1，但 HashMap 的总容量还很小（< 64），HashMap 会认为当前更紧要的任务是通过扩容 resize() 来减少哈希冲突，而不是进行树化。只有当总容量足够大时，单个桶的超长链表才被认为是“值得”树化的。

第 4 步：更新值并返回

  • 逻辑： 如果经过第 3 步后，e 不为 null，说明在表中找到了一个 key 完全相同的节点 e。此时，用新的 value 覆盖 e 的旧 value，并把 oldValue 返回。

第 5 步：收尾工作

  • 逻辑： 如果代码能执行到这里，说明之前没有找到相同的 key，并且成功插入了一个全新的节点。
    1. modCount++：修改次数加一。
    2. if (++size > threshold)：HashMap 的总元素数量 size 加一，并检查是否超过了扩容阈值 threshold。如果超过，就调用 resize() 方法对整个哈希表进行扩容。
    3. return null：因为是新插入的，没有旧值，所以返回 null。
深入扩容：resize 方法

resize() (扩容) 是 HashMap 的另一个核心机制，它在哈希表容量不足时被调用，以保证性能。这是一个高成本操作，但其内部实现非常巧妙。

resize() 的触发条件

  1. 初始化创建： 第一次调用 put 时，哈希桶数组 table 为 null，此时会调用 resize() 创建初始数组（默认容量16）。
  2. 元素数量超过阈值（最常见）： 成功插入一个新元素后，若 size > threshold (容量 * 负载因子)，则必须调用 resize() 进行扩容。
  3. 树化前的检查： 当链表长度达到8，但数组总容量小于64时，HashMap 会放弃树化，转而优先进行 resize() 扩容。

resize() 的工作流程 (JDK 8+)

  1. 创建新数组： 创建一个容量是原先两倍的新哈希桶数组 newTab。

  2. 遍历旧数组并迁移元素： 遍历旧数组 oldTab 的每一个哈希桶，将其中的节点迁移到新数组。最巧妙之处在于，HashMap 并不需要为每个元素重新计算哈希索引。

  3. 高效迁移算法： 由于容量是2的幂，一个旧桶里的所有元素在扩容后，只可能分布到新数组的 两个 位置上：原索引位置 j 和 j + oldCapacity (原索引 + 旧容量)的位置。
    HashMap 通过检查元素 hash 值的特定 bit 位 (e.hash & oldCap) 来决定：

    • 如果结果为 0，元素在新数组中的索引保持不变。
    • 如果结果不为 0，元素的索引就变为 原索引 + 旧容量。
      通过这种方式，HashMap 只需将一个旧桶的链表拆分成两个子链表，然后分别移动到新数组的对应位置即可，极大地提高了扩容效率。
    假设 `oldCapacity` = 16，扩容后 `newCapacity` = 32。现在要迁移 `index = 5` 的桶，里面有4个元素（Key A, B, C, D）。

**关键技巧：** `resize` 只需检查每个元素 `hash` 值和 `oldCapacity` (16, 二进制 `0001 0000`) 的 `&` 运算结果。

| Key | Hash 值 | 二进制表示 | `hash & 16` 的结果 | 新位置 |

    | :– | :— | :— | :— | :— |
    | A | 5 | 0000 0101 | ... & 0001 0000 = 0 | 不变 (low) |
    | B | 21 | 0001 0101 | ... & 0001 0000 = 16 | 改变 (high) |
    | C | 37 | 0010 0101 | ... & 0001 0000 = 0 | 不变 (low) |
    | D | 53 | 0011 0101 | ... & 0001 0000 = 16 | 改变 (high) |

    **迁移过程：**
1.  `resize` 方法遍历旧桶中的链表，根据 `hash & 16` 的结果，将链表拆分成两个子链表：`low` 链表 (`A -> C`) 和 `high` 链表 (`B -> D`)。
2.  将 `low` 链表整个放入新数组的**原索引位置** `newTab[5]`。
3.  将 `high` 链表整个放入新数组的**“原索引 + 旧容量”**位置 `newTab[5 + 16]`，即 `newTab[21]`。

**结论：** 通过一次遍历和一次位运算，`HashMap` 就完成了旧桶到新桶的迁移，避免了对每个元素重新计算完整哈希的开销。

3. LinkedHashMap：维持插入或访问顺序的 Map

LinkedHashMap 是 HashMap 的一个子类。

内部原理

它继承了 HashMap 的所有特性（数组+链表+红黑树），并额外维护一个双向链表，用于记录键值对的顺序。

  • 插入顺序 (默认): 每次 put 一个新的键值对时，它会被添加到双向链表的末尾。遍历时，按插入顺序返回。
  • 访问顺序: LinkedHashMap 有一个特殊的构造函数 LinkedHashMap(int initialCapacity, float loadFactor, boolean accessOrder)。如果最后一个参数 accessOrder 为 true，那么每次调用 get 方法访问一个已存在的键值对时，该键值对会被移动到双向链表的末尾。这个特性非常适合用来实现 LRU (Least Recently Used) 缓存。

4. TreeMap：自动排序的 Map

TreeMap 能确保其中的键值对始终根据键 (Key) 处于排序状态。

内部原理

TreeMap 的底层是一个红黑树 (Red-Black Tree) 数据结构。所有的键值对都作为红黑树的节点存在。

性能特点

  • 有序: 键总是按照自然顺序或 Comparator 定义的顺序排列。
  • 性能稳定: put, get, remove 等操作的时间复杂度都是 O(log n)。

5. ConcurrentHashMap：线程安全的 Map

ConcurrentHashMap 是 java.util.concurrent 包下的一个类，是高并发场景下 Map 的首选。

内部原理 (JDK 8+)

它借鉴了 HashMap 的结构（数组+链表+红黑树），但通过使用 CAS (Compare-And-Swap) 操作和 synchronized 关键字实现了高效的线程安全。

  • 分段锁思想: 它不是对整个 Map 加锁，而是只对哈希桶数组中的某个位置（即链表或红黑树的头节点）加锁。当不同线程操作不同哈希桶时，它们之间不会产生竞争，从而大大提高了并发性能。
  • 只有当多个线程同时要修改完全相同的哈希桶时，才会有一个线程被 synchronized 阻塞，其他线程不受影响。

历史的尘埃：Hashtable

Hashtable 是一个源自 JDK 1.0 的、古老的键值对数据结构，可以看作是 HashMap 的一个线程安全的早期版本。虽然现在已不推荐使用，但了解它有助于理解 Java 集合框架的演进。

核心特性与 HashMap 的主要区别

  1. 线程安全 (Thread-Safety)

    • Hashtable 是线程安全的。 它通过在几乎所有公开方法（如 put, get）上添加 synchronized 关键字来实现，这相当于给整个数据结构上了一把大锁。虽然保证了数据一致性，但在高并发环境下性能极差。

  2. 对 null 值的支持

    • Hashtable 不允许 null 键和 null 值。 任何 null 存入都会抛出 NullPointerException。
    • HashMap 则允许一个 null 键和多个 null 值。

  3. 继承体系

    • Hashtable 继承自一个现已废弃的 Dictionary 类。
    • HashMap 继承自 AbstractMap，是更现代的设计。

为什么 Hashtable 被“淘汰”了？

Hashtable 提供的全局锁机制性能太差，无法满足现代高并发应用的需求。在需要线程安全的场景下，我们现在有了更好的选择：ConcurrentHashMap，它通过 CAS 和分段/桶锁等技术实现了高得多的并发性能。

总结对比

特性 Hashtable HashMap ConcurrentHashMap
线程安全 是 (全局锁) 否 是 (分段/桶锁)
性能 差 好 极好

附录：哈希冲突的常见解决方法

当不同键计算出相同地址时，如何存放和查找它们，就是“哈希冲突解决方法”要处理的问题。主要分为两大类：开放寻址法 (Open Addressing) 和 链地址法 (Separate Chaining)。

1. 开放寻址法 (Open Addressing)

这种方法不创建任何外部数据结构（如链表），而是坚持将所有元素都存放在哈希表数组内部。如果一个位置 index 被占用了，就按照某种规则去探测下一个可用的空位。

  • a. 线性探测 (Linear Probing): 如果 index 位置被占用，就尝试 index+1；如果还被占用，就尝试 index+2，以此类推。这种方法实现简单，但容易产生“一次聚集”现象，即冲突的元素在数组中连成一片，影响效率。
  • b. 二次方探测 (Quadratic Probing): 如果 index 位置被占用，就依次尝试 index+1², index-1², index+2²… 能有效缓解“一次聚集”，但会产生“二次聚集”。
  • c. 再哈希法 (Double Hashing): 使用两个哈希函数，hash1() 计算初始地址，hash2() 计算探测的步长。这是开放寻址法中效果最好的一种，能有效避免聚集。

2. 链地址法 (Separate Chaining)

这是 java.util.HashMap 所采用的方法。 它将哈希表的每个数组单元（哈希桶）作为一个“头指针”。所有哈希到这个位置的元素，都以链表（或 HashMap 中的红黑树）的形式串联在这个“头指针”后面。

总结对比

方法 原理 优点 缺点
线性探测 (hash(k) + i) % n 实现简单，缓存友好 一次聚集
二次方探测 (hash(k) + i²) % n 缓解一次聚集 二次聚集
再哈希法 (hash1(k) + i*hash2(k)) % n 聚集问题最少 计算成本高
链地址法 数组 + 链表/红黑树 直观，删除方便，负载高时表现好 额外空间开销，对缓存不友好

6. 高频面试题

Q1: HashMap 的 put 方法流程是怎样的？

A:

  1. 计算 key 的哈希值，定位到哈希桶数组的索引。
  2. 如果该位置为 null，直接创建一个新节点存入。
  3. 如果该位置不为 null，遍历该位置的链表或红黑树。
  4. 在遍历过程中，如果发现一个节点的 key 与当前要存入的 key 相同（hashCode 相同且 equals 为 true），则用新 value 覆盖旧 value，并返回旧 value。
  5. 如果遍历完所有节点都没有发现相同的 key，则将新节点添加到链表的末尾或红黑树中。
  6. 添加后，检查链表长度是否达到树化阈值（8），如果达到则将链表转化为红黑树。

Q2: HashMap 和 Hashtable 有什么区别？

A:

  • 线程安全: HashMap 是线程不安全的，Hashtable 是线程安全的。Hashtable 几乎所有方法都用了 synchronized 修饰，性能较差。
  • null 值: HashMap 允许一个 null 键和多个 null 值。Hashtable 不允许 null 键或 null 值。
  • 父类: HashMap 继承自 AbstractMap，Hashtable 继承自 Dictionary (一个已被废弃的类)。
  • 性能: 由于锁机制，HashMap 的性能远高于 Hashtable。在需要线程安全时，应使用 ConcurrentHashMap 而不是 Hashtable。

Q3: HashMap 的默认初始容量和负载因子是多少？为什么？

A:

  • 默认初始容量 (initial capacity): 16。
  • 默认负载因子 (load factor): 0.75。
  • 原因: 这是时间和空间成本之间的一个权衡。容量太小容易导致频繁的哈希冲突和扩容，影响性能。容量太大浪费内存。负载因子 0.75 是一个经验值，统计学上表明它能很好地平衡哈希冲突的概率和内存的使用率。
