什么是装饰器模式？

装饰器模式 (Decorator Pattern) 是一种结构型设计模式，它允许你通过将对象放入包含行为的特殊“包装”对象中，来为原对象动态地添加新的功能。这种模式在不改变原对象类定义的情况下，为其扩展功能，是继承关系的一个替代方案。

通俗地讲，装饰器模式就像给一个“素颜”的人化妆。这个人（原始对象）本身具有基本的功能（如吃饭、睡觉）。现在，我们可以给他“装饰”上不同的妆容（如晚宴妆、舞台妆），每种妆容都为他增添了新的外观和气质，但并没有改变他作为“人”的本质。你可以一层一层地化妆，比如先打底，再画眼影，再涂口红，每一层都是一个装饰器。

模式结构

装饰器模式主要包含以下角色：

  1. Component (组件): 定义了原始对象和装饰器对象的通用接口。
  2. ConcreteComponent (具体组件): 实现了组件接口的原始对象，即被装饰的对象。
  3. Decorator (装饰器): 抽象装饰器类，它实现了组件接口，并持有一个组件对象的引用。它的接口与组件接口完全一致，以便它可以透明地包装组件。
  4. ConcreteDecorator (具体装饰器): 实现了抽象装饰器类，负责向组件添加新的职责。

优缺点

优点

  1. 遵循开闭原则: 你可以在不修改现有代码的情况下，引入新的装饰器来扩展功能。
  2. 比继承更灵活: 你可以在运行时动态地添加或删除功能，并且可以任意组合多个装饰器，而继承是静态的，无法实现这一点。
  3. 避免类爆炸: 相较于通过继承来组合功能，装饰器模式可以避免创建大量的子类。

缺点

  1. 增加大量小类: 使用装饰器模式会产生许多细小的类，增加了系统的复杂性。
  2. 调试困难: 装饰器的嵌套层次可能很深，增加了调试和理解代码的难度。

装饰器模式 vs 适配器模式

  • 目的不同: 适配器模式的目的是解决接口不兼容的问题，它关注的是“转换”。而装饰器模式的目的是动态地为对象添加功能，它关注的是“增强”。
  • 实现方式不同: 适配器通常只包装一个对象，而装饰器可以被多层包装。

经典应用：Java I/O

Java 的 I/O 库是装饰器模式最经典的应用。例如，FileInputStream 是一个具体的组件，用于从文件读取字节。我们可以用 BufferedInputStream (一个具体装饰器) 来包装它，为其增加缓冲功能以提高性能。然后，我们还可以再用 DataInputStream (另一个具体装饰器) 来包装 BufferedInputStream，为其增加读取基本数据类型（如 int, double）的功能。
