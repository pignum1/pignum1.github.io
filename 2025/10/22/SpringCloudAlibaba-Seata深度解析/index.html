Seata 深度解析：分布式事务的一站式解决方案

分布式事务是微服务架构中最具挑战性的课题之一。Seata 旨在解决这种环境下数据“要么全成功，要么全失败”的一致性难题。

1. Seata 的核心架构：三大角色

Seata 的运行是由三个组件协同编排的一场“接力赛”：

  • TC (Transaction Coordinator) - 事务协调者: 独立的 Server 服务端，负责维护全局和分支事务的状态，驱动最终的提交或回滚。
  • TM (Transaction Manager) - 事务管理器: 嵌入在应用中，负责开启全局事务、提交或回滚全局事务（定义事务边界）。
  • RM (Resource Manager) - 资源管理器: 嵌入在应用中，管理分支事务的资源（如数据库），向 TC 汇报状态并执行补偿逻辑。

2. AT 模式核心原理：Seata 的“无侵入”魔法

AT 模式是基于 改良版两阶段提交 (2PC) 实现的。它通过数据源代理和 Undo Log，在保证高性能的同时实现了最终一致性。

2.1 一阶段 (Phase 1)：本地提交与拍照

  1. SQL 拦截: 代理数据源拦截业务 SQL。
  2. 写前镜像 (Before Image): 查询数据被修改前的状态。
  3. 执行业务: 执行真正的业务 SQL。
  4. 写后镜像 (After Image): 查询数据被修改后的状态。
  5. 全局锁申请: 向 TC 申请该行数据的 全局锁 (Global Lock)。
  6. 本地提交: 将业务修改与 Undo Log 在同一个本地事务内提交。提交后立即释放数据库本地锁。

2.2 二阶段 (Phase 2)：异步决断

  • 全局提交 (Commit): TC 发现全员成功，通知各 RM。RM 异步删除 Undo Log，性能极高。
  • 全局回滚 (Rollback): TC 发现有人失败，通知各 RM。RM 利用 Before Image 逆向生成 SQL 回滚数据。

3. 一致性控制的基石：隔离性与脏写检测

这是 Seata 能够在释放本地锁后依然保证一致性的关键。

3.1 全局锁 (Global Lock)

  • 作用: 防止不同分布式事务之间产生脏写。
  • 逻辑: 只有拿到了全局锁的分布式事务，才有权在一阶段进行提交。如果锁被占用，则必须等待。

3.2 脏写检测 (Dirty Write Check)

  • 风险: 一阶段释放本地锁后，可能有非 Seata 事务（不受控的 SQL）修改了数据。
  • 对策: 在回滚前，RM 会对比“数据库当前值”与“Undo Log 中的 After Image”。
    • 一致: 安全，执行回滚。
    • 不一致: 发生了脏写。Seata 为了数据安全会立即停止自动回滚，进入重试逻辑并发出告警，转由人工介入。

4. 事务模式深度对比

模式 一致性 性能 代码侵入 原理
AT 最终一致 高 无 代理数据源 + Undo Log 自动补偿
TCC 最终一致 极高 高 手动编写 Try-Confirm-Cancel 逻辑
Saga 最终一致 高 高 长事务补偿，适用于外部系统调用
XA 强一致 低 无 标准 2PC，二阶段才释放数据库锁

5. Seata 与 CAP 定理

  • AT 模式选择了 AP: 它通过一阶段快速提交（牺牲强一致性 C）换取了极高的可用性（A），并通过二阶段补偿实现了最终一致性。
  • XA 模式选择了 CP: 它在一阶段不提交，直到二阶段才释放锁，保证了强一致性（C），但严重影响了并发吞吐量（A）。

6. 常见面试题精选

Q1: Seata AT 模式下，读操作是否会读到“脏数据”？

  • 默认隔离级别是“读未提交”，确实可能读到。如果需要“读已提交”，必须使用 SELECT ... FOR UPDATE，这会触发代理数据源对全局锁的检查。

Q2: 为什么 AT 模式需要三级缓存？（误，这是 Spring 考点，Seata 考点是：如果回滚失败怎么办？）

  • 如果二阶段回滚失败（如网络中断），TC 会发起定时重试任务，直到回滚成功或达到最大重试次数转人工处理。这保证了事务的最终一致性。
