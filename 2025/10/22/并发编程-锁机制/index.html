并发编程 - 锁机制 (synchronized, Lock)

锁机制是 Java 并发编程中用于保证线程安全、实现数据同步的关键手段。理解 synchronized 关键字和 java.util.concurrent.locks.Lock 接口的原理和使用，对于编写正确、高效的并发程序至关重要。

1. 为什么需要锁？

在多线程环境下，当多个线程同时访问和修改共享资源时，如果没有适当的同步机制，就可能导致数据不一致、逻辑错误等并发问题。

  • 并发问题示例: i++ 操作
    i++ 看起来是一个简单的操作，但它实际上包含三个步骤：
    1. 读取 i 的值。
    2. 将 i 的值加 1。
    3. 将新值写回 i。
      在多线程环境下，如果线程 A 读取 i 的值为 0，正准备加 1，此时线程 B 也读取 i 的值为 0。然后线程 A 将 i 更新为 1，线程 B 也将 i 更新为 1。最终 i 的值是 1，而不是期望的 2。这就是线程不安全。

锁的作用就是保证在同一时刻，只有一个线程能够访问被保护的共享资源，从而避免并发问题，实现线程安全。

2. synchronized 关键字

synchronized 是 Java 内置的同步机制，它基于 JVM 的监视器锁 (Monitor)。

2.2 synchronized 的特性

  • 原子性: 保证被同步代码块或方法内的操作是原子的，不会被其他线程打断。
  • 可见性: 线程释放锁前，会把工作内存中的共享变量刷新到主内存；线程获取锁后，会清空工作内存，从主内存重新加载共享变量。
  • 有序性: 保证被同步代码块或方法内的操作有序执行，防止指令重排。
  • 可重入性: 同一个线程可以多次获取同一个锁，不会死锁。例如，一个线程已经持有了某个对象的锁，当它再次尝试获取该对象的锁时，可以直接进入。
  • 非公平锁: 线程获取锁的顺序不保证与请求顺序一致。

2.3 底层实现

  • 同步代码块: 由 monitorenter 和 monitorexit 字节码指令实现。monitorenter 指令在进入同步代码块时尝试获取锁，monitorexit 指令在退出同步代码块时释放锁。
  • 同步方法: 通过方法修饰符上的 ACC_SYNCHRONIZED 标志实现。JVM 会隐式地为该方法加上 monitorenter 和 monitorexit。

2.4 锁优化 (JDK 1.6+)

为了提高 synchronized 的性能，JVM 引入了多种锁优化技术，实现了锁的升级过程。锁的状态会根据竞争情况从低到高升级，且此过程不可逆。

  • 偏向锁 (Biased Locking):

    • 场景: 适用于只有一个线程反复进入同步块的场景。

    • 过程: 线程第一次获取锁时，会在对象头中记录该线程 ID。后续该线程再次进入同步块时，只需检查线程 ID 是否匹配，无需任何同步操作，效率极高。

  • 轻量级锁 (Lightweight Locking):

    • 场景: 当有第二个线程尝试获取偏向锁时，偏向锁会升级为轻量级锁。

    • 过程: 竞争的线程不会立即阻塞，而是通过 CAS (Compare-And-Swap) 操作尝试在自己的栈帧中创建锁记录来获取锁。

  • 自旋锁 (Spin Locking):

    • 场景: 当轻量级锁获取失败时，线程不会立即挂起（挂起和唤醒线程是昂贵的内核操作），而是会进行短暂的自旋（执行空循环），在原地等待锁被释放。

    • 优点: 适用于锁持有时间极短的场景，可以避免线程上下文切换的开销。

  • 重量级锁 (Heavyweight Locking):

    • 场景: 如果自旋一定次数后仍未获取到锁，或者有更多线程加入竞争，轻量级锁会膨胀为重量级锁。

    • 过程: 此时，锁会依赖于操作系统的互斥量 (Mutex) 来实现。未获取到锁的线程会被阻塞，并进入等待队列，放弃 CPU，等待被唤醒。这是开销最大的锁状态。

3. Lock 实现原理：AQS 深度解析

synchronized 是基于 JVM 的监视器锁 (Monitor) 实现的，而 ReentrantLock 以及 CountDownLatch、Semaphore 等并发工具，都是基于一个共同的框架——AbstractQueuedSynchronizer (AQS) 来构建的。

AQS 是一个用于构建锁和同步器的抽象框架。它通过一个 int 类型的 state 变量来表示同步状态，并使用一个内置的 FIFO 双向队列 (CLH 队列) 来管理所有等待获取同步状态的线程。

3.1 AQS 的核心设计

AQS 的设计精髓在于将“同步状态的管理”和“等待线程的管理”这两大核心任务清晰地分离开来。

  • state (同步状态):
    • 这是一个 volatile int 类型的变量，用于表示当前的同步状态。
    • state 的含义由具体的子类来定义。例如，在 ReentrantLock 中：
      • state = 0 表示锁未被任何线程持有。
      • state > 0 表示锁已被持有，其值代表了锁的重入次数。
  • CLH 等待队列:
    • 这是一个虚拟的、双向链表结构的队列，用于存放所有等待获取锁的线程。
    • 当一个线程尝试获取锁失败后，AQS 会将该线程封装成一个 Node 节点，并将其加入到队列的尾部。
    • 当持有锁的线程释放锁时，AQS 会唤醒队列头部的下一个节点所对应的线程。
  • 独占模式 (Exclusive) vs. 共享模式 (Shared):
    • 独占模式: 同一时刻只允许一个线程获取同步状态。ReentrantLock 就是独占模式。
    • 共享模式: 同一时刻允许多个线程获取同步状态。CountDownLatch 和 Semaphore 就是共享模式。

3.2 ReentrantLock 的 lock() 方法详解

我们以 ReentrantLock (非公平锁) 的 lock() 方法为例，一步步剖析其底层实现。

lock() 流程概览:

  1. 尝试获取锁: 尝试通过 CAS (Compare-And-Swap) 操作将 state 从 0 修改为 1。
    • 如果成功，将锁的持有者设置为当前线程，lock() 过程结束。
    • 如果 state 不为 0，检查当前锁的持有者是否为当前线程。如果是，则增加 state 的值（实现可重入），lock() 过程结束。
  2. 获取锁失败: 如果上述尝试失败（锁被其他线程持有），则进入 AQS 的标准流程：
    • a. 加入等待队列: 将当前线程封装成一个 Node 节点，并加入到 CLH 等待队列的尾部。
    • b. 自旋与阻塞: 线程进入一个“自旋”循环。在每次循环中，它会检查自己是否是队列中的第二个节点（即仅次于头节点）。
      • 如果是，它会再次尝试获取锁。
      • 如果不是，或者再次尝试获取锁失败，它就会通过 LockSupport.park(this) 方法将自己阻塞，放弃 CPU，等待被唤醒。
  3. 被唤醒: 当持有锁的线程调用 unlock() 释放锁时，它会唤醒队列中的下一个等待线程（即 unpark 操作）。被唤醒的线程会从 park 的地方继续执行，回到第 2.b 步的自旋循环中，再次尝试获取锁。

3.3 ReentrantLock 的 unlock() 方法详解

unlock() 的过程相对简单：

  1. 尝试释放锁: 调用 tryRelease(1) 方法。
    • 将 state 的值减 1。
    • 如果 state 减 1 后不为 0，说明是重入锁的释放，unlock() 过程结束。
    • 如果 state 减 1 后等于 0，说明锁已被完全释放。此时，将锁的持有者设置为 null。
  2. 唤醒后继者: 如果锁被完全释放，检查等待队列的头节点，并唤醒其后继节点所对应的线程 (LockSupport.unpark(thread))。

总结:
Lock 的实现核心在于 AQS。AQS 通过一个 state 变量和一套精密的线程排队、阻塞、唤醒机制，高效地实现了各种同步需求。理解了 AQS，就等于掌握了 Java 大半个 java.util.concurrent 包的精髓。

5. 死锁、活锁、饥饿

在并发编程中，除了线程安全问题，还需要警惕以下几种线程活性问题：

  • 死锁 (Deadlock): 两个或多个线程互相持有对方所需的资源，导致所有线程都无法继续执行。
    • 产生的四个必要条件 (必须同时满足):
      1. 互斥条件 (Mutual Exclusion): 资源在任意时刻只能被一个线程持有。这是锁的基本特性，通常无法破坏。
      2. 请求与保持条件 (Hold and Wait): 线程在已持有一个资源的情况下，又请求其他线程正持有的资源，但因请求被阻塞时，它不会释放自己已持有的资源。
      3. 不剥夺条件 (No Preemption): 线程已获得的资源，在未使用完毕之前，不能被其他线程强行剥夺，只能由持有者主动释放。
      4. 循环等待条件 (Circular Wait): 存在一个线程资源的循环等待链。例如，线程 A 等待线程 B 的资源，而线程 B 又在等待线程 A 的资源。
    • 如何避免: 破坏上述四个条件中的任意一个即可。最常见的策略是破坏“请求与保持”（例如一次性申请所有资源）和“循环等待”（例如保证所有线程按相同顺序获取锁）。
  • 活锁 (Livelock): 线程没有阻塞，但它们不断地改变状态，却无法取得任何进展。例如，两个线程互相谦让资源，导致谁也无法先执行。
  • 饥饿 (Starvation): 某个线程一直无法获取所需的资源，导致无法执行。例如，高优先级线程总是抢占低优先级线程的资源，或者非公平锁导致某些线程一直无法获取锁。

5. 并发编程的基石：CAS (Compare-And-Swap)

CAS，即“比较并交换”，是现代并发编程的基石，也是 ReentrantLock 等高级并发工具能够实现高性能的关键所在。

5.1 CAS 是什么？

CAS 是一条源自硬件（CPU）的原子指令。它的功能是：“我认为内存地址 V 的值应该是 A，如果是，那就把它更新成 B；如果不是，那就不做任何操作。”

这个操作包含三个核心操作数：

  1. V: 要更新的内存地址 (Memory Location)。
  2. A: 预期的旧值 (Expected Old Value)。
  3. B: 要更新的新值 (New Value)。

整个“比较”和“交换”的过程是一步完成的，由 CPU 保证其原子性，不会被任何其他线程打断。

5.2 为什么使用 CAS？（与锁的对比）

传统的锁机制（如 synchronized）是一种悲观锁。它假设并发冲突总会发生，所以线程在访问共享资源前，必须先获取锁。如果获取不到，线程就会被阻塞，进入挂起状态，这涉及到昂贵的线程上下文切换。

而 CAS 是乐观锁思想的典型实现。它假设并发冲突是小概率事件，所以它会：

  1. 乐观地执行: 线程直接读取内存中的值，进行计算，然后尝试用 CAS 更新回去。
  2. 冲突检测: 在更新时，通过 CAS 检查内存中的值是否还是当初读取的那个“预期旧值”。
    • 如果值没变: 说明没有其他线程修改过它，更新成功。整个过程没有加锁，也没有阻塞。
    • 如果值变了: 说明在计算期间有其他线程修改了数据，本次 CAS 更新失败。但线程不会被阻塞。
  3. 失败后重试: 更新失败的线程通常会进入一个自旋循环 (Spin Loop)，即重新读取新值、重新计算、再次尝试 CAS，直到成功为止。

核心区别: synchronized 等悲观锁是通过阻塞线程来避免冲突，而 CAS 是通过不阻塞、循环重试来解决冲突。

5.3 CAS 的优点

  1. 高性能/低开销: 在并发冲突不激烈的情况下，CAS 避免了传统锁机制中线程阻塞和唤醒所带来的巨大性能开销（上下文切换）。这是其最大的优势。
  2. 无阻塞: 线程在 CAS 操作中始终处于可运行状态，不会被挂起。这提高了程序的响应能力。
  3. 避免死锁: CAS 操作不涉及获取锁，因此天然地避免了因锁顺序不当等问题导致的死锁。

5.4 CAS 的经典问题：ABA 问题

CAS 并非完美，它存在一个著名的“ABA”问题。

  • 问题描述:
    1. 线程 T1 读取内存值 V，得到 A。
    2. 此时，线程 T2介入，将值从 A 修改为 B，然后又修改回 A。
    3. 线程 T1 回来执行 CAS，发现内存值 V 仍然是 A，于是执行成功。
  • 问题本质: 线程 T1 认为数据没有被修改过，但实际上它已经被改动过了。在大多数情况下，这可能没问题。但在某些场景下（例如，链表操作），这可能会导致严重的数据不一致。
  • 解决方案: 版本号机制。在每次更新变量时，除了更新值，也更新一个版本号。CAS 操作需要同时检查值和版本号。Java 中的 AtomicStampedReference 类就是用来解决 ABA 问题的，它通过一个“时间戳”（stamp）来充当版本号。

总结:
CAS 是实现高性能、无锁并发数据结构和算法的基石。Java 的 java.util.concurrent.atomic 包（如 AtomicInteger, AtomicReference）就是完全基于 CAS 实现的，而 AQS 框架在尝试获取锁时也大量使用了 CAS 来避免线程立即阻塞。

6. 锁机制实战案例

理论结合实践是最好的学习方式。下面是几个简单而经典的实战案例，来帮助您快速理解不同锁的使用场景。

案例一：synchronized 保证线程安全的计数器

  • 问题: 多个线程同时对一个共享变量 count 进行 i++ 操作，导致最终结果小于预期。
  • 解决方案: 使用 synchronized 关键字修饰 increment() 方法，保证每次只有一个线程能执行该方法。
  • 分析: 如果没有 synchronized，count++ 这个非原子性操作在多线程下会产生竞态条件。synchronized 保证了该操作的原子性，确保了最终结果的正确性。

案例二：ReentrantLock 实现可中断和可超时的锁等待

  • 问题: 线程 A 持有锁，线程 B 在无限期等待。如果希望线程 B 在等待一段时间后能放弃等待，去做别的事情，synchronized 无法做到。
  • 解决方案: 使用 ReentrantLock 的 tryLock() 方法。
  • 分析: tryLock() 赋予了线程在获取锁时更多的灵活性，可以避免因无限期等待而导致的线程假死或性能问题。

案例三：ReadWriteLock 提升读多写少场景的性能

  • 问题: 一个共享数据，有大量的线程读取它，但只有很少的线程修改它。如果使用 synchronized，读操作之间也会互斥，导致性能低下。
  • 解决方案: 使用 ReentrantReadWriteLock，允许多个读线程同时访问，但写线程独占。
  • 分析: 多个读线程可以同时进入 get 方法，而写线程在执行时，所有读线程和其他写线程都必须等待。这在读多写少的场景下，极大地提高了并发度。
