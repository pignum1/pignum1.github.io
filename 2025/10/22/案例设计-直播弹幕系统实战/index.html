直播弹幕系统架构：应对百万并发长连接与消息分发

直播弹幕系统的核心挑战在于 “极高实时性”、“海量长连接” 以及 “瞬间消息扩散风暴”。

1. 物理连接层：单机百万长连接的秘诀

传统的 HTTP 每次请求都要握手，开销大。直播弹幕必须使用 WebSocket (基于 TCP 的双全工协议)。

1.1 Netty 与 Epoll 优化

  • Netty 核心: 使用 NioEventLoopGroup (Reactor 模式) 异步非阻塞处理 IO。
  • 系统级调优:
    1. 文件描述符 (FD): ulimit -n 1048576。Linux 默认 1024，必须开大。
    2. TCP 内存: 调整 tcp_rmem 和 tcp_wmem，减少每个连接的内存占用。
    3. Epoll: 在 Linux 下强制开启 EpollEventLoopGroup。

1.2 物理连接集群路由

假如 100w 用户连接在 10 台网关服务上：

  • Session 存储: 利用 Redis (Hash 结构) 存储 UserId -> GatewayIP 的映射。
  • 消息路由: 当 A 想发私信给 B，Server 查 Redis 发现 B 在网关 2，于是将消息转发给网关 2 进行推送。

2. 消息推送层：对抗扩散风暴 (Fan-out)

场景: 一个拥有 100w 在线观众的大主播，每秒产生 1000 条弹幕。

  • 数学计算: $1000 \times 1,000,000 = 1,000,000,000$ (10亿) 次推送/秒。单机网关必挂。

2.1 解决方案：多级“挡板”与聚合

  1. 聚合推送 (Batching):
    • 网关层不要每来一条就推一条。
    • 策略: 每 200ms 聚合该主播的所有弹幕，打包成一个 JSON 数组，一次性推给在线客户端。推送次数瞬间降为 5 次/秒。
  2. 分级推送 (Prioritization):
    • 等级划分: 只有大额打赏、特权用户的弹幕保证 100% 必达。
    • 丢弃采样: 面对疯狂刷屏，对普通弹幕进行随机采样丢弃，保证前端不卡顿。
  3. 消息扇出策略:
    • 利用 MQ (Kafka/RocketMQ) 的广播模式或 Redis 的 Pub/Sub。
    • 网关节点订阅“房间 ID”的主题。一旦有新弹幕，所有网关节点都能收到。

3. 存储与历史消息

  • 最近弹幕: 使用 Redis 的 ZSet。score 为时间戳，value 为消息。
    • 用户进场：ZRANGE room_101 -50 -1 取最后 50 条。
  • 全量归档: 异步将弹幕写入 HBase 或 Elasticsearch，供后续数据回放或回看。

4. 极端挑战：重连风暴 (Reconnection Storm)

场景: 某个网关节点挂了，上面的 10w 个连接同时重连。

  • 后果: 握手 CPU 爆表，瞬间压死系统。
  • 对策:
    1. 随机延迟 (Jitter): 客户端检测到断开后，随机等待 1-10 秒再重连。
    2. HTTP 握手限流: 网关层对 Upgrade: websocket 的 HTTP 请求进行严格限流。

5. 架构师 P7 级面试细节

Q: 为什么不直接用 Redis Pub/Sub 来做消息转发？

  • 缺点: Redis Pub/Sub 是“发后即忘”的。如果网关节点网络抖动一秒，这一秒的消息就全丢了，没有缓冲区，也不支持持久化。生产环境首选 MQ。

Q: 如何检测客户端是否已经掉线？

  • 双向心跳 (Keep-Alive):
    • 客户端每 30s 发一个 Ping。
    • 服务端回一个 Pong。
    • 服务端维护一个 IdleStateHandler。如果 60s 没收到 Ping，主动断开连接，释放 FD。
