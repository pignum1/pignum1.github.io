并发编程 - 核心面试知识点汇总

本篇笔记旨在高度浓缩“并发编程与多线程”模块的核心知识点，以便于快速复习，加深理解，并从容应对面试。

Part 1: 线程基础与 JMM (Java 内存模型)

1. 线程的 6 种状态及其转换

  • NEW (新建): 线程已创建但未启动。
  • RUNNABLE (可运行): 调用 start() 后进入此状态。它包含了 Ready (就绪, 等待CPU调度) 和 Running (运行中) 两个子状态。
  • BLOCKED (阻塞): 等待进入 synchronized 同步块/方法时。
  • WAITING (无限期等待): 调用 Object.wait(), Thread.join(), LockSupport.park()。
  • TIMED_WAITING (有时限等待): 调用 Thread.sleep(long), Object.wait(long) 等。
  • TERMINATED (终止): run() 方法执行完毕或因异常退出。

2. volatile 关键字

  • 两大作用:
    1. 保证可见性: 对 volatile 变量的写操作会立即刷新到主内存，读操作会强制从主内存加载最新值。
    2. 禁止指令重排: 通过插入内存屏障，防止编译器和处理器为了优化而改变指令的执行顺序。
  • 与 synchronized 的区别:
    • volatile 是轻量级的，只能修饰变量；synchronized 是重量级的，可以修饰方法和代码块。
    • volatile 不保证原子性；synchronized 保证原子性。
    • volatile 不会造成线程阻塞；synchronized 会造成线程阻塞。

Part 2: synchronized 与锁机制

1. synchronized 的三种用法与锁对象

  • 修饰普通方法: 锁对象是当前实例 (this)。
  • 修饰静态方法: 锁对象是当前类的 Class 对象。
  • 修饰代码块: 锁对象是 () 中指定的对象。

2. synchronized 底层原理与锁优化 (锁升级)

  • 底层: 依赖 JVM 的 monitorenter 和 monitorexit 字节码指令，以及对象头中的 Mark Word 来记录锁信息。
  • 锁升级过程 (不可逆):
    1. 无锁状态: 初始状态。
    2. 偏向锁: 当只有一个线程访问时，在 Mark Word 中记录该线程 ID，后续该线程进出同步块无需 CAS，效率最高。
    3. 轻量级锁: 当出现第二个线程竞争时，锁升级。竞争线程通过 CAS 自旋尝试获取锁，避免线程阻塞。
    4. 重量级锁: 如果自旋失败或竞争加剧，锁膨胀为重量级锁，依赖操作系统的互斥量 (Mutex)。失败的线程会被阻塞，进入等待队列。

3. 死锁

  • 产生的四个必要条件:
    1. 互斥条件: 资源独占。
    2. 请求与保持条件: 持有资源的同时请求新资源。
    3. 不剥夺条件: 资源不能被强行抢占。
    4. 循环等待条件: 形成线程与资源的循环等待链。
  • 如何避免: 破坏后三个条件中的任意一个，最常见的是按序加锁来破坏循环等待条件。

Part 3: Lock 接口与 AQS

1. Lock vs. synchronized

特性 synchronized Lock (以 ReentrantLock 为例)
实现方式 关键字 (JVM层面) 接口 (API层面)
释放锁 自动释放 必须手动释放 (在finally中)
功能 基本功能 更强大：可中断、可超时、公平/非公平可选、可绑定多条件
性能 JDK 1.6+ 优化后性能优异 性能优异，更灵活

2. AQS (AbstractQueuedSynchronizer)

  • 是什么: 构建锁和同步器的核心框架。ReentrantLock, Semaphore 等都基于它。
  • 核心设计:
    • state (同步状态): 一个 volatile int 变量，其含义由子类定义（如在 ReentrantLock 中表示重入次数）。
    • CLH 队列: 一个 FIFO 双向队列，用于管理所有等待获取锁的线程。获取锁失败的线程被封装成 Node 加入队尾并阻塞。

3. CAS (Compare-And-Swap)

  • 是什么: CPU 的原子指令，用于实现无锁并发。
  • 原理: 乐观锁思想。假设没有冲突，直接尝试更新。如果内存值与预期值一样，则更新；否则失败并重试（自旋）。
  • ABA 问题: 值从 A->B->A，CAS 无法察觉。解决方案是版本号机制，如 AtomicStampedReference。

Part 4: JUC 核心工具

1. ThreadPoolExecutor (线程池)

  • 7大核心参数: corePoolSize, maximumPoolSize, keepAliveTime, unit, workQueue, threadFactory, handler。
  • 工作原理: 任务提交后，按“核心线程 -> 任务队列 -> 最大线程 -> 拒绝策略”的流程处理。
  • Executors 风险: newFixedThreadPool 和 newCachedThreadPool 等可能因无界队列或无界最大线程数导致 OOM。生产环境必须手动创建。
  • 合理配置: CPU 密集型任务配置 CPU核心数+1 个线程；I/O 密集型任务配置更多线程 (核心数 / (1-阻塞系数))。

2. ConcurrentHashMap (JDK 1.8)

  • 核心优势: 兼顾线程安全与高性能。
  • 锁机制: CAS + synchronized 锁节点。
  • get() 为何不加锁: Node 数组及关键属性由 volatile 修饰，保证了内存可见性。
  • 为何用 synchronized 代替 ReentrantLock: 在锁粒度极小的情况下，JVM 优化后的 synchronized 内存开销更小、实现更简洁。
