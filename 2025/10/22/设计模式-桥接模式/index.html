什么是桥接模式？

桥接模式 (Bridge Pattern) 是一种结构型设计模式，它旨在将抽象部分（Abstraction）与它的实现部分（Implementation）分离，使它们都可以独立地变化。通过提供抽象和实现之间的桥梁结构，来实现这种解耦。

这个定义比较抽象，让我们用一个例子来理解。假设你要画不同颜色（如红色、蓝色）的形状（如圆形、方形）。如果使用继承，你可能需要创建 RedCircle, BlueCircle, RedSquare, BlueSquare 等大量的类。这里有两个变化的维度：形状和颜色。桥接模式就是用来处理这种多维度变化的场景的。它会将“形状”这个抽象概念和“颜色”这个实现概念分离开，让它们可以自由组合，而不是通过继承绑定在一起。

模式结构

桥接模式包含以下核心角色：

  1. Abstraction (抽象部分): 定义了抽象类的接口，并持有一个 Implementor (实现部分) 的引用。它通常是抽象类，而不是接口。
  2. RefinedAbstraction (精确抽象): 继承或实现了 Abstraction，并可以添加自己独特的业务逻辑。
  3. Implementor (实现部分接口): 定义了实现类的接口，这个接口不一定要与 Abstraction 的接口完全一致。它只提供基本的操作。
  4. ConcreteImplementor (具体实现): 实现了 Implementor 接口，是实现部分接口的具体实现。

优缺点

优点

  1. 分离抽象和实现: 这是桥接模式最大的优点。它使得抽象部分和实现部分可以沿着各自的维度独立地变化，而不会相互影响。
  2. 极大地提高了可扩展性: 当系统需要在一个或多个维度上进行扩展时，只需增加相应的类即可，无需修改现有代码，符合开闭原则。
  3. 避免类爆炸: 对于多维度变化的场景，它避免了使用继承导致的大量子类。

缺点

  1. 增加了系统的复杂性: 桥接模式要求正确地识别出系统中两个独立变化的维度，这增加了系统的设计难度。
  2. 理解难度较高: 对于新手来说，理解“抽象”和“实现”的分离可能需要一些时间。

桥接模式 vs. 适配器模式

  • 目的不同: 桥接模式的目的是分离抽象和实现，以便它们可以独立变化，通常在系统设计初期就会使用。而适配器模式的目的是兼容两个不兼容的接口，通常在系统后期，为了兼容遗留代码或第三方库时使用。
  • 结构不同: 桥接模式是双向的，抽象和实现都可以变化。适配器模式是单向的，它只是为了让被适配者适用于目标接口。

经典应用

  1. JDBC 驱动程序: 这是桥接模式最经典的应用。Java 的 java.sql 包定义了一套标准的 JDBC 接口（如 Connection, Statement），这就是抽象部分。而各大数据库厂商（MySQL, Oracle, PostgreSQL 等）则提供了各自的驱动实现，这就是具体实现部分。应用程序代码只需要面向标准的 JDBC 接口编程，而 JDBC 的 DriverManager 通过“桥梁”来加载和使用具体的数据库驱动，从而实现了 Java 程序与具体数据库实现的解耦。
  2. 不同平台的 GUI 框架: 在一个图形用户界面（GUI）框架中，“窗口”可以是一个抽象概念。这个窗口可以在不同的操作系统（Windows, macOS, Linux）上被实现。桥接模式可以将“窗口”的通用功能（如 draw, resize）与其在特定操作系统上的具体实现分离开来。
