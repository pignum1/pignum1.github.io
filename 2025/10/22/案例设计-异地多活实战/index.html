异地多活架构设计：挑战物理极限

当业务规模达到支付宝、微信级别，或者金融系统要求 99.999% 可用性时，单机房容灾（Active-Standby）已无法满足需求。我们需要 异地多活 (Multi-Geo Multi-Active)，即多个机房同时对外提供服务。

1. 核心挑战：物理距离与 CAP 定理

  • 同城双活: 距离 < 100km，光纤延迟 < 2ms。近似看作内网，数据库可同步。
  • 异地多活: 距离 > 1000km（如北京-上海），延迟 ~30ms。
    • 问题: 数据库不能做同步写（Sync Replication），否则一个事务要 60ms+，性能无法接受。必须采用 异步复制。
    • 代价: 数据存在延迟。如果切流，可能会丢数据或读到旧数据。

2. 终极架构：单元化 (Unitization / Set / Cell)

为了解决跨机房延迟问题，必须遵循 “就近访问，流量闭环” 原则。

2.1 流量路由与切片

  • 分片键 (Sharding Key): 通常选择 User ID。
  • 路由规则:
    • UserId % 100，0-49 分给 北京单元 (Set A)，50-99 分给 上海单元 (Set B)。
    • 网关层 (Nginx/Gateway) 解析 Cookie/Header 中的 UserId，将请求分发到对应机房的入口。

2.2 单元封闭 (Local Close Loop)

  • 原则: 属于 Set A 的用户，其所有业务逻辑（Web -> Service -> Redis -> DB）必须全部在 Set A 机房内完成。
  • 禁止: 严禁 Service A 调用 Service B（跨机房 RPC），严禁 DB A 连接 DB B。
  • 例外: 全局共享数据（如“配置中心”、“汇率”、“商品基本信息”）通常采用“一写多读”模式，由中心机房异步同步到各单元。

3. 数据层同步与冲突解决

3.1 数据库双向同步 (DTS / Otter)

  • 北京 DB 和 上海 DB 之间建立双向同步通道（基于 Binlog）。
  • 冲突: 理论上，因为按 User ID 严格切分了流量，同一个 User 不会同时在两地操作。
  • 防御: 万一流量路由错了？
    • Last Write Wins (LWW): 以时间戳最新的为准（需要 NTP 时钟同步）。
    • 冲突报错: 检测到数据版本不一致，直接报警人工介入。

3.2 全局唯一 ID (Snowflake 进阶)

  • 标准 Snowflake: 41bit时间戳 + 10bit机器ID + 12bit序列号。
  • 多活改造:
    • 将 10bit机器ID 拆分为 3bit 机房ID + 7bit 机器ID。
    • 机房 1: ID 以 001 开头。
    • 机房 2: ID 以 002 开头。
    • 效果: 无论怎么切流、合并数据，全局主键永不冲突。

4. 灾难恢复与切流 (Failover)

场景: 北京机房整体断电（地震/火灾）。

  1. 切流指令: 管控中心下发指令，修改全局 DNS 或边缘网关路由规则。
    • 原规则: 0-49 -> 北京, 50-99 -> 上海。
    • 新规则: 0-99 -> 上海。
  2. 数据追平:
    • 由于是异步复制，北京断电瞬间，可能有几秒数据还没传到上海。
    • 策略: 此时上海机房承接这部分用户，允许他们读取旧数据，或者写入新数据（产生分支）。等北京恢复后，通过冲突合并工具修复数据。
  3. 禁写保护: 在切流初期，为防止脏数据，可对受影响用户开启“只读模式”几分钟，待数据同步追平后再开放写入。

5. 无法单元化的业务怎么办？

有些业务无法按 User ID 拆分（如：库存扣减、秒杀）。

  • 方案: 依然采用 同城双活 或 中心化部署。
  • 牺牲: 非中心机房的用户访问这些特定业务时，必须跨机房 RPC，忍受 30ms 延迟（业务可接受范围内）。

6. 架构师 P7 级思考题

Q: 异地多活中，Redis 缓存需要同步吗？

  • A: 通常 不需要。缓存是 DB 的副本。切流到新机房后，虽然缓存是空的（Cold Cache），但可以通过预热或“缓存击穿保护”机制慢慢重建。同步 Redis 成本极高且易循环复制。

Q: 为什么说“异地多活”是架构的“核武器”？

  • A: 成本极高（机器翻倍、开发运维难度指数级上升）。只有当业务体量产生的价值（或宕机损失）远超成本时，才考虑实施。不要为了技术而技术。
