JavaSE - 反射 (Reflection)

反射（Reflection）是 Java 语言在运行时动态地获取类信息并操作对象的能力。它允许程序在运行时检查类、接口、字段和方法，而无需在编译时知道这些信息。这是 Java 语言非常强大和灵活的特性，也是许多高级框架（如 Spring、Hibernate、JUnit）实现其核心功能的基石。

1. 反射的基石：Class 类

java.lang.Class 类是反射的入口。每个类在 JVM 中都有一个对应的 Class 对象，这个对象包含了该类的所有结构信息（构造器、方法、字段等）。

获取 Class 对象的三种方式

  1. Class.forName("全限定类名"): 最常用，通过类的全限定名获取。
  2. 类名.class: 通过类本身的 class 属性获取。
  3. 对象.getClass(): 通过对象的 getClass() 方法获取。

2. 反射的主要用途

通过 Class 对象，我们可以获取并操作类的各种成员。

2.1 获取构造器并创建对象 (Constructor)

  • getConstructors(): 获取所有 public 构造器。
  • getDeclaredConstructors(): 获取所有声明的构造器（包括 private, protected, default）。
  • getConstructor(参数类型...): 获取指定 public 构造器。
  • getDeclaredConstructor(参数类型...): 获取指定声明的构造器。

2.2 获取字段并操作属性 (Field)

  • getFields(): 获取所有 public 字段。
  • getDeclaredFields(): 获取所有声明的字段（包括 private）。
  • getField(字段名): 获取指定 public 字段。
  • getDeclaredField(字段名): 获取指定声明的字段。

2.3 获取方法并调用 (Method)

  • getMethods(): 获取所有 public 方法（包括继承的）。
  • getDeclaredMethods(): 获取所有声明的方法（包括 private，不包括继承的）。
  • getMethod(方法名, 参数类型...): 获取指定 public 方法。
  • getDeclaredMethod(方法名, 参数类型...): 获取指定声明的方法。

3. 反射的优缺点

优点

  • 动态性: 运行时获取类信息，动态创建对象、调用方法、操作字段，极大地增强了程序的灵活性和扩展性。
  • 解耦: 可以在编译时不知道具体实现类的情况下，通过反射调用其方法，实现松耦合。
  • 框架基础: 许多流行框架（如 Spring 的 IoC/AOP、ORM 框架的数据映射、JUnit 的测试注解）都依赖反射实现。

缺点

  • 性能开销: 反射操作通常比直接调用代码慢很多，因为它涉及动态解析、安全检查等。
  • 安全性问题: setAccessible(true) 可以绕过 Java 的访问控制，可能破坏封装性。
  • 代码复杂性: 反射代码通常比直接代码更复杂、更难以阅读和维护。
  • 类型安全降低: 反射操作在编译时无法进行类型检查，错误可能在运行时才暴露。

4. 反射的应用场景

  • 框架开发: Spring IoC 容器通过反射实例化 Bean，AOP 通过反射实现动态代理。
  • ORM 框架: Hibernate、MyBatis 等通过反射将数据库记录映射到 Java 对象。
  • 单元测试: JUnit 通过反射查找并执行测试方法。
  • 动态代理: JDK 动态代理就是基于反射实现的。
  • 注解处理器: 通过反射读取注解信息，进行相应的处理。

5. 动态代理：JDK vs. CGLIB

反射最重要和最广泛的应用之一就是动态代理。动态代理是一种在运行时动态创建代理对象的技术，它允许我们在不修改源码的情况下，增强或拦截目标对象的方法。Spring AOP 的实现就完全依赖于动态代理。

1. JDK 动态代理

  • 实现原理:

    • JDK 动态代理是 Java 原生支持的，它利用反射机制，在运行时动态地创建一个实现了一个或多个接口的代理类。
    • 它要求目标类必须实现接口。代理类会继承 java.lang.reflect.Proxy 类，并实现目标类所实现的所有接口。
    • 所有对代理对象方法的调用，都会被转发到一个统一的 InvocationHandler 接口的 invoke 方法中，我们可以在这个 invoke 方法里编写增强逻辑（如日志、事务、权限校验）。

  • 核心组件:

    • java.lang.reflect.Proxy: 用于创建代理对象的工厂类。
    • java.lang.reflect.InvocationHandler: 一个接口，所有的方法调用都会被分派到它的 invoke 方法。

  • 优点:

    • Java 原生支持，无需引入任何第三方库。
    • 基于接口，强制实现了面向接口编程，耦合度更低。

  • 缺点:

    • 硬性限制: 目标类必须实现接口，否则无法创建代理。对于没有实现接口的类，JDK 动态代理无能为力。

2. CGLIB 动态代理

  • 实现原理:

    • CGLIB (Code Generation Library) 是一个强大的、高性能的代码生成库。它不依赖于接口，而是通过继承目标类来创建代理。
    • 它在运行时动态地生成一个目标类的子类作为代理对象，并重写目标类中所有非 final 的方法。
    • 所有对代理对象方法的调用，都会被一个 MethodInterceptor 接口的 intercept 方法拦截，我们可以在这个方法里编写增强逻辑。

  • 核心组件:

    • net.sf.cglib.proxy.Enhancer: 用于创建代理对象的工厂类。
    • net.sf.cglib.proxy.MethodInterceptor: 一个接口，所有的方法调用都会被分派到它的 intercept 方法。

  • 优点:

    • 没有接口限制: 即使目标类没有实现任何接口，CGLIB 也能为其创建代理。这是它相比 JDK 动态代理最大的优势。

  • 缺点:

    • 无法代理 final 方法: 因为 CGLIB 是通过继承和重写来实现的，所以它无法代理被 final 关键字修饰的方法。
    • 需要引入第三方库。
    • 在某些情况下，由于是生成子类，其创建代理对象的速度可能比 JDK 动态代理稍慢。

总结对比

特性 JDK 动态代理 CGLIB 动态代理
实现基础 接口 继承
核心原理 反射 ASM (字节码操作)
目标类要求 必须实现接口 无需实现接口
代理限制 无 不能代理 final 方法/类
依赖 Java 原生 第三方库 (cglib.jar)
性能 在方法调用上，早期版本 CGLIB 性能更高，但现代 JDK 版本中两者性能差距已很小。 在对象创建上，可能略慢于 JDK 代理。

Spring AOP 中的选择

Spring AOP 在选择使用哪种代理时，遵循以下规则：

  • 如果目标对象实现了接口，Spring AOP 默认使用 JDK 动态代理。
  • 如果目标对象没有实现接口，Spring AOP 会自动切换到使用 CGLIB 动态代理。
  • 你也可以通过配置（proxy-target-class="true"）来强制 Spring AOP 始终使用 CGLIB 动态代理。
