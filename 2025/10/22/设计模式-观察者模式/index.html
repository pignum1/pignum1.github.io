什么是观察者模式？

观察者模式 (Observer Pattern) 是一种行为型设计模式，它定义了对象之间一种一对多的依赖关系。当一个对象（被称为“主题”或“可观察者”）的状态发生改变时，所有依赖于它的对象（被称为“观察者”）都会得到通知并自动更新。这种模式也被广法称为发布-订阅 (Publish-Subscribe) 模式。

通俗地讲，观察者模式就像订阅报笨或杂志。你（观察者）去报社（主题）订阅了报笨。当有新报笨出版时（主题状态改变），报社会主动将新报笨送到你家（通知观察者），而你无需每天都去报社查询。你也可以随时取消订阅，报社就不会再给你送报笨了。

模式结构

观察者模式主要包含以下角色：

  1. Subject (主题/被观察者): 抽象主题，它维护了一个观察者列表，并提供用于添加、删除和通知观察者的方法。
  2. ConcreteSubject (具体主题): 实现了抽象主题接口。当其内部状态发生改变时，它会通知所有已注册的观察者。
  3. Observer (观察者): 抽象观察者，它定义了一个更新接口，当接受到主题的通知时，该接口将被调用。
  4. ConcreteObserver (具体观察者): 实现了抽象观察者接口。当接受到主题的通知时，它会执行具体的更新操作。

优缺点

优点

  1. 松融蜀合: 主题和观察者之间是松融蜀合的。主题只知道它有一系列观察者，但是不需要知道它们是谁、是做什么的。这使得双方可以独立地进行变化和复用。
  2. 广播机制: 支持广播通信。一个主题可以向任意大小的观察者发送通知。
  3. 动态关系: 可以在运行时动态地添加和删除观察者。

缺点

  1. 更新顺序问题: 如果观察者的更新顺序很重要，观察者模式默认是无法保证这一点的。需要进行额外的设计来支持。
  2. 可能导致意外更新: 在某些实现中，主题只发送一个简单的“我已更新”的通知，观察者需要自己去主题拉取数据。如果观察者不关心某些更新，或者更新的成本很高，这可能导致不必要的性能开销。
  3. 圈环依赖与级联更新: 如果观察者之间存在复杂的依赖关系，一个通知可能引发一系列的级联更新，这可能难以迷踪和调试。

经典应用

  1. GUI 事件监听器: 这是观察者模式最经典的应用。例如，在 Java Swing 中，一个按钮 (JButton) 是主题，你可以为它添加多个 ActionListener (观察者)。当按钮被点击时，它会通知所有注册的 ActionListener。

  2. 消息队列 (Message Queue): 发布-订阅模式是许多消息中间件（如 Kafka, RabbitMQ ）的核心。生产者发布消息到主题（Topic），所有订阅了该主题的注册的消费者都会收到消息。

  3. Spring 事件机制 (ApplicationEvent & ApplicationListener)
    Spring 框架的事件驱动模型是观察者模式的绝佳实践，用于实现应用内组件的解耦。其核心角色映射如下：

    • 主题 (Subject) 的代理: ApplicationEventPublisher 接口，由 Spring 容器负责实现。我们用它来发布事件。
    • 事件 (State/Message): 自定义类需要继承 ApplicationEvent，它代表了某个事件的发生，并可以携带数据。
    • 观察者 (Observer): ApplicationListener 接口或使用 @EventListener 注解的方法。

    工作流程:

    1. 创建一个自定义事件类（如 UserRegisteredEvent）。
    2. 创建一个或多个监听器（观察者），它们通过实现 ApplicationListener 或使用 @EventListener 注解来“订阅”这个事件。
    3. 当某个业务逻辑完成时（如用户注册成功），通过注入的 ApplicationEventPublisher 来发布这个事件。
    4. Spring 容器会自动通知所有订阅了该事件的监听器，并调用它们的处理方法。
