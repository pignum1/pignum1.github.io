HTTP/HTTPS 协议

HTTP (HyperText Transfer Protocol, 超文本传输协议) 是一个用于分布式、协作式和超媒体信息系统的应用层协议。它是万维网（World Wide Web）数据通信的基础。

1. HTTP 的特点

  • 基于 TCP/IP: HTTP 协议构建在 TCP 协议之上，利用 TCP 提供的可靠数据传输服务。
  • 请求-响应模型 (Request-Response): 通信由客户端发起，服务器进行响应。客户端发送一个 HTTP 请求，服务器返回一个 HTTP 响应。
  • 无状态 (Stateless): 服务器不保存任何关于客户端过去请求的信息。每个请求都被视为独立的事务。这简化了服务器的设计，但为了在不同请求间维持状态（如用户登录），需要引入 Cookie 和 Session 机制。

2. HTTP 请求报文 (Request)

一个 HTTP 请求由三部分组成：请求行、请求头、请求体。

  • 请求行 (Request Line): 方法 (Method) + URL + 协议版本
  • 请求头 (Headers): 包含关于客户端、请求本身等的附加信息，以 Key: Value 形式存在。
  • 请求体 (Body): 包含需要发送给服务器的数据，例如 POST 请求中的表单数据。

常用 HTTP 方法 (Methods)

  • GET: 请求获取指定资源。通常只用于读取数据，没有请求体。
  • POST: 向指定资源提交数据，请求服务器进行处理（例如，提交表单、上传文件）。数据包含在请求体中。
  • PUT: 用请求体中的数据替换目标资源的全部内容。
  • DELETE: 删除指定的资源。
  • HEAD: 类似于 GET，但服务器在响应中只返回头部，不返回实体部分。用于获取资源的元信息。

3. HTTP 响应报文 (Response)

一个 HTTP 响应由三部分组成：状态行、响应头、响应体。

  • 状态行 (Status Line): 协议版本 + 状态码 (Status Code) + 状态描述
  • 响应头 (Headers): 包含关于服务器、响应数据等的附加信息。
  • 响应体 (Body): 包含服务器返回的实际内容，如 HTML 页面、JSON 数据等。

常用 HTTP 状态码 (Status Codes)

  • 1xx (信息性): 表示请求已接收，继续处理。
  • 2xx (成功)
    • 200 OK: 请求成功。
  • 3xx (重定向)
    • 301 Moved Permanently: 永久重定向。
    • 302 Found: 临时重定向。
    • 304 Not Modified: 资源未被修改，客户端可以使用缓存的版本。
  • 4xx (客户端错误)
    • 400 Bad Request: 请求语法错误。
    • 401 Unauthorized: 请求需要用户认证。
    • 403 Forbidden: 服务器拒绝执行请求。
    • 404 Not Found: 请求的资源不存在。
  • 5xx (服务器错误)
    • 500 Internal Server Error: 服务器内部错误。
    • 503 Service Unavailable: 服务器暂时无法处理请求（例如，过载或停机维护）。

4. HTTPS: 安全的 HTTP

HTTPS (HTTP Secure) 并非一个新协议，它是在 HTTP 的基础上，通过 TLS/SSL 协议提供加密处理、身份认证和数据完整性保护的一种安全通信方式。

HTTPS = HTTP + TLS/SSL

TLS/SSL 的核心功能

  1. 数据加密: 通信内容使用对称加密进行加密，防止被窃听。对称加密的密钥是在连接建立时通过非对称加密协商的。
  2. 身份认证: 通过 数字证书 (Digital Certificate) 来验证服务器的身份，防止中间人攻击。证书由受信任的证书颁发机构 (CA) 签发。
  3. 数据完整性: 通过消息认证码 (MAC) 来校验数据在传输过程中是否被篡改。

HTTPS 连接建立过程 (TLS 握手)

下面是 TLS 1.2 握手过程的简化步骤和一个可视化的时序图。

  1. 客户端 Hello: 客户端向服务器发送它支持的 TLS 版本、加密套件列表和一个随机数 Client Random。
  2. 服务器 Hello: 服务器从中选择一个加密套件，返回自己的数字证书、一个随机数 Server Random。
  3. 客户端验证与密钥交换:
    • 客户端使用操作系统的内置根证书来验证服务器证书的合法性。
    • 验证通过后，客户端再生成一个随机数 Pre-master Secret，并用证书中的公钥进行加密，发送给服务器。
  4. 服务器解密与会话密钥生成:
    • 服务器用自己的私钥解密，获取 Pre-master Secret。
    • 现在，客户端和服务器都拥有了三个相同的随机数 (Client Random, Server Random, Pre-master Secret)。双方使用相同的算法，各自独立地生成本次会话所使用的对称加密密钥 (Session Key)。
  5. 握手完成: 双方互发一个“已加密”的握手完成消息。此后，所有的 HTTP 通信都将使用这个会话密钥进行对称加密。
