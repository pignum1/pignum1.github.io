JVM - 垃圾回收 (Garbage Collection, GC)

垃圾回收（GC）是 Java 语言的一大特性，它自动管理内存，使得程序员无需手动释放对象，从而专注于业务逻辑。理解 GC 的工作原理，对于编写高性能、低延迟的 Java 应用，以及排查线上内存问题至关重要。

GC 的核心任务是回答三个问题：

  1. 哪些内存需要回收？
  2. 什么时候回收？
  3. 如何回收？

1. 如何判断对象已“死”？

1.1 引用计数法 (Reference Counting)

  • 原理: 给对象中添加一个引用计数器，每当有一个地方引用它时，计数器值就加 1；当引用失效时，计数器值就减 1。任何时刻计数器为 0 的对象就是不可能再被使用的。
  • 优点: 实现简单，判定效率高。
  • 缺点: 无法解决对象之间循环引用的问题。例如，对象 A 和 B 相互引用，但没有其他任何外部引用指向它们，此时它们的引用计数都不为 0，导致无法被回收。主流的 Java 虚拟机都没有选用引用计数法来管理内存。

1.2 可达性分析算法 (Reachability Analysis)

这是主流商用程序语言（如 Java、C#）所采用的方法。

  • 原理: 通过一系列的称为 “GC Roots” 的对象作为起始点，从这些节点开始向下搜索，搜索所走过的路径称为引用链 (Reference Chain)。当一个对象到 GC Roots 没有任何引用链相连时，则证明此对象是不可用的。

  • 图解:

    • 红色部分: 对象 I 和 J 虽然相互引用，但它们与任何 GC Roots 之间都没有引用链，因此它们是不可达的，被视为可回收的垃圾。

  • 哪些对象可以作为 GC Roots？

    • 虚拟机栈（栈帧中的本地变量表）中引用的对象。
    • 本地方法栈中 JNI（即一般说的 Native 方法）引用的对象。
    • 方法区中类静态属性引用的对象。
    • 方法区中常量引用的对象。
    • 被同步锁（synchronized 关键字）持有的对象。

  • GC Roots 是如何被确认的？

    • 概念层面: GC Roots 是一组在当前时间点必须存活的对象引用，它们是垃圾回收器进行可达性分析的起点。
    • 实现层面: JVM 并不是在 GC 时去动态“搜索”GC Roots，而是在一个称为安全点 (Safepoint) 的特定位置，通过一个名为 OopMap 的数据结构来直接获取这些根引用的信息。
      • OopMap: 在 JIT 编译器编译代码时，它会分析并在代码的特定位置（安全点）记录下当前栈上和寄存器中哪些位置是对象引用。
      • 安全点 (Safepoint): 程序并不是在任意位置都可以停下来进行 GC，而是在预先定义好的“安全点”才能暂停（如循环末尾、方法调用后等）。
      • 流程: 当 GC 发生时，所有线程运行到最近的安全点暂停。然后，GC 线程通过 OopMap 立即知道了栈上的所有引用，再结合方法区的静态引用等，就构建起了完整的 GC Roots 集合，从而开始高效的可达性分析。

2. 常见的垃圾回收算法

2.1 标记-清除算法 (Mark-Sweep)

  • 过程:
    1. 标记 (Mark): 从 GC Roots 开始，标记所有可达的对象。
    2. 清除 (Sweep): 遍历整个堆，清除所有未被标记的对象。
  • 缺点:
    1. 效率问题: 标记和清除两个过程的效率都不高。
    2. 空间问题: 清除之后会产生大量不连续的内存碎片，可能导致以后需要分配较大对象时，无法找到足够的连续内存而不得不提前触发另一次垃圾收集。

2.2 复制算法 (Copying)

  • 过程: 将可用内存按容量划分为大小相等的两块，每次只使用其中的一块。当这一块的内存用完了，就将还存活着的对象复制到另外一块上面，然后再把已使用过的内存空间一次清理掉。
  • 优点: 实现简单，运行高效，不会产生内存碎片。
  • 缺点: 内存利用率低，代价是牺牲了一半的内存。
  • 应用: 主要用于新生代的垃圾回收。HotSpot VM 将新生代划分为一块较大的 Eden 空间和两块较小的 Survivor 空间（From 和 To），比例通常是 8:1:1。每次只使用 Eden 和其中一块 Survivor。

2.3 标记-整理算法 (Mark-Compact)

  • 过程:
    1. 标记 (Mark): 同标记-清除算法。
    2. 整理 (Compact): 让所有存活的对象都向一端移动，然后直接清理掉边界以外的内存。
  • 优点: 解决了标记-清除算法的内存碎片问题。
  • 缺点: 整理过程需要移动对象，效率相对较低。
  • 应用: 主要用于老年代的垃圾回收。

2.4 分代收集算法 (Generational Collection)

这是当前商业虚拟机都采用的主流思想。

  • 核心思想: 根据对象存活周期的不同将内存划分为几块，一般是把 Java 堆分为新生代 (Young Generation) 和老年代 (Old Generation)。
  • 策略:
    • 新生代: 每次垃圾收集时都有大批对象死去，只有少量存活，选用复制算法。
    • 老年代: 对象存活率高、没有额外空间对它进行分配担保，必须使用标记-清除或者标记-整理算法。

3. HotSpot VM 中的垃圾收集器

垃圾收集器是内存回收的具体实现。理解它们的特性、优势和适用场景，是进行 JVM 调优的关键。

核心概念

  • Stop-The-World (STW): 在执行垃圾回收时，所有用户线程都会被暂停。STW 的时间越短，用户感受到的卡顿就越少。
  • 吞吐量 (Throughput): 用户代码运行时间 / (用户代码运行时间 + 垃圾收集时间)。高吞吐量意味着 CPU 能高效地执行用户代码，但可能会以较长的 STW 为代价。
  • 延迟 (Latency): 指的是单次 GC 的 STW 时间。低延迟意味着用户几乎感受不到卡顿，这对于交互式应用（如网站、桌面应用）至关重要。
  • 串行 (Serial): GC 过程由单个线程完成。
  • 并行 (Parallel): GC 过程由多个线程同时完成，但仍然会 STW。
  • 并发 (Concurrent): GC 的某些阶段可以与用户线程一起执行，从而显著减少 STW 时间。

垃圾收集器对比总表

收集器组合 新生代算法 老年代算法 特点 核心优势 适用场景

各收集器详细说明

  • Serial / Serial Old:

    • 最古老、最简单的收集器。想象成一个清洁工在打扫卫生，打扫期间，所有人（用户线程）都必须停下来等他扫完。虽然慢，但因为只有一个清洁工，所以管理成本最低。

  • Parallel Scavenge / Parallel Old:

    • Parallel 意味着“并行”。想象成一群清洁工（多线程）同时开始打扫卫生。虽然所有人仍然需要停下来等他们，但因为人多，打扫得更快，总的打扫时间（GC 时间）变短了，所以大家能更快地回去工作。这使得在单位时间内，工作的总产出（吞吐量）更高。

  • CMS (Concurrent Mark Sweep):

    • Concurrent 意味着“并发”。想象成一个聪明的清洁工团队，他们只在两个很短的时刻要求所有人停一下（初始标记、重新标记），其他大部分打扫时间（并发标记、并发清除）都允许大家继续工作。
    • 优点: 极大地降低了 STW 时间，用户体验好。
    • 缺点:
      1. CPU 敏感: 因为要和用户线程抢 CPU，所以在 CPU 核心数少的情况下，可能会影响用户程序的性能。
      2. 内存碎片: 基于“标记-清除”算法，会产生大量内存碎片。
      3. 浮动垃圾: 在并发清除阶段，用户线程可能又会产生新的垃圾，这些垃圾只能等到下一次 GC 再清理。

  • G1 (Garbage-First):

    • G1 是一个里程碑式的收集器。它不再将堆简单地划分为新生代和老年代，而是划分为上百个大小相等的独立区域 (Region)。
    • 它会跟踪每个 Region 中垃圾的价值（回收能释放多大空间），并优先回收那些“价值最高”的 Region。
    • 优点:
      1. 可预测的停顿: 允许用户设定一个期望的最大停顿时间（-XX:MaxGCPauseMillis），G1 会尽力满足。
      2. 无内存碎片: 整体上基于“标记-整理”算法，不会产生碎片。
      3. 在延迟和吞吐量之间取得了很好的平衡。

  • ZGC / Shenandoah (JDK 11+):

    • 这两个是面向未来的、最前沿的收集器。它们通过更复杂的技术（如着色指针、读屏障），使得几乎所有的 GC 工作都能与用户线程并发执行，从而将 STW 时间压缩到惊人的毫秒甚至亚毫秒级别。它们是为解决超大堆内存和超低延迟这两个终极目标而生的。

4. G1 与 ZGC 深度解析

G1 和 ZGC 是现代垃圾收集器的代表，理解它们的内部细节对于进行高级 JVM 调优和构建高性能应用非常有帮助。

G1 (Garbage-First)

G1 的设计目标是在可预测的停顿时间模型下，实现高吞吐量。它不再将堆视为连续的新生代和老年代，而是引入了全新的 Region 模型。

  • 核心设计：Region 模型

    1. 化整为零: G1 将整个 Java 堆划分为多个大小相等（通常为 1MB~32MB）的独立区域 (Region)。
    2. 角色动态分配: 每个 Region 在运行时可以扮演不同的角色：Eden、Survivor、Old (老年代)。这种设计使得 GC 不再需要对整个新生代或老年代进行回收，而是可以按需回收一部分 Region。
    3. Humongous Region: 对于超过 Region 容量一半的大对象，G1 会使用一个或多个连续的特殊 Region (Humongous Region) 来存储。

  • 回收策略：Garbage-First (垃圾优先)

    1. 价值导向: G1 的核心思想是“垃圾优先”。它会跟踪每个 Region 中垃圾的价值——即回收这个 Region 能释放多少空间，以及预估的回收耗时。
    2. 构建回收价值集 (Collection Set): 在进行 Mixed GC（混合回收）时，G1 会根据用户设定的最大停顿时间（-XX:MaxGCPauseMillis），选择性地将所有新生代 Region 和部分回收价值最高的老年代 Region 组成一个回收价值集 (CSet)。
    3. 增量回收: G1 的回收是以 CSet 为单位进行的，而不是回收整个堆。这使得单次 GC 的停顿时间更容易被控制在目标范围内。

  • 性能提升的关键技术

    1. 记忆集 (Remembered Set):
      • 问题: 在只回收部分 Region 时，如何知道其他 Region 是否有对象引用了当前 Region 中的对象？
      • 解决: G1 为每个 Region 维护一个记忆集，该数据结构记录了“谁引用了我”。当 GC 一个 Region 时，只需扫描它的记忆集，就能找到所有的外部引用，而无需扫描整个堆。
    2. 写屏障 (Write Barrier):
      • 问题: 如何维护记忆集？
      • 解决: G1 使用写屏障技术。当程序执行 obj.field = another_obj 这样的赋值操作时，JVM 会插入一小段代码（写屏障），这段代码会检查 another_obj 和 obj 是否位于不同的 Region。如果是，就会更新 another_obj 所在 Region 的记忆集，记录下这个新的引用关系。

ZGC (Z Garbage Collector)

ZGC 是为解决超大堆内存（TB 级别）和超低延迟（亚毫秒级 STW）这两个终极目标而设计的。它的几乎所有工作都是并发的。

  • 核心设计：着色指针 (Colored Pointers)

    • 原理: ZGC 并没有使用传统的 Region 模型中的记忆集，而是巧妙地利用了 64 位操作系统中指针的高位。在 64 位系统中，一个指针只有低 40 多位用于寻址，高位通常是 0。ZGC 将这些高位挪作他用，用来存储元数据，如对象的标记状态。
    • 元数据: ZGC 在指针上存储了 4 个 bit 的元数据：Marked0, Marked1, Remapped, Finalizable。
    • 优势: 当 GC 需要知道一个对象的状态时，它不需要去访问对象头，只需看一下指向该对象的指针本身，就能获取信息。这使得 GC 的很多操作（如标记）可以和用户线程并发进行，而不会产生竞态条件。

  • 回收策略与并发处理
    ZGC 的整个 GC 过程几乎都是并发的，STW 只存在于几个非常短暂的瞬间（如初始标记）。

    1. 并发标记: GC 线程和用户线程并发执行。当用户线程访问一个对象时，ZGC 通过指针上的标记位就能知道该对象是否已被标记。
    2. 并发整理/重定位: 这是 ZGC 最核心的技术。
      • 重定位集 (Relocation Set): ZGC 选定一组需要被整理的 Region。
      • 并发移动: ZGC 会将这些 Region 中的存活对象移动到新的 Region。
      • 读屏障 (Read Barrier): 在并发移动期间，如果用户线程访问了一个已经被移动的对象（通过旧地址），读屏障会拦截这次访问，从转发表 (Forwarding Table) 中找到对象的新地址，更新该引用，然后再让用户线程继续访问。
      • 自愈 (Self-Healing): 这种通过读屏障在访问时修复引用的机制，被称为“自愈”。它保证了即使对象正在被并发移动，用户线程也能正确地访问到它。

  • 性能提升的关键技术

    1. 着色指针: 实现了信息与指针的绑定，是所有并发操作的基础。
    2. 读屏障: 解决了并发移动对象时，用户线程如何安全访问的问题。这是 ZGC 能够实现并发整理的关键。
    3. 不分代: ZGC 目前是一个不分代的收集器（虽然未来的版本可能会引入分代），它对整个堆进行统一处理。

JVM-性能调优
JVM - 性能调优 (JVM Performance Tuning)JVM 性能调优是 Java 高级工程师必备的技能之一。它涉及通过调整 JVM 参数、优化代码和监控工具来提升 Java 应用程序的性能、稳定性和响应速度。调优是一个系统性的工程，需要结合具体的应用场景和业务需求。 1. JVM 调优的目标在开始调优之前，首先要明确目标，因为不同的目标可能需要不同的调优策略。 降低延迟 (Latency): 减少 GC 停顿时间，提高响应速度。这对于用户交互频繁的应用（如 Web 应用、GUI 应用）至关重要。 提高吞吐量 (Throughput): 提高单位时间内处理的请求数量。这对于后台批处理、数据分析等任务密集型应用更重要。 减少内存占用: 优化内存使用，降低 OutOfMemoryError (OOM) 风险，尤其是在内存受限的环境中。 提高稳定性: 减少 GC 频率，避免频繁 Full GC 导致的长时间停顿。 2.
