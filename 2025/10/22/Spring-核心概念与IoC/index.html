Spring 核心概念之一：IoC (控制反转)

Spring 框架之所以如此强大和流行，其根基就在于两大核心思想：IoC (Inversion of Control，控制反转) 和 AOP (Aspect-Oriented Programming，面向切面编程)。本篇笔记将深入探讨 IoC。

1. IoC 是什么？—— 一种设计思想

  • 定义: IoC 是一种设计思想，而不是一种具体的技术。它的核心思想是：将原本由程序代码直接控制的对象创建、依赖关系维护，反转给一个外部的“容器”来管理。
  • 传统方式 (正向控制): 如果一个类 A 需要依赖另一个类 B，那么类 A 内部通常会通过 new B() 的方式来创建和控制类 B 的实例。
  • IoC 方式 (控制反转): 在 IoC 模式下，OrderService 不再主动创建 UserDao，而是被动地等待外部容器将 UserDao 的实例注入进来。
  • 核心目的: 解耦。通过 IoC，组件之间的耦合度大大降低。OrderService 不再关心 UserDao 是如何被创建和管理的，它只关心如何使用。这使得系统更加灵活、可维护、易于测试。

2. DI (Dependency Injection, 依赖注入) —— IoC 的实现方式

  • DI 是 IoC 的一种具体实现方式。IoC 是一种思想，而 DI 是实现这种思想最主流的手段。
  • 核心: 就是指容器在运行时，动态地将某种依赖关系注入到组件之中。
  • 常见注入方式:
    1. 构造函数注入 (Constructor Injection): 通过构造函数的参数来注入依赖。这是 Spring 官方最推荐的方式，因为它可以保证对象在创建时所有依赖都已准备好，确保了对象的不可变性和完整性。
    2. Setter 方法注入 (Setter Injection): 通过 setXxx() 方法来注入依赖。这种方式更灵活，允许注入可选的或可变的依赖。
    3. 字段注入 (Field Injection): 直接在成员变量上使用注解（如 @Autowired）进行注入。这种方式代码最简洁，但在 Spring 容器之外的环境中难以测试，且可能隐藏依赖关系，因此需谨慎使用。

3. Spring IoC 容器

  • IoC 容器: Spring 框架就是 IoC 思想的实现者。它提供了一个 IoC 容器，负责创建、配置和管理应用程序中的对象（在 Spring 中被称为 Bean），并维护它们之间的依赖关系。
  • Bean: 任何由 Spring IoC 容器管理的对象，都称为一个 Bean。
  • 核心接口:
    • BeanFactory: IoC 容器的基础接口，提供了最基本的 Bean 管理功能，采用懒加载（lazy-loading）策略，即在第一次 getBean() 时才创建 Bean 实例。
    • ApplicationContext: BeanFactory 的子接口，也是我们最常用的容器接口。它提供了更多企业级功能，如国际化、事件发布、AOP 集成等。它默认采用预加载（eager-loading）策略，即在容器启动时就创建所有单例 Bean，能更早地发现配置问题。

4. Bean 的生命周期

一个 Bean 在 Spring IoC 容器中从创建到销毁会经历一系列阶段，理解这个过程对于进行自定义扩展非常重要：

  1. 实例化 (Instantiation): Spring 容器通过反射创建 Bean 的实例。
  2. 属性填充 (Populate Properties): 容器根据配置，通过依赖注入（DI）为 Bean 的属性赋值。
  3. 初始化 (Initialization):
    • a. Aware 接口: 如果 Bean 实现了 BeanNameAware, BeanFactoryAware 等接口，Spring 会调用相应的方法，将容器自身的信息注入给 Bean。
    • b. BeanPostProcessor (前置处理): 调用 postProcessBeforeInitialization() 方法，允许在初始化之前对 Bean 进行自定义处理。
    • c. 初始化方法:
      • 首先调用由 @PostConstruct 注解的方法。
      • 然后调用 afterPropertiesSet() 方法（如果 Bean 实现了 InitializingBean）。
      • 最后调用在 XML 配置中指定的自定义 init-method。
    • d. BeanPostProcessor (后置处理): 调用 postProcessAfterInitialization() 方法，这是 AOP 动态代理实现的关键点。
  4. 使用 (In Use): Bean 准备就绪，可以被应用程序使用。
  5. 销毁 (Destruction):
    • 容器关闭时，如果 Bean 是单例的，则会进入销毁阶段。
    • 销毁方法:
      • 首先调用由 @PreDestroy 注解的方法。
      • 然后调用 destroy() 方法（如果 Bean 实现了 DisposableBean）。
      • 最后调用在 XML 配置中指定的自定义 destroy-method。
