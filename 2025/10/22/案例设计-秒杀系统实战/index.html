秒杀系统架构设计：从 10w 到 1000w QPS

秒杀是典型的高并发、写多读少（或写少读极多）场景。其核心挑战在于：如何在短时间内处理数百万倍于平时的请求，同时保证数据一致性（不超卖、不少卖）。

1. 架构全景与分层漏斗

设计秒杀系统的核心思想是 “层层过滤，漏斗模型”。将请求尽量拦截在上游，只有极少数有效请求能到达数据库。

1.1 流量漏斗模型

  1. CDN 层: 拦截 90% 静态资源请求（图片、CSS、JS）。
  2. 网关层 (Nginx/Gateway): 拦截恶意刷单、IP 限流、身份校验。
  3. 服务层 (Service):
    • 业务限流: 令牌桶算法，拒绝超载请求。
    • 缓存扣减: Redis 预减库存。
  4. 数据层 (DB): 异步写入订单，最终持久化。

2. 核心痛点与解决方案 (P7 必考)

2.1 超卖问题 (Over-selling) —— 致命 Bug

  • 错误做法: select stock from table; if (stock > 0) update stock - 1;
    • 在高并发下，多个线程同时读到 stock=1，导致卖出多份。
  • 正确做法 1 (数据库悲观锁): update stock set num = num - 1 where id = 1 and num > 0;
    • 缺点: 数据库锁竞争严重，吞吐量低。
  • 正确做法 2 (Redis + Lua 原子扣减) —— 推荐:
    • 提前把库存预热到 Redis。
    • 使用 Lua 脚本执行 get stock; if (stock > 0) decr stock; return true;。
    • Lua 脚本保证了 检查+扣减 的原子性。
    • Redis 单线程性能极高（10w+ QPS），完美适配。

2.2 最终一致性与库存回滚机制

由于 Redis 预扣减与 DB 落库是异步的，必然存在短暂的数据不一致。

  • 设计哲学：Redis 为准，DB 异步跟进
    • Redis 库存: 代表“可销售名额”。只要这里不超，系统就不会超卖。
    • DB 库存: 代表“最终成交数据”。
  • 不一致的处理逻辑:
    1. 正向路径: Redis 扣减 -> MQ 异步 -> DB 扣减。
    2. 逆向回滚 (原子性保障): 如果由于网络抖动、用户未支付、订单取消导致订单失效，必须执行 “补偿事务”。
      • 延时任务检测: 订单服务监听延时队列，发现订单失效。
      • 双向加回: 先加回 DB 库存，再加回 Redis 库存（INCR）。
  • 极端情况策略:
    • 少卖 (Over-retention): 相比超卖，少卖是可以接受的。通过每日凌晨的对账脚本，可以将由于系统异常导致的“死扣”库存手动恢复到 Redis 中，供次日秒杀继续。

2.3 流量削峰 (Peak Shaving)

  • 场景: 秒杀瞬间流量巨大，直接打数据库必挂。
  • 对策: MQ 异步下单。
    1. Redis 扣减库存成功后，发送“创建订单”消息到 MQ。
    2. 立即返回前端“排队中…”。
    3. 后台订单服务慢慢消费 MQ，创建订单入库。
    4. 前端轮询查询订单状态。

2.4 热点隔离 (Hotspot Isolation)

  • 场景: 某一款爆款商品瞬间流量过大，拖垮整个 Redis 集群。
  • 对策:
    1. 业务隔离: 秒杀服务独立部署，数据库独立。
    2. 数据隔离: 将热点 Key 部署到单独的 Redis 实例（或集群分片）。
    3. 本地缓存: 利用 Caffeine 在应用层拦截热点读请求。

3. 防刷与风控

  • 图形验证码: 秒杀前必须输入，拉长请求时间，防止机器瞬时并发。
  • 隐藏秒杀地址: 按钮点击后后端动态生成随机 URL，前端跳转，防止接口被提前扫描。
  • IP/用户限流: 同一 IP/User 每秒只能请求 1 次。

4. 兜底方案 (Failover)

  • 降级: 如果 Redis 挂了，走数据库？绝对不行！ 直接返回“活动太火爆”，保护数据库。
  • 限流: Sentinel 配置系统总 QPS 阈值，超过直接拒绝。

6. 极致优化：挑战千万级 QPS 的硬核手段

当业务量突破单点 Redis 物理极限时，需要动用架构级的“拆分”与“聚合”手段。

6.1 Redis 热点分片 (Key Backup / Sharding)

  • 痛点: 爆款商品的 Key 落在 Redis 集群的单一分片，导致该节点 CPU 和带宽瞬时爆表（Hot Key）。
  • 对策: Key 副本冗余。
    • 将 stock:item_101 复制为 stock:item_101_1, stock:item_101_2 … stock:item_101_N。
    • 副本分布在不同的 Redis 物理节点上。
    • 路由算法: hash(userId) % N。
    • 价值: 将单点热点压力均匀打散到全集群。

6.2 分段库存锁 (Inventory Segmenting)

借鉴 ConcurrentHashMap 的分段锁思想，提升写入并发度。

  • 原理: 将 10000 个库存拆分为 10 个段，每段 1000 个，分别存储在 10 个独立的 Redis Key 中。
  • 路由: 按照 userId % 10 进行库存扣减。
  • 解决“枯竭”: 如果某个段卖完了，客户端逻辑自动尝试下一个段（自旋或重定向），或者由后台脚本动态平衡各段库存。

6.3 请求合并 (Request Collapsing)

  • 原理: 在微服务层（或网关层），利用队列将 5ms 内收到的 100 个“减 1 库存”请求聚合。
  • 操作: 最终只向 Redis 发送一条 DECRBY stock 100 指令。
  • 价值: 极大降低了网络 I/O 次数和 Redis 的命令处理压力，实现“量变到质变”的性能跃迁。

6.4 动态热点探测与本地“秒杀”

  • 热点发现: 利用 Sentinel 或大厂自研的流量监控组件，实时发现 QPS 异常增长的商品 ID。
  • 热点推送到本地:
    1. 监控系统发现热点。
    2. 通过配置中心（Nacos/ZK）将“热点标志”推送到所有业务 Server。
    3. Server 开启本地缓存（Caffeine），直接在本地拦截 99% 的读请求。
    4. 极端情况下，甚至可以在本地预扣减一部分库存，再定时同步回 Redis。

7. 总结：一个请求的生命周期 (更新版)

  1. 用户点击秒杀 -> Nginx 拦截静态资源。
  2. 网关鉴权、限流。
  3. 业务服务 -> Redis Lua 脚本原子扣减库存。
    • 库存不足: 直接返回“已抢光”。
    • 库存充足:
      1. 发送订单消息到 MQ。
      2. 返回前端“排队中”。
  4. 订单服务消费 MQ -> 写数据库 -> 发送延时消息（用于超时回滚）。
  5. 前端轮询 -> 查到订单成功 -> 跳转支付。
