Redis 核心原理深度解析

Redis 是一个开源的、基于内存的、使用 C 语言编写的 Key-Value 数据库。它不仅性能极高，而且支持丰富的数据结构，被广泛应用于缓存、消息队列、分布式锁等场景。

1. 为什么 Redis 这么快？

Redis 单机 QPS 可达 10w+，这主要归功于其对系统资源极致利用的结果。

1.1 纯内存操作

内存读写速度远超磁盘。Redis 所有的业务数据都存储在内存中，这是高性能的根本。

1.2 单线程模型：化繁为简

很多人认为多线程才快，但 Redis 证明了单线程在特定场景下的优越性：无上下文切换开销、无锁竞争损耗。

1.3 核心杀手锏：I/O 多路复用 (epoll)

如果只有单线程，如何处理成千上万的并发连接？Redis 采用了 Reactor (反应堆) 设计模式：

  • 非阻塞 I/O: Redis 的 Socket 从不“死等”数据。
  • epoll (内核监听): 线程将连接交给内核的 epoll 模块，只负责处理那些已经准备好数据的活跃连接。

注意：执行过程始终是单线程的！
为了让你彻底理清 epoll 与执行的关系，看一个请求的完整生命周期：

  1. 等待 (Wait): 主线程调用 epoll_wait 等待事件。
  2. 读取 (Read): 从 Socket 读取字节流。
  3. 解析 (Parse): 将字节流解析为 Redis 命令。
  4. 执行 (Execute): 唯一主线程在内存中执行逻辑。 这是单线程的核心，保证了命令的原子性和无锁化。
  5. 写回 (Write): 将结果返回给客户端。

1.4 Redis 6.0 的“多线程”真相

Redis 6.0 引入多线程是为了解决网络 I/O 瓶颈，但其核心控制逻辑依然极其严密。

1.4.1 节奏控制：主线程如何协调 I/O 线程？

主线程作为“指挥官”，通过三个阶段的同步屏障 (Barrier) 来确保顺序：

  1. 并发读取 (Read Phase): 主线程将活跃 Socket 分配给 I/O 线程并自旋等待其解析完数据。在此之前，没有任何命令会被执行。
  2. 串行执行 (Execute Phase): 此时 I/O 线程空闲。主线程独自一人串行执行所有命令。
  3. 并发写回 (Write Phase): 执行完毕后，主线程将结果分发给 I/O 线程并发写回网卡。

1.5 深度思考：如何保证请求的先后顺序？

如果是单线程执行，一个客户端实例连续发送多次读写请求，Redis 如何保证它们的执行顺序？

这主要依靠 TCP 协议 与 单线程模型 的完美配合：

  1. TCP 顺序保证: TCP 是面向字节流的有序协议。客户端先发的请求 A 一定先于请求 B 到达服务端的网卡缓冲区。
  2. Socket 缓冲区 (FIFO): 服务端操作系统将收到的请求按序排队存入 Socket 输入缓冲区。
  3. 串行解析执行: Redis 主线程从缓冲区读取数据时，会严格按照字节流的先后顺序解析命令。由于执行是单线程的，它必须处理完前一个命令并写回响应，才会开始解析下一个命令。
  • 结论: 对于单个客户端连接，请求的顺序是严格保证的（FIFO）。
  • 注意: 对于多个客户端并发请求，由于 epoll 返回的事件顺序以及网络延迟，不同客户端间的命令会发生交织执行，但单个客户端内部的顺序依然维持。

2. 核心数据结构及其底层实现

2.1 String 底层实现：SDS (简单动态字符串)

Redis 构建了一种名为 SDS 的结构体来替代 C 原生字符串。

  • 优势: O(1) 获取长度、杜绝缓冲区溢出、内存预分配（减少系统调用）、二进制安全。

2.2 ZSet 底层实现：跳表 (SkipList)

a. 物理结构 (C 源码定义)
b. 查找逻辑：电梯哲学

想象在 100 层楼找 88 房间：先坐高层快线电梯跨越，发现过头了再下沉到低层慢速电梯，最后走楼梯精确定位。平均复杂度 O(log N)。

2.3 内存优化：ZipList (压缩列表)

Hash、List、ZSet 在元素较少时的默认实现。一整块连续内存，彻底消灭指针开销。

  • 隐患: 连锁更新 (Cascading Update)。当一个节点长度微变，可能触发后续所有节点的级联扩容，导致性能暴跌。

2.4 存储支柱：Hash 表 (Dict)

  • 渐进式 Rehash: 为了避免大 Map 扩容导致的瞬间卡顿，Redis 同时维护两个哈希表 ht[0] 和 ht[1]。每次执行请求时顺带搬迁一个哈希桶。

3. 持久化机制：RDB vs AOF

3.1 RDB (全量快照)

  • 核心：写时复制 (COW): 父进程 fork 出子进程写盘，父进程修改数据时复制页副本，不影响子进程快照。

3.2 AOF (增量日志)

  • AOF 重写 (Rewrite): 为了防止文件无限增大，Redis 会启动子进程，读取内存中的当前状态生成最精简指令覆盖旧文件。

3.3 混合持久化 (Redis 4.0+)

  • 原理: AOF 重写时，文件头部用 RDB 格式，尾部用 AOF 格式。兼顾了恢复速度和数据安全性。

4. 高可用架构与缓存实战 (P7 必考)

4.1 哨兵模式 (Sentinel) —— 自动故障转移

哨兵负责监控、提醒和自动故障转移。

  • 主客观下线:
    1. 主观下线 (SDOWN): 单个哨兵检测到超时。
    2. 客观下线 (ODOWN): 达到法定人数（Quorum）一致认为下线，触发选举。

4.2 Redis Cluster —— 水平扩展与分片

当单机性能达到瓶颈时，Redis Cluster 通过分片实现海量数据存储。

  • 哈希槽 (Hash Slot): 预设 16384 个槽。槽位 = CRC16(key) % 16384。数据通过槽位实现与物理节点的解耦。
  • 故障转移机制 (Failover):
    1. 发现故障: 节点间通过 Gossip 协议 交换状态。若超过半数 Master 认为某节点宕机，该节点被标记为 FAIL。
    2. Slave 晋升: 宕机 Master 的从节点会发起选举，剩余 Master 投票。胜出的 Slave 自动提升为新 Master。
    3. 槽位接管: 新 Master 接管旧 Master 的所有哈希槽，并广播通知全集群。数据并不会迁移到其他分片，而是由“替补”直接接管。
    4. 集群可用性: 若某个分片主从全挂且 cluster-require-full-coverage 为 yes，则整个集群不可用。
  • 去中心化: 每个节点都记录了槽位与节点的映射图。若请求的 Key 不在当前节点，节点返回 MOVED 引导客户端重定向至新 Master。

4.3 缓存实战：三大天坑防御指南

  1. 缓存穿透: 查不存在的数据。方案: 布隆过滤器 或 缓存空值。
  2. 缓存击穿: 热点 Key 过期。方案: 互斥锁 (Mutex) 或 逻辑过期。
  3. 缓存雪崩: 大量 Key 同时过期。方案: TTL 随机化、集群部署。
