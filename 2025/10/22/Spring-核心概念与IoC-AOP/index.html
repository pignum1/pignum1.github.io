Spring 核心概念：IoC 与 AOP

Spring 框架之所以如此强大和流行，其根基就在于两大核心思想：IoC (Inversion of Control，控制反转) 和 AOP (Aspect-Oriented Programming，面向切面编程)。

第一部分：IoC (控制反转)

1. IoC 是什么？—— 一种解耦的设计思想

  • 定义: IoC 是一种设计思想。它的核心是：将原本由程序代码直接控制的对象创建、依赖关系维护，反转给一个外部的“容器”来管理。
  • 核心目的: 解耦。通过 IoC，组件之间的耦合度大大降低，使得系统更加灵活、可维护、易于测试。

2. DI (Dependency Injection, 依赖注入) —— IoC 的实现方式

  • DI 是 IoC 的一种具体实现方式。
  • 常见注入方式:
    1. 构造函数注入: Spring 官方最推荐的方式，能保证依赖的完整性和不可变性。
    2. Setter 方法注入: 灵活，允许注入可选的或可变的依赖。
    3. 字段注入 (@Autowired): 代码简洁，但有一定弊端（如测试困难），需谨慎使用。

3. Spring IoC 容器

  • IoC 容器: Spring 提供的，负责创建、配置和管理应用程序中的对象（Bean）及其依赖关系。
  • 核心接口:
    • BeanFactory: IoC 容器的基础接口，采用懒加载策略。
    • ApplicationContext: BeanFactory 的子接口，功能更强大，默认采用预加载策略，能更早发现配置问题。

4. Bean 的生命周期

一个 Bean 在 IoC 容器中从创建到销毁会经历一系列阶段：

  1. 实例化: Spring 通过反射创建 Bean 实例。
  2. 属性填充: 容器通过 DI 为 Bean 的属性赋值。
  3. 初始化:
    • a. 执行 Aware 接口方法。
    • b. 执行 BeanPostProcessor 的前置处理。
    • c. 执行 @PostConstruct -> InitializingBean.afterPropertiesSet() -> 自定义 init-method。
    • d. 执行 BeanPostProcessor 的后置处理（AOP 代理在此发生）。
  4. 使用: Bean 准备就绪。
  5. 销毁: 容器关闭时，执行 @PreDestroy -> DisposableBean.destroy() -> 自定义 destroy-method。

第二部分：AOP (面向切面编程)

1. AOP 是什么？—— 一种模块化横切关注点的思想

  • 定义: AOP 是一种编程范式，它允许我们模块化地处理横切关注点。
  • 横切关注点: 指那些散布在应用程序中多个模块的、重复的功能。最典型的例子就是日志记录、事务管理、安全控制、性能监控等。
  • AOP 的解决方案: AOP 允许我们将这些横切关注点从核心业务逻辑中分离出来，定义到一个独立的模块中（称为“切面”），然后通过声明的方式，在程序运行的特定时机（“连接点”）将这些功能动态地织入到我们的业务代码中。

2. AOP 核心术语

  • Aspect (切面): 对横切关注点的模块化封装。
  • Join Point (连接点): 程序执行过程中的一个特定点。在 Spring AOP 中，连接点总是方法的执行。
  • Advice (通知): 在切面的某个特定连接点上执行的操作。如 @Before, @After, @Around 等。
  • Pointcut (切点): 一个匹配连接点的表达式，定义了通知应该被应用到哪些连接点上。
  • Weaving (织入): 将切面应用到目标对象，并创建代理对象的过程。

3. Spring AOP 的实现原理

Spring AOP 是在运行时，通过动态代理技术来实现的。

  • 两种代理方式:
    • JDK 动态代理: 如果目标对象实现了接口，Spring 默认使用 JDK 动态代理。
    • CGLIB 代理: 如果目标对象没有实现接口，Spring 会使用 CGLIB 生成一个目标对象的子类作为代理。
  • 工作流程: 容器启动时，发现 Bean 上有切面配置，会为该 Bean 创建一个代理对象。当调用该 Bean 的方法时，实际是调用代理对象的方法，代理方法在执行目标方法前后，会先执行切面定义的通知逻辑。

第三部分：IoC 与 AOP 核心面试题

1. 谈谈你对 Spring IoC 的理解。

  • 思路: 是什么 (IoC 是解耦思想，DI 是实现) -> 为什么 (解决传统 new 对象的耦合问题) -> 怎么做 (Spring 容器通过扫描、创建、注入来管理 Bean) -> 注入方式 (推荐构造函数注入)。

2. Bean 的生命周期是怎样的？

  • 思路: 按顺序描述四大阶段：实例化 -> 属性填充 -> 初始化 (包含 Aware 接口、BeanPostProcessor 前置处理、初始化方法、BeanPostProcessor 后置处理) -> 销毁。特别指出 AOP 代理在 BeanPostProcessor 后置处理中发生。

3. Spring 如何解决循环依赖？

  • 前提: 仅限单例的、Setter 或字段注入的 Bean。构造函数注入无法解决。
  • 核心机制: 三级缓存。
    1. 一级缓存 (singletonObjects): 存放完全初始化好的 Bean。
    2. 二级缓存 (earlySingletonObjects): 存放早期暴露的 Bean（已实例化，未初始化）。
    3. 三级缓存 (singletonFactories): 存放能生成早期 Bean 的工厂。
  • 过程简述 (A 依赖 B, B 依赖 A): 创建 A，实例化后将其工厂放入三级缓存 -> 注入 A 的属性时发现 B，去创建 B -> 创建 B，实例化后将其工厂也放入三级缓存 -> 注入 B 的属性时发现 A，从三级缓存中获取 A 的工厂，生成 A 的早期对象放入二级缓存，并注入给 B。B 完成创建 -> A 获取到 B，完成创建。

4. 谈谈你对 Spring AOP 的理解。

  • 思路: 是什么 (模块化横切关注点) -> 核心术语 (Aspect, Advice, Pointcut) -> 怎么做 (运行时动态代理) -> 两种代理方式 (JDK 动态代理基于接口，CGLIB 基于继承)。

5. Spring 的事务是如何实现的？

  • 核心: 基于 AOP 和数据库事务。
  • 实现原理: Spring 为 @Transactional 注解的 Bean 创建一个代理对象。
    • 事务开启: 在目标方法执行前，代理对象开启一个数据库事务。
    • 方法执行: 执行原始的业务逻辑。
    • 事务提交/回滚: 如果方法正常执行完毕，则提交事务；如果方法抛出运行时异常或 Error，则回滚事务。
  • 失效场景: 非 public 方法、内部调用 (绕过了代理)、异常被 catch。
