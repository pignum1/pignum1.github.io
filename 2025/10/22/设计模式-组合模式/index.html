什么是组合模式？

组合模式 (Composite Pattern) 是一种结构型设计模式，它允许你将对象组合成树形结构，以表示“部分-整体”的层次关系。组合模式使得客户端可以统一地处理单个对象（叶子节点）和由多个对象组成的组合对象（容器节点）。

通俗地讲，组合模式就像电脑里的文件系统。一个文件夹（容器）里面可以包含文件（叶子），也可以包含其他文件夹（容器）。对于用户来说，无论是文件还是文件夹，都可以执行类似的操作，比如查看属性、移动、删除等。你无需关心你操作的究竟是一个单独的文件，还是一个包含了成百上千个文件和子文件夹的复杂文件夹，你的操作方式是统一的。

模式结构

组合模式主要包含以下角色：

  1. Component (组件): 是组合中对象的抽象，为组合中的所有对象（叶子和容器）定义了统一的接口。它声明了用于访问和管理子组件的接口（对于容器是具体实现，对于叶子则是空实现）。
  2. Leaf (叶子): 表示组合中的叶子节点对象，它没有子节点。它实现了组件接口。
  3. Composite (容器/组合): 表示组合中的容器节点对象，它可以包含子节点（叶子或其他容器）。它实现了组件接口，并实现了管理子组件的方法。
  4. Client (客户端): 通过组件接口来操作组合中的对象。

优缺点

优点

  1. 简化客户端代码: 客户端可以统一地处理所有对象（无论是叶子还是容器），无需编写复杂的 if-else 判断，这使得客户端代码非常简洁。
  2. 易于扩展: 可以很容易地增加新的组件类型（新的叶子或容器），只需让它们实现统一的组件接口即可，符合开闭原则。
  3. 清晰的层次结构: 能够清晰地表示对象的“部分-整体”层次结构。

缺点

  1. 设计变得过于通用: 有时，你可能希望限制一个容器中只能添加特定类型的组件。但在组合模式中，由于叶子和容器都实现了相同的接口，编译系统无法在编译时对此进行检查，只能在运行时进行判断。
  2. 接口定义困难: 组件接口的设计可能很棘手。如果将添加/删除子组件的方法放在组件接口中（如本例），那么叶子类就需要提供一个空的或抛出异常的实现，这违反了接口隔离原则。如果只在容器类中定义这些方法，又会破坏透明性，客户端需要区分叶子和容器。

经典应用

  1. 图形用户界面 (GUI): GUI 框架是组合模式的经典应用。一个窗口 (Window) 是一个容器，它可以包含面板 (Panel)；一个面板又是一个容器，它可以包含按钮 (Button)、文本框 (TextBox) 等。按钮和文本框是叶子。你可以统一地对这些组件执行 draw() 或 repaint() 操作。
  2. 组织架构: 在表示公司或部门的组织架构时，一个部门（容器）可以包含员工（叶子）和其他子部门（容器）。
  3. 抽象语法树 (AST): 在编译器或解释器中，源代码被解析成一个树形结构（AST）。每个节点，无论是表示一个操作符、一个变量还是一个代码块，都可以被统一处理。
  4. java.awt.Container: Java AWT/Swing 中的 Container 和 Component 类就是组合模式的实现。
