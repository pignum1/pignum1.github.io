高并发与高可用实战：挡、疏、导的艺术

在高并发场景下，系统的设计核心不再是“如何实现功能”，而是“如何优雅地应对极端流量”。

1. 负载均衡：流量的“守门员”

大厂通常采用 LVS + Nginx 的混合架构。

1.1 四层 LVS vs 七层 Nginx

  • LVS (OSI 第四层 - 传输层):
    • 原理: 仅根据 IP 和 端口进行转发。
    • 性能: 极高（百万级 QPS）。它不解析 HTTP 协议，只负责把报文甩给后面的 Nginx。
  • Nginx (OSI 第七层 - 应用层):
    • 原理: 解析 HTTP 头部、URL、Cookie。
    • 功能: 支持动静分离、Gzip 压缩、反向代理、SSL 卸载。
  • 最佳实践: LVS 作为总入口（虚拟 IP），将流量负载到多台 Nginx 上，Nginx 再负载到真正的业务微服务上。

2. 限流算法：保护系统的“大闸”

当流量超过系统极限时，限流是最后的尊严。

2.1 漏桶算法 (Leaky Bucket)

  • 原理: 水（请求）进入桶，桶底下有个洞，以恒定速率滴出。
  • 特点: 绝对平滑。适合需要强制平稳处理的场景，无法应对突发流量。

2.2 令牌桶算法 (Token Bucket) —— 行业主流 (Guava RateLimiter)

  • 原理: 系统以固定速率往桶里放令牌。请求进来必须拿走一个令牌，否则被限流。
  • 特点: 允许突发流量。如果桶里攒了 100 个令牌，瞬时进来 100 个请求可以被直接放行。

3. 熔断与降级：系统的“保险丝”

3.1 熔断 (Circuit Breaker)

  • 机制: 当下游服务错误率超过阈值，直接“断开”调用，返回兜底数据。
  • 状态机: Closed (正常) -> Open (熔断) -> Half-Open (半开，尝试恢复)。

3.2 降级 (Fallback)

  • 逻辑: 在熔断或系统过载时，执行备选方案。
    • 例子: 推荐系统挂了，降级为显示“热门排行榜”。
    • 例子: 评论系统太慢，暂时关闭评论显示。

4. 数据一致性：缓存与数据库的“爱恨情仇”

这是 P7 面试的高频死穴。在高并发下，如何保证缓存和数据库一致？

4.1 核心痛点：先更新谁？

  1. 先更库，再更缓存: 若更完库，更缓存失败，则旧数据永驻缓存。
  2. 先更库，再删缓存: 主流方案 (Cache-Aside Pattern)。虽然仍有极小概率不一致，但配合 TTL 可接受。

4.2 终极解决方案：基于 Binlog 的异步同步

为了彻底解耦业务代码与缓存逻辑，目前行业 P7 级的标准做法是使用 数据订阅机制。

  • 核心组件：Alibaba Canal

    • 原理: Canal 模拟 MySQL Slave 的交互协议，伪装成从库向 Master 发送 dump 协议。Master 以为它是从库，便将 Binlog 推送给它。
    • 流程:
      1. 业务逻辑只管更新数据库。
      2. Canal 监听 Binlog 变动。
      3. Canal 将变动消息发送到 MQ (RocketMQ/Kafka)。
      4. 专门的 缓存服务 订阅 MQ，根据消息精准删除/更新 Redis。
    • 优势: 业务代码 0 侵入；利用 MQ 的重试机制实现最终一致性；可以同时支持 ES 索引更新、数据分析等多个下游。

  • 进阶：Alibaba Otter

    • 如果涉及到跨机房、跨城市的数据库双向同步（异地多活），阿里还提供了一个基于 Canal 开发的更高级工具 —— Otter。它解决了 Canal 仅能单向订阅的问题，并能处理双向同步中的冲突。

6. 性能优化巅峰：分库分表与异步化

当缓存和索引已经榨干最后一点性能，而数据量依然以亿级增长时，必须通过分布式存储手段进行水平扩容。

6.1 分库分表的抉择

  • 垂直分库: 按业务模块拆分。例如将 User 和 Order 从一个库挪到两个库。解决了数据库连接池的竞争，实现了业务解耦。
  • 水平分表: 将一张大表（如 t_order）按某种规则拆分成 t_order_0 到 t_order_N。解决了单表数据量过大导致的 B+ 树过深、索引查询变慢的问题。

6.2 核心算法与分片键 (Sharding Key)

  • 取模 (Mod): 扩展性差（扩容需全量搬迁），但数据分布极其均匀。
  • 范围 (Range): 易于扩容，但存在“热点问题”（新数据全堆在最后一个分片）。
  • 最佳实践: 采用 取模 + 预分片。初始直接分 256 或 1024 张表，支撑未来 5-10 年增长。

6.3 架构师必须面对的“副作用”

  1. 跨库 Join 难题:
    • 方案 A: 字段冗余（空间换时间）。
    • 方案 B: ER 分片。确保关联的数据（如用户与该用户的订单）落入同一个分片。
    • 方案 C: 应用层组装。多次查询后在代码中通过 Map 聚合。
  2. 分布式 ID: 必须弃用数据库自增，改用 Snowflake 或 美团 Leaf。
  3. 分页查询优化:
    • 禁止深度分页（如 limit 1000000, 10）。
    • 利用“上一页最后一条 ID”进行过滤查询（即 where id > max_id limit 10）。

6.4 异步化实战：削峰填谷与最终一致性

在高并发环境下，同步处理每一项任务会拖慢响应速度，并增加数据库压力。

  • 核心与非核心链路剥离:
    • 核心逻辑: 扣减库存、创建订单（同步完成，保证实时性）。
    • 非核心逻辑: 发送短信通知、积分累计、物流初始化（通过 MQ 异步处理）。
  • 削峰填谷: 利用 MQ 的缓冲能力。当 QPS 瞬时激增（如秒杀开始），请求先打入 MQ，后台服务根据自身处理能力匀速消费，保护后端数据库不被打死。
  • 事务消息 (RocketMQ 为例):
    1. 生产者发送“半消息”到 Broker。
    2. 执行本地数据库事务。
    3. 根据本地事务结果二次确认（Commit/Rollback）。
    4. 确保了“数据库更新”与“异步通知”的原子性，实现最终一致性。

6.5 多级缓存架构：追求极致 RT

单靠 Redis 也会遇到网络 IO 瓶颈。P7 级架构通常采用 L1 本地缓存 + L2 分布式缓存。

  • L1 - Caffeine (本地内存): 读写速度极快（纳秒级），无网络开销。
  • L2 - Redis (分布式缓存): 保证多实例间的数据共享，容量大。
  • 协同逻辑:
    1. 请求先查 L1。
    2. L1 未命中查 L2，并回填 L1。
    3. L2 未命中查 DB，并回填 L2 和 L1。
  • 热点探测机制: 通过日志分析或 Sentinel 监控，发现“大热点 Key”，实时推送到所有业务实例的本地缓存中，防御“单点 Redis 流量过载”。
  • 防抖机制: 设置不同的过期时间，并配合随机偏移（Jitter），防止缓存同时失效导致的缓存雪崩。

7. 高可用实战思考题

Q: 异地多活架构中，如何解决数据同步延迟导致的读写冲突？

  • 方案: 单元化 (Unitization)。按照用户 ID 进行 Hash 路由，某个用户永远只在 A 机房读写，A 异步同步到 B。只有当 A 挂了，才切换到 B。

Q: 为什么 Nginx 负载均衡中常用“一致性哈希”？

  • 目的: 保证相同的请求落在相同的 Nginx 节点上。这对于本地缓存命中率至关重要（避免每台 Nginx 都存一份全量缓存）。
