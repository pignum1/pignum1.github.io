JavaSE - 异常处理 (Exception Handling)

异常处理是 Java 语言中一个强大的机制，它提供了一种结构化的方式来处理程序在运行时可能出现的错误或“异常”情况，从而增强程序的健壮性和容错性。

1. 异常体系结构

Java 中所有的异常都派生自 java.lang.Throwable 类。Throwable 有两个主要的子类：Error 和 Exception。

Error (错误)

  • 定义: Error 类及其子类代表了 Java 运行时环境中出现的、非常严重的、通常无法恢复的错误。这些错误通常与 JVM 本身的资源耗尽或硬件问题有关。
  • 特点: 应用程序不应该也无法捕获和处理 Error。
  • 示例:
    • OutOfMemoryError: 内存溢出错误。
    • StackOverflowError: 栈溢出错误（通常由无限递归引起）。

Exception (异常)

Exception 类及其子类代表了程序本身可以处理的异常情况。它又分为两大类：

  1. 受检异常 (Checked Exception)

    • 定义: 除了 RuntimeException 及其子类之外的所有 Exception。它们是编译器在编译阶段强制要求程序员必须处理的异常。
    • 处理方式: 必须使用 try-catch 块捕获，或者在方法签名上使用 throws 关键字声明抛出。
    • 目的: 提醒开发者注意并处理那些在正常情况下也可能发生的、可预见的外部问题。
    • 示例: IOException (读写文件时发生错误), SQLException (数据库访问错误), ClassNotFoundException。

  2. 非受检异常 / 运行时异常 (Unchecked Exception / RuntimeException)

    • 定义: RuntimeException 类及其所有子类。
    • 特点: 编译器不强制要求处理。这些异常通常是由程序自身的逻辑错误引起的。
    • 目的: 如果这类异常频繁发生，通常意味着代码存在 Bug，应该修改代码逻辑来避免它，而不是通过 try-catch 来掩盖问题。
    • 示例:
      • NullPointerException: 对 null 对象进行操作。
      • ArrayIndexOutOfBoundsException: 数组索引越界。
      • ClassCastException: 类型转换异常。
      • IllegalArgumentException: 非法参数异常。

2. 异常处理机制

try-catch-finally

这是 Java 中处理异常的核心语法结构。

  • try 块: 包围可能产生异常的代码。
  • catch 块: 用于捕获并处理特定类型的异常。可以有多个 catch 块，用于捕获不同类型的异常。如果异常类型有继承关系，应该将子类异常的 catch 块放在父类异常的前面。
  • finally 块: 无论 try 块中是否发生异常，finally 块中的代码总会被执行。它通常用于执行资源释放操作，如关闭文件流、数据库连接等，以确保资源不会泄露。

throws 关键字

如果一个方法内部可能会抛出受检异常，但它本身不打算处理，可以使用 throws 关键字在方法签名上声明该异常，将处理责任“甩锅”给调用者。

throw 关键字

throw 用于在代码中主动抛出一个异常对象。通常用于在检测到某种错误逻辑时，手动创建一个异常实例并将其抛出。

3. try-with-resources (Java 7+)

在 finally 块中释放资源虽然可行，但代码比较冗长。Java 7 引入了 try-with-resources 语句，可以更优雅地处理资源释放。

  • 要求: 资源类必须实现 java.lang.AutoCloseable 接口。
  • 工作方式: 在 try 关键字后的括号中声明并初始化资源。当 try 块执行完毕或发生异常时，JVM 会自动调用这些资源的 close() 方法。

4. 异常处理最佳实践

  1. 只捕获需要处理的异常: 不要用一个宽泛的 catch (Exception e) 来捕获所有异常，这会掩盖程序中潜在的 RuntimeException。
  2. 不要“生吞”异常: 空的 catch 块是“异常黑洞”，它会悄无声息地隐藏问题，给调试带来巨大困难。至少应该记录日志。
  3. finally 优先于 return: 如果 try 块和 finally 块中都有 return 语句，finally 块中的 return 会覆盖 try 块的 return。应避免在 finally 中使用 return。
  4. 抛出具体的异常: 尽量抛出能描述具体问题的异常类型，而不是笼统的 Exception。
  5. 优先使用 try-with-resources: 对于需要关闭的资源，优先使用 try-with-resources 来确保其被正确关闭。
