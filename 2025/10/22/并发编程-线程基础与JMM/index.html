并发编程 - 线程基础与 JMM

并发编程是 Java 高级开发中不可或缺的一部分。理解线程的生命周期、状态转换以及 Java 内存模型（JMM）是进行高效、安全并发编程的基础。

1. 线程与进程

  • 进程 (Process): 操作系统资源分配的最小单位。每个进程都有自己独立的内存空间、文件句柄等资源。
  • 线程 (Thread): CPU 调度的最小单位，是进程中的一个执行流。一个进程可以包含多个线程，这些线程共享进程的内存空间和资源。

2. Java 中线程的创建方式

2.1 implements Runnable vs extends Thread

特性 implements Runnable (推荐) extends Thread
本质 任务 (Task) 线程 (Worker)
耦合度 低 (任务与线程解耦) 高 (任务与线程耦合)
继承限制 无 (类可以继承其他类) 有 (Java 单继承限制)
资源共享 容易 (多个线程可共享一个 Runnable 实例) 复杂 (需要将共享资源设为 static)
设计思想 面向接口编程 面向实现编程

2.2 实现 Runnable 接口 (推荐)

  • 核心思想: 将“做什么”（任务）和“谁去做”（线程）分离开。Runnable 接口只定义了一个 run() 方法，它代表一个需要被执行的任务。
  • 优点:
    1. 解耦，更灵活: 同一个 Runnable 任务可以被多个 Thread 或线程池执行。
    2. 避免单继承限制: 你的类可以继承其他类。
    3. 易于资源共享: 多个线程可以共享同一个 Runnable 实例及其成员变量。

2.3 继承 Thread 类

  • 核心思想: 将“任务”和“执行者”绑定在一起。你创建的类本质上就是一个“自带任务的特殊线程”。
  • 缺点:
    1. 高耦合: 任务逻辑和线程控制逻辑紧密地耦合在同一个类中。
    2. 单继承限制: 如果类有其他父类，此路不通。
    3. 资源共享复杂: 每个 Thread 实例都是一个独立的对象，它们的成员变量不共享。如果多个线程需要共享资源，该资源必须被定义为 static。

2.4 实现 Callable 接口 (带返回值和异常)

  • 优点: call() 方法可以有返回值，也可以抛出异常。
  • 使用: 通常与 FutureTask 或线程池 (ExecutorService) 配合使用。

2.5 核心区别：thread.start() vs thread.run()

这是一个基础但至关重要的区别。

  • thread.start(): 创建一个新线程，并由新线程去执行 run() 方法。这是一个异步/并发调用。
  • thread.run(): 不创建新线程，只是在当前线程中执行 run() 方法。这是一个同步/顺序调用。

结论: 如果你想实现多线程并发，必须调用 start() 方法。

3. 线程池 (Thread Pool)

在实际生产环境中，直接 new Thread() 是极不推荐的，因为线程的创建和销毁是昂贵的资源操作。线程池是现代并发编程的基石。

3.1 为什么需要线程池？(核心思想)

想象一个银行，有很多客户来办业务。

  • 没有线程池 (来一个客户，new 一个柜员):

    • 每来一个客户，银行就临时招聘一个新柜员（new Thread()）。客户办完业务，柜员就被解雇（Thread 销毁）。
    • 问题: 开销巨大，且无法控制并发数量，可能导致系统资源耗尽。

  • 有线程池 (固定数量的柜员):

    • 银行预先招聘好固定数量的柜员（核心线程），让他们待命。
    • 客户来了，就去空闲的柜台办业务。
    • 如果柜台都满了，新来的客户就在大厅的等候区（任务队列）排队。
    • 优势:
      1. 降低资源消耗: 复用已存在的线程，减少线程创建和销毁的开销。
      2. 提高响应速度: 任务到达时，无需等待线程创建即可执行。
      3. 提高可管理性: 可以统一管理、分配、调优和监控线程。

3.2 核心实现：ThreadPoolExecutor

Java 中线程池的核心实现是 java.util.concurrent.ThreadPoolExecutor。理解其构造函数中的七大核心参数是配置线程池的关键。

  1. corePoolSize: 核心线程数。
  2. maximumPoolSize: 当任务队列满了，可以创建的最大线程数。
  3. keepAliveTime: 超出 corePoolSize 的空闲线程的存活时间。
  4. unit: keepAliveTime 的时间单位。
  5. workQueue: 任务队列，常用的有 ArrayBlockingQueue (有界)、LinkedBlockingQueue (近乎无界)、SynchronousQueue (不存储元素)。
  6. threadFactory: 用于创建新线程，可以自定义线程名称等。
  7. rejectedExecutionHandler: 当线程池和任务队列都满了之后的拒绝策略。常用的有 AbortPolicy (抛异常)、CallerRunsPolicy (调用者执行)、DiscardPolicy (丢弃)、DiscardOldestPolicy (丢弃最老的)。

3.3 如何配置线程池？

实际开发中，强烈推荐手动创建 ThreadPoolExecutor，而不是使用 Executors 工具类（如 newFixedThreadPool），因为后者可能使用无界队列导致 OOM。

  • CPU 密集型任务 (计算密集型):
    • 配置: 线程数 = CPU 核心数 + 1。
  • I/O 密集型任务 (网络请求、数据库操作):
    • 配置: 线程数 = CPU 核心数 * 2 或 线程数 = CPU 核心数 / (1 - 阻塞系数)。可以配置更多的线程，以便在一个线程等待 I/O 时，其他线程能利用 CPU。

3. 线程的生命周期与状态

Java 线程有 6 种状态，定义在 Thread.State 枚举中。

    %%{init: {'theme': 'neutral'}}%%
graph TD
    A[NEW] --> B[RUNNABLE]
    B -- 获取锁失败 --> C[BLOCKED]
    C -- 获取锁成功 --> B
    B -- wait()/join()/park() --> D[WAITING]
    D -- notify()/unpark() --> B
    B -- sleep()/wait(long) --> E[TIMED_WAITING]
    E -- 时间到/notify()/unpark() --> B
    B -- run()结束/异常 --> F[TERMINATED]

    classDef state fill:#cff,stroke:#333,stroke-width:2px;
    class A,B,C,D,E,F state;

  • NEW (新建): 线程被创建但尚未启动。

  • RUNNABLE (可运行): 这是 Java 虚拟机中线程执行的状态。它包含了两个子状态：

    • Ready (就绪): 线程已经准备好运行，正在等待操作系统的 CPU 调度器分配时间片。
    • Running (运行中): 线程已经获得了 CPU 时间片，正在执行代码。
      从 JVM 的视角看，这两个子状态都统一为 RUNNABLE。

  • BLOCKED (阻塞): 线程等待获取监视器锁（synchronized 块或方法）。

  • WAITING (等待): 线程无限期等待另一个线程执行特定操作（如 Object.wait(), Thread.join(), LockSupport.park()）。

  • TIMED_WAITING (有时限等待): 线程在指定时间量内等待另一个线程执行特定操作（如 Thread.sleep(), Object.wait(long), Thread.join(long), LockSupport.parkNanos(), LockSupport.parkUntil()）。

  • TERMINATED (终止): 线程已执行完毕。

一个处于 RUNNABLE 状态的线程，其下一个状态可能是：

  1. BLOCKED: 试图进入一个已被其他线程持有的 synchronized 块。
  2. WAITING: 调用了 Object.wait(), Thread.join() 等方法。
  3. TIMED_WAITING: 调用了 Thread.sleep(long), Object.wait(long) 等方法。
  4. TERMINATED: run() 方法执行完毕或因异常退出。
  5. 仍然是 RUNNABLE: 线程的时间片用完，从“运行中”回到“就绪”状态，等待下一次 CPU 调度。

4. 线程的常用方法

  • start(): 启动线程，使线程进入 RUNNABLE 状态，并调用其 run() 方法。
  • run(): 线程执行的任务代码。
  • sleep(long millis): 使当前线程进入 TIMED_WAITING 状态，休眠指定时间，不释放锁。
  • join(): 等待线程终止。当前线程会进入 WAITING 状态，直到调用 join() 的线程执行完毕。
  • interrupt(): 中断线程。它并不会立即停止线程，而是设置线程的中断标志位。被中断的线程可以检查这个标志位来决定如何响应中断。
  • yield(): 线程让步。提示调度器当前线程愿意放弃当前 CPU 使用权，让其他线程运行，但调度器不一定会采纳。

5. Java 内存模型 (Java Memory Model, JMM)

JMM 是 Java 虚拟机规范中定义的一种内存模型，它屏蔽了各种硬件和操作系统的内存访问差异，保证了 Java 程序在多线程环境下的可见性、原子性、有序性。

5.1 主内存与工作内存

  • 主内存 (Main Memory): 所有线程共享的内存区域，对应物理内存。所有共享变量都存储在主内存中。
  • 工作内存 (Working Memory): 每个线程独有的内存区域，对应 CPU 缓存。线程对变量的所有操作都在工作内存中进行，不能直接读写主内存。线程对共享变量的修改，首先写入工作内存，然后刷新到主内存。

5.2 JMM 的三大特性

  1. 可见性 (Visibility):

    • 定义: 一个线程对共享变量的修改，另一个线程能够立即看到。
    • 保证方式:
      • volatile 关键字: 保证可见性和有序性，但不保证原子性。当一个变量被 volatile 修饰时，对它的写操作会立即刷新到主内存，对它的读操作会从主内存重新加载。
      • synchronized 关键字: 保证可见性、原子性和有序性。在释放锁之前，会把工作内存中的共享变量刷新到主内存；在获取锁之后，会清空工作内存，从主内存重新加载共享变量。
      • final 关键字: 被 final 修饰的字段在构造器中一旦初始化完成，并且构造器没有逸出，那么在其他线程中就能看到 final 字段的初始化值。

  2. 原子性 (Atomicity):

    • 定义: 一个或多个操作，要么全部执行成功，要么全部不执行，并且在执行过程中不会被任何因素打断。
    • 保证方式:
      • synchronized 关键字: 保证被锁住的代码块是原子性的。
      • java.util.concurrent.atomic 包下的原子类: 如 AtomicInteger, AtomicLong, AtomicReference 等，通过 CAS (Compare-And-Swap) 操作保证原子性。

  3. 有序性 (Ordering):

    • 定义: 程序执行的顺序按照代码的先后顺序执行。
    • 指令重排: 编译器和处理器为了优化性能，可能会对指令进行重排，但会保证单线程内的执行结果不变。
    • 保证方式:
      • volatile 关键字: 禁止指令重排。它会在写操作后插入一个写屏障，在读操作前插入一个读屏障，从而保证有序性。
      • synchronized 关键字: 保证有序性。它会禁止其修饰的代码块内部的指令重排，以及代码块与外部指令的重排。
      • happens-before 原则: JMM 定义的一系列偏序关系，保证了多线程操作的可见性。例如：
        • 程序次序规则：在一个线程内，按照程序代码顺序，前面的操作 happens-before 后面的操作。
        • 管程锁定规则：一个 unlock 操作 happens-before 后面对同一个锁的 lock 操作。
        • volatile 变量规则：对一个 volatile 变量的写操作 happens-before 后面对这个变量的读操作。
        • 线程启动规则：Thread 对象的 start() 方法 happens-before 此线程的每一个动作。
        • 线程终止规则：线程中的所有操作 happens-before 对此线程的终止检测。
        • 线程中断规则：对线程 interrupt() 方法的调用 happens-before 被中断线程的代码检测到中断事件的发生。
        • 对象终结规则：一个对象的初始化完成 happens-before 它的 finalize() 方法的开始。
        • 传递性：如果 A happens-before B，且 B happens-before C，那么 A happens-before C。

6. volatile 关键字深度解析

volatile 被称为“轻量级的 synchronized”，因为它在多线程环境下，不通过加锁就能保证共享变量的可见性和有序性。

6.1 volatile 的两大核心功能

a. 保证可见性 (Visibility)
  • 问题: 在 JMM 中，每个线程都有自己的工作内存（对应 CPU 缓存）。当一个线程修改了共享变量后，这个修改会先保存在自己的工作内存中，并不会立即同步到主内存。此时，其他线程的工作内存中仍然是旧值，导致数据不一致。
  • volatile 的作用: 当一个变量被声明为 volatile 后，JMM 会保证：
    1. 写操作: 对这个变量的写操作会立即将新值刷新到主内存中。
    2. 读操作: 在读取这个变量之前，会强制从主内存中重新加载最新的值到工作内存。
  • 效果: 通过这种机制，volatile 确保了任何一个线程对该变量的修改，都会对其他所有线程立即可见。
b. 禁止指令重排 (Ordering)
  • 问题: 为了提高性能，编译器和处理器可能会对代码指令进行重排序。在单线程中，重排序会保证最终结果一致，但在多线程中，它可能会破坏程序的逻辑。
    • 经典案例：双重检查锁定 (Double-Checked Locking) 的单例模式 instance = new Singleton() 这行代码实际上分为三步：1. 分配内存空间；2. 初始化对象；3. 将 instance 引用指向分配的内存地址。如果发生指令重排，顺序可能变为 1 -> 3 -> 2。此时，线程 A 执行了 1 和 3，但还没执行 2，instance 已经不为 null 了。如果线程 B 此时进入，它会判断 instance 不为 null 并直接返回一个尚未初始化完成的对象，导致程序出错。
  • volatile 的作用: volatile 关键字可以禁止对其修饰的变量进行指令重排优化。
  • 效果: 在 volatile 变量的赋值操作前后，编译器和 JVM 会插入内存屏障 (Memory Barriers)，确保其前面的操作一定在其前面完成，其后面的操作一定在其后面开始。

6.2 volatile 的实现原理

volatile 的功能是通过内存屏障 (Memory Barriers) 来实现的。内存屏障是一种 CPU 指令，它有两个主要作用：

  1. 防止指令重排: 确保屏障前后的指令不会被重排序。
  2. 刷新处理器缓存: 强制将对缓存的修改操作立即写入主内存，或使其他 CPU 的缓存失效。

当一个变量被声明为 volatile 时，JMM 会在生成的字节码中，围绕这个变量的读写操作插入相应的内存屏障，从底层硬件层面保证了 volatile 的可见性和有序性。

6.3 volatile vs synchronized

特性 volatile synchronized
保证 可见性、有序性 可见性、有序性、原子性
级别 变量级别 代码块/方法级别
加锁 不加锁，不阻塞线程 加锁，会阻塞线程
性能 轻量级，性能更高 重量级，性能相对较低

总结:
volatile 是一个轻量级的同步工具，适用于一写多读、状态标记等场景，它不保证原子性（例如 i++ 操作就不是原子的）。当你需要保证一个代码块的原子性时，仍然必须使用 synchronized 或 java.util.concurrent.locks.Lock。
