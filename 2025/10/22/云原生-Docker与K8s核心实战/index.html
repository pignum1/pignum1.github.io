云原生架构：从容器化到容器编排

云原生的核心在于利用容器化和编排技术，实现应用的快速交付、弹性伸缩和故障自愈。

1. Docker：容器化的基石

1.1 分层存储与 Copy-on-Write

Docker 镜像是只读的，由多个层（Layer）叠加而成。

  • 分层: 基础镜像（如 Ubuntu）+ 环境（如 JDK）+ 代码。
  • Copy-on-Write: 只有当需要修改底层文件时，才会将文件复制到最上面的“读写层”。这极大节省了存储空间和镜像传输效率。

1.2 镜像瘦身：P7 级最佳实践

  1. 使用精简镜像: 优先选择 alpine (仅 5MB) 或 slim 版本。
  2. 多阶段构建 (Multi-stage Build):
    • 第一阶段：编译代码（需要完整的 JDK/Maven 环境）。
    • 第二阶段：仅拷贝编译好的 JAR 到运行环境（只需要极简的 JRE）。
    • 结果: 镜像体积可从 800MB 缩减至 100MB 左右。
  3. 减少层数: 合并 RUN 指令，清理临时缓存。

2. Kubernetes (K8s)：生产环境的指挥官

K8s 的本质是 “期望状态管理”：你定义一个 YAML 描述“我想要 3 个 Pod”，K8s 就拼命保证线上永远有 3 个。

2.1 核心组件与对象

  • Pod: 最小调度单位，一个 Pod 内的容器共享网络栈（localhost 互访）。
  • Deployment: 管理 Pod 的副本数、滚动更新（Rolling Update）。
  • Service: Pod 的逻辑代理，解决 Pod IP 经常变动的问题（提供负载均衡）。
  • Ingress: 集群入口，实现域名转发、SSL 卸载（类似集群内部的 Nginx）。

2.2 生产级运维：健康检查

  • Liveness Probe (存活检查): Pod 挂了？重启它。
  • Readiness Probe (就绪检查): Pod 启动还没加载完数据？先别把流量导进来。

2.3 弹性伸缩 (HPA)

根据 CPU/内存利用率，自动增加或减少 Pod 数量。应对突发流量的利器。

3. 服务网格 (Service Mesh) —— Istio

当微服务数量破百，熔断、限流、灰度发布的配置变得极其复杂。Istio 出现了。

  • Sidecar (边车模式): 在每个 Pod 里塞进一个 Envoy 代理。所有的网络流量先过 Envoy。
  • 价值:
    • 业务解耦: 开发者只管写逻辑，限流、加密、监控全部交给 Istio 平台。
    • 金丝雀发布: 完美实现“让 5% 的用户先体验新版，没问题再全量”。

4. 可观测性：系统的“眼睛”

  • 指标监控 (Metrics): Prometheus 抓取数据，Grafana 绘图。
  • 日志系统 (Logging): ELK (Elasticsearch + Logstash + Kibana) 或更轻量级的 Loki。
  • 链路追踪 (Tracing): SkyWalking 或 Jaeger。解决“一个请求报错，到底是哪个服务出的问题”的痛点。

5. 架构师 P7 级思考题

Q: 容器里的数据持久化怎么做？

  • A: 利用 K8s 的 PV (Persistent Volume) 和 PVC (Persistent Volume Claim)。通过 StorageClass 动态挂载云盘（如阿里云 ESSD），保证容器重启数据不丢。

Q: 为什么要用 K8s，不用传统的 VM 部署？

  • A: 极高的资源利用率、秒级启动、极致的横向扩展能力、统一的基础设施声明（YAML）。
