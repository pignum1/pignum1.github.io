Sentinel 深度解析：分布式系统的流量防卫兵

Sentinel 是阿里巴巴开源的流量治理组件，其核心职责是：在流量洪峰到来时，保护系统不被冲垮；在依赖的服务出现问题时，及时熔断，防止雪崩效应。

1. Sentinel 核心理念：一切皆资源

在 Sentinel 中，所有的保护对象都被抽象为资源 (Resource)。

  • 资源可以是：一个方法、一段代码、一个 REST 接口、一个 Dubbo 服务等。
  • 入口: SphU.entry("resourceName")。如果资源被允许访问，则正常返回；如果被限流或熔断，则抛出 BlockException。

2. 核心架构：ProcessorSlotChain (插槽链) 源码深度剖析

Sentinel 的执行流程由一个插槽链驱动。每一项功能（限流、降级、统计）都是链条上的一个节点。

2.1 插槽链是如何诞生的？(SPI 机制)

Sentinel 并没有在代码中写死插槽列表，而是采用了极其灵活的 SPI (Service Provider Interface) 机制：

  1. SlotChainProvider: 当一个资源第一次被访问时，它会触发链条的创建。
  2. SpiLoader: 它扫描配置文件，加载所有的 ProcessorSlot 实现。
  3. DefaultSlotChainBuilder: 根据每个 Slot 类上的 @Order 注解对它们进行升序排列。

2.2 插槽链的执行机制

  • 结构: ProcessorSlotChain 内部维护了一个单向链表。
  • 流转: 每个 Slot 执行完逻辑后，通过调用 fireEntry() 触发下一个 Slot。这是一种典型的责任链模式。

2.3 核心统计插槽：StatisticSlot

这是唯一一个在请求进入 (entry) 和 退出 (exit) 阶段都要干活的节点。它负责维护并发线程数，并统计 QPS、RT 和异常数。

3. 统计的灵魂：LeapArray (滑动窗口算法) 源码级解析

Sentinel 能够精准统计秒级流量，解决了传统计数器算法在“时间临界点”统计不准的问题。

3.1 核心挑战：临界点问题

假设限流 100 QPS。请求在 0.9s 发送 100 个，1.1s 发送 100 个。传统计数器认为两秒都未超限，但 0.9s-1.1s 期间实际并发为 200，足以冲垮系统。滑动窗口通过更细粒度的“格子”解决了这个问题。

3.2 寻找格子的源码逻辑 (三步曲)

  1. 定位索引: idx = time / windowLength % arrayLength。
  2. 判断状态:
    • 空: CAS 创建。
    • 有效: 直接返回。
    • 过期: 触发“滑动”，CAS 重置旧格子数据并更新开始时间。
  3. 高性能累加: 内部使用 LongAdder 替代 AtomicLong，通过分散热点彻底解决了高并发下的原子变量竞争损耗。

4. 流量控制策略 (Flow Control)

4.1 直接拒绝 (Direct Reject)

  • 原理: 快速失败。请求到达时立即对比 当前 QPS + 1 与阈值。

4.2 预热 (Warm Up / 冷启动)

  • 原理: 改进版令牌桶。让阈值从 设定值 / 3 开始，在预热时间内线性增加到设定值。防止冷系统因瞬时高压而崩溃。

  • 原理: 漏桶算法。强制请求以固定间隔通过。

5. 熔断降级策略 (Circuit Breaking)

5.1 熔断状态机

  • CLOSED: 正常。
  • OPEN: 熔断。拒绝所有请求。
  • HALF_OPEN: 半开。允许一个请求探测，成功则关闭熔断，失败则重新打开。

5.2 触发策略

  1. 慢调用比例: RT 超过阈值的请求比例达到上限。
  2. 异常比例 / 异常数: 业务执行抛出的错误达到设定阈值。

6. 进阶：热点参数限流 (ParamFlowSlot)

针对具体的业务参数（如特定的商品 ID、用户 ID）进行精准限流。

6.1 实现原理

为每一个特定的参数值维护一套独立的令牌桶。它内部使用了 LRU (最近最少使用) 淘汰策略来管理海量参数值的统计内存，防止 OOM。

6.2 复杂对象设计：如何处理 JSON Body 内字段？

ParamFlowSlot 默认识别基础参数。对于 JSON 对象，有三种设计方案：

  1. 手动提取: 业务代码中先解析字段，手动传入 SphU.entry。性能最高，但代码侵入重。
  2. AOP + SpEL (推荐): 自定义切面，利用 SpEL 表达式（如 #order.itemId）动态提取字段。解耦性好。
  3. 网关级拦截: 在网关层利用 GatewayFlowSlot 的 Body 解析策略。架构级解耦，保护最彻底。

7. 常见面试题

Q1: Sentinel 为什么比 Hystrix 性能好？

  • Sentinel 采用信号量隔离（并发线程数控制），避免了线程切换开销；且使用了高性能的滑动窗口与 LongAdder 统计。

Q2: Sentinel 如何实现规则实时生效？

  • 通过 ReadableDataSource 接口集成 Nacos/Apollo。当规则在配置中心修改，Sentinel 客户端会收到推送并立即更新内存规则。
