Spring MVC 核心原理与请求流程

Spring MVC 是基于 Servlet API 构建的原始 Web 框架。它采用了 Front Controller (前端控制器) 设计模式，所有的请求都通过一个统一的入口 DispatcherServlet 进行分发。

1. 核心组件 (The Big 4)

在 Spring MVC 中，有四个接口最为关键，它们共同完成了请求的路由、执行和展示。

组件名 职责 核心作用
HandlerMapping 路由查找 根据请求的 URL 找到对应的 Handler (通常是 Controller 中的方法) 以及拦截器。
HandlerAdapter 业务执行 负责调用 Handler。由于 Handler 形式多样（注解、接口等），适配器屏蔽了调用细节。
HandlerExceptionResolver 异常处理 处理在请求执行过程中抛出的异常，将其转化为特定的错误页面或 JSON。
ViewResolver 结果渲染 将逻辑视图名（如 “index”）解析为真正的物理视图（如 JSP 或 HTML）。

2. 请求处理全流程 (9 个步骤)

这是一个 HTTP 请求在 Spring MVC 内部经历的“朝圣之旅”。

  1. 请求到达: 所有的请求首先到达 DispatcherServlet。
  2. 获取执行链: DispatcherServlet 请求 HandlerMapping 寻找处理器。返回一个 HandlerExecutionChain（包含 Handler 和一组 Interceptors）。
  3. 获取适配器: DispatcherServlet 寻找能执行该 Handler 的 HandlerAdapter。
  4. 执行拦截器 (Pre): 调用所有拦截器的 preHandle 方法。
  5. 执行 Handler: 适配器调用 Controller 的业务逻辑。
  6. 执行拦截器 (Post): 调用所有拦截器的 postHandle 方法。
  7. 异常处理: 如果过程中报错，由 HandlerExceptionResolver 接管。
  8. 视图解析: ViewResolver 根据逻辑视图名找到物理页面（现代 RESTful API 通常跳过此步，直接返回 JSON）。
  9. 渲染与返回: 渲染页面，并最后执行拦截器的 afterCompletion 方法。

3. Spring MVC vs. WebFlux

随着异步编程的兴起，Spring 5 引入了 WebFlux。

特性 Spring MVC Spring WebFlux
模型 Servlet Stack (阻塞) Reactive Stack (非阻塞)
并发模型 线程池（一请求一线程） 事件循环（少量的固定线程）
依赖 Servlet API Project Reactor (Flux/Mono)
适用场景 传统关系型数据库、阻塞式调用 高并发、长连接、非阻塞 I/O

4. 常见问题 (Q&A)

Q: Spring MVC 是如何将请求参数绑定到 Controller 方法参数上的？
A: 靠的是 HandlerMethodArgumentResolver。Spring 预设了许多解析器，分别处理 @RequestParam, @PathVariable, @RequestBody 等。

Q: 如何在 Spring MVC 中实现全局异常处理？
A: 使用 @ControllerAdvice + @ExceptionHandler。这本质上是往 HandlerExceptionResolver 链条里注册了一个高效的处理器。

Q: 拦截器 (Interceptor) 和 过滤器 (Filter) 的区别？

  • Filter: Servlet 规范定义，在 DispatcherServlet 之前执行，处理最通用的逻辑。
  • Interceptor: Spring MVC 定义，能感知到 Handler 信息，在 DispatcherServlet 之后、Controller 之前执行。

5. 源码深度解析：DispatcherServlet.doDispatch

doDispatch 是 Spring MVC 的心脏，它控制着请求处理的完整生命周期。我们将核心逻辑拆解为“一找、二适、三拦、四演、五善后”。

核心源码流程

关键步骤详解

  1. getHandler (寻址): 遍历所有注册的 HandlerMapping，找到能处理当前 URL 的 Handler。返回的不仅仅是 Handler 对象，而是一个包裹了所有匹配拦截器的 HandlerExecutionChain。
  2. getHandlerAdapter (适配): 遍历 HandlerAdapter，找到能执行当前 Handler 的适配器。这屏蔽了 Handler 实现方式的差异（无论是注解方法、还是实现 Controller 接口）。
  3. applyPreHandle (安检): 顺序执行所有拦截器的 preHandle。这是权限校验的最佳时机。
  4. handle (执行): 真正的业务逻辑执行。Controller 的方法被调用，参数绑定 (ArgumentResolver) 也是在这一步发生的。
  5. processDispatchResult (收尾):
    • 异常处理: 检查是否有异常抛出，如果有，调用 processHandlerException 寻找异常解析器。
    • 视图渲染: 如果返回了 ModelAndView，调用 render 方法进行页面渲染（前后端分离场景下通常跳过此步）。
    • 全局清理: 无论成功失败，最终都会触发拦截器的 afterCompletion，用于资源释放（如 ThreadLocal 清理）。
