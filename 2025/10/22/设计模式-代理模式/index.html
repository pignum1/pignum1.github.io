什么是代理模式？

代理模式 (Proxy Pattern) 是一种结构型设计模式，它为你提供了一个对象的替代品或占位符，以便控制对该对象的访问。使用代理，你可以在将请求传递给实际服务对象之前或之后，执行某些操作。

通俗地讲，代理就像一个“中介”或“经纪人”。比如，你想买一套国外的房子（目标对象），但你对当地的法律、市场都不了解，直接去买会很麻烦。于是你找到了一个当地的房产中介（代理）。你所有的购房请求都先发给中介，中介会帮你处理各种琐事（如验证房源、办理手续），然后再去和房主进行实际的交易。这个中介就控制和管理了你对房子的直接访问。

模式结构

代理模式主要包含以下角色：

  1. Subject (主题接口): 定义了真实对象和代理对象的共同接口。这样，任何使用真实对象的地方都可以使用代理对象。
  2. RealSubject (真实主题): 实现了主题接口，是代理所代表的真实对象，是最终执行业务逻辑的实体。
  3. Proxy (代理): 实现了主题接口，并持有一个真实主题的引用。它可以在将请求委派给真实主题之前或之后，执行额外的操作。

代理的类型与代码示例

代理模式主要分为静态代理和动态代理两大类。

1. 静态代理

静态代理是指在编译时就已经创建好了代理类。代理类和真实主题类都实现同一个接口，代理类中硬编码了对真实主题的调用。

静态代理 vs. 动态代理：使用场景对比

静态代理的使用场景

  1. 目标对象数量少且固定: 当需要代理的类不多（例如，只有一两个），并且它们的业务逻辑在未来很少改变时。因为每个代理类都需要手动编写，数量多了维护成本会急剧上升。
  2. 对性能有极致要求: 静态代理在编译时就生成了 .class 文件，相比于动态代理在运行时通过反射生成，少了一层反射的开销，执行速度更快。虽然这点差距在绝大多数应用中微不足道，但在某些极端高性能场景下可能会被考虑。
  3. 代理逻辑差异大: 如果每个代理类的附加逻辑都完全不同，无法用一个统一的 InvocationHandler 来处理，那么为每个真实类编写一个特定的静态代理类可能更直接。

动态代理的使用场景

  1. AOP（面向切面编程）: 这是动态代理最核心、最广泛的应用场景。框架（如 Spring）需要在不侵入业务代码的情况下，为一批符合规则的方法（切点）统一添加事务、安全、日志等横切关注点。通过动态代理，可以批量地为这些方法所在的类生成代理，并在方法调用前后织入增强逻辑。
  2. RPC（远程过程调用）框架: 客户端调用一个本地接口，动态代理拦截调用，并将方法名和参数序列化后通过网络发送到远端服务器。例如 Dubbo、gRPC 的客户端存根（Stub）就是通过动态代理实现的，让远程调用看起来就像本地调用一样。
  3. ORM 框架: 像 MyBatis 中，你只定义了 Mapper 接口而没有实现类。当你调用接口方法时，MyBatis 通过动态代理为你生成一个代理对象，该代理对象会拦截方法调用，然后根据方法名和注解去执行对应的 SQL 语句。
  4. 需要为大量类提供统一代理逻辑的场景: 当需要代理的类数量很多，但附加的代理逻辑是标准化的（例如，统一的日志记录、权限校验），使用动态代理可以极大地减少代码量。你只需要编写一个 InvocationHandler，就可以为所有实现了某个接口的类创建代理。

常见面试题

  1. 问：什么是代理模式？它解决了什么问题？

    • 答: 代理模式是为其他对象提供一种代理以控制对这个对象的访问。它解决了在不修改目标对象代码的前提下，为其增加额外控制和操作（如权限校验、日志记录、懒加载、远程通信等）的问题，实现了对目标对象的访问控制和功能增强。

  2. 问：静态代理和动态代理有什么区别？

    • 答:
      • 实现方式: 静态代理需要手动为每个目标类编写一个代理类，在编译期就确定了代理关系。动态代理则是在运行时通过反射机制动态地创建代理类，无需手动编写。
      • 灵活性: 动态代理非常灵活，一个调用处理器 (InvocationHandler) 可以服务于实现了相同接口的多个目标对象，代码复用性高。而静态代理每增加一个目标类，就需要新增一个代理类。
      • 适用性: 静态代理适用于代理类较少、关系固定的情况。动态代理适用于需要为大量类提供统一代理逻辑的场景，是实现 AOP、RPC 等框架功能的基石。

  3. 问：JDK 动态代理和 CGLIB 动态代理有什么区别？

    • 答: 这是非常高频的面试题。主要区别在于实现原理和限制：
      • 实现原理: JDK 动态代理是基于接口的。它要求目标对象必须实现一个或多个接口。生成的代理类和目标类是兄弟关系，都实现了相同的接口。CGLIB 是基于继承的，它通过创建一个目标类的子类来实现代理，因此不需要目标类实现接口。
      • 限制: JDK 代理无法为没有实现接口的普通类创建代理。CGLIB 虽然可以为普通类创建代理，但无法为 final 类或 final 方法创建代理，因为它们不能被继承或重写。
      • 性能: 在早期版本中，CGLIB 因为不使用反射而性能通常优于 JDK 代理。但在现代 JDK 版本中（如 1.8+），JDK 动态代理的性能得到了很大优化，两者差距已经很小，有时 JDK 代理甚至更快。

  4. 问：谈谈你对 Spring AOP 的理解，它是如何选择代理方式的？

    • 答: Spring AOP 是面向切面编程，用于将横切关注点（如事务、日志）与业务逻辑分离，是典型的代理模式应用。Spring AOP 在创建代理时会进行智能选择：
      • 如果目标对象实现了接口，Spring 默认会使用 JDK 动态代理。
      • 如果目标对象没有实现接口，Spring 会使用 CGLIB 动态代理。
      • 你也可以通过配置（proxy-target-class="true"）来强制 Spring 使用 CGLIB 代理。

优缺点

优点

  1. 职责清晰: 真实主题只需关注核心业务逻辑，而代理可以处理与业务无关的附加任务（如权限校验、日志记录）。
  2. 开闭原则: 可以在不修改真实主题的情况下，通过增加代理来扩展功能。
  3. 控制访问: 代理可以作为访问控制的屏障，保护真实主题。
  4. 懒加载 (Lazy Loading): 对于创建成本高的对象，可以使用虚拟代理来实现懒加载，优化系统性能。

缺点

  1. 增加复杂性: 引入了额外的代理类，增加了系统的复杂性。
  2. 可能降低性能: 请求需要通过代理转发，可能会引入一定的性能延迟（尽管通常可以忽略）。

经典应用

  1. Spring AOP: Spring 的面向切面编程 (AOP) 就是基于动态代理实现的。它允许你在方法执行前后添加通知 (Advice)，如事务管理 (@Transactional)、日志记录、安全检查等。
  2. Java RMI (远程方法调用): RMI 使用代理作为“存根 (Stub)”，客户端调用存根的方法，存根负责将调用信息打包并通过网络发送到远程服务器，服务器端的“骨架 (Skeleton)”接收请求并调用真实对象的方法。
  3. MyBatis 框架: 当你调用 Mapper 接口的方法时，MyBatis 会为你创建一个动态代理，这个代理会拦截方法调用，然后去执行对应的 SQL 语句。
  4. Hibernate/JPA 懒加载: 当你从数据库加载一个实体时，其关联的集合（如 List）可能是一个代理对象。只有当你第一次访问这个集合时，代理才会真正去数据库加载数据。
