分布式系统核心理论与 P7 级实践

1. 理论基石：CAP 与 BASE

1.1 CAP 定理的真相

在分布式系统中，P (分区容错性) 是客观存在的，无法避免（网络总会断）。因此，CAP 实际上是在分区发生时，选择 CP 还是 AP：

  • CP (强一致性): 为了数据绝对准确，宁可拒绝服务。适用于金融转账（Zookeeper, Redis 强一致模式）。
  • AP (高可用性): 为了保证用户能用，宁可数据暂时不一致。适用于社交、推荐系统（Eureka, Cassandra）。

1.2 BASE 理论 (BASE = Basically Available, Soft state, Eventually consistent)

这是对 CAP 中 AP 的延伸。核心思想是：即使无法达到强一致性，也可以采用适当的方式达到最终一致性。

2. 分布式 ID：从 Snowflake 到大厂实践

2.1 原生 Snowflake 的痛点

  1. 时钟回拨: 机器时钟校准导致 ID 重复。
  2. 机器 ID 分配: 手动配置 workerId 极其繁琐且易出错。

2.2 美团 Leaf 的深度优化

美团 Leaf 提供了两种模式，解决了大多数工程问题：

  • Leaf-segment (号段模式):
    • 原理: 每次从 DB 取一个 ID 号段（如 1000-2000）加载到内存。
    • 优势: 即使 DB 挂了，内存里的号段仍能支撑一段时间；通过 双 Buffer 机制，在号段消耗到 10% 时异步加载下一个号段，消除瞬时 TP99 抖动。
  • Leaf-snowflake (雪花模式):
    • 自动分配: 利用 Zookeeper 的顺序节点自动分配 workerId。
    • 回拨处理: 启动时检查时钟；运行中若发现回拨（偏差小则等待，偏差大则报警并拒绝服务）。

3. 共识算法深度剖析：Raft

Raft 是目前生产环境下最稳健的算法。理解它需要关注 任期 (Term) 机制。

3.1 脑裂 (Split-Brain) 如何解决？

当网络分区时，可能会出现两个 Leader（旧 Leader 和新选举出的 Leader）。

  • 处理机制: Raft 引入了 Term。每个 Leader 发出的心跳都带有 Term。
  • 结果: 旧 Leader 的 Term 较小，Follower 收到请求后会拒绝，并回复：“你的任期已结束”。旧 Leader 收到高任期回复后，立即退位。由于旧 Leader 无法获得超过半数的确认，它无法提交任何数据，从而保证了一致性。

3.2 日志安全性 (Log Matching)

如果一个节点的日志落后于集群，它绝对无法成为 Leader。Raft 通过对比“最后一条日志的 Term 和 Index”来保证只有拥有最全数据的节点才能上位。

4. 分布式锁：不仅仅是 SETNX

4.1 Redis 红锁 (Redlock) 的争议

Redis 作者 Antirez 提出了 Redlock 以解决单点故障问题：向 N 个独立节点申请锁，过半成功则认为获取。

  • 分布式专家 Martin 的质疑:
    1. 时钟跳跃: 如果某台机器时钟突然变快，锁会提前过期。
    2. GC 暂停: 进程在获取锁后进入长时间 GC，锁过期了，GC 结束进程以为自己还持有锁。
  • 结论: 如果业务对安全性要求达到“金融级”，请务必使用 Zookeeper 锁。

4.2 Zookeeper 锁的优越性

  • 临时顺序节点: 自动解决客户端宕机导致死锁的问题。
  • Watch 机制: 客户端只需监听前一个节点的删除事件，避免了 Redis 的频繁轮询（Spin Lock），性能抖动更小。

5. 核心面试挑战

Q: 为什么 Raft 要求必须有奇数个节点？

  • 不是必须，但奇数更具“性价比”。例如 3 节点能容忍 1 个坏，4 节点依然只能容忍 1 个坏（因为 50% 无法通过过半验证）。

Q: 如何在海量数据环境下实现分布式锁的性能优化？

  • 分段锁 (Lock Stripping): 将一个锁拆成 10 个（类似 ConcurrentHashMap），通过 Key % 10 将并发压力分流。
