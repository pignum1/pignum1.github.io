Web技术栈-核心概念

本篇笔记将聚焦于 Java 后端 Web 开发的基石，逐一深入 Servlet 规范、HTTP 协议、Web 服务器以及 RESTful API 设计。

第一部分：Servlet 详解

1. Servlet 是什么？

  • 定义: Servlet (Server Applet) 是运行在 Web 服务器（如 Tomcat）中的一个 Java 小程序。它的核心职责是接收来自客户端的 HTTP 请求，处理该请求，并生成 HTTP 响应返回给客户端。
  • 本质: Servlet 是 Java 定义的一套 API 规范。

2. Servlet 的生命周期

Servlet 的生命周期由 Servlet 容器管理，其核心特点是单例。

  1. 加载与实例化: 默认在第一个请求到达时，或可配置在 Web 应用启动时。仅一次。
  2. 初始化 (init() 方法): 实例化之后调用，用于一次性初始化操作。仅一次。
  3. 处理请求 (service() 方法): 每当有请求到达时，容器会创建一个新线程调用此方法。每次请求都会调用。
  4. 销毁 (destroy() 方法): Web 应用关闭或 Servlet 实例被移除时调用，用于资源清理。仅一次。

3. 线程安全问题

  • 根源: Servlet 实例是单例的，但 service() 方法会被多个线程并发访问。
  • 风险: 在 Servlet 中定义可变的成员变量会导致线程安全问题。
  • 最佳实践: 不要使用成员变量来保存与请求相关的状态，应在局部变量中处理。

第二部分：Filter 详解

  • 定义: 一个可以拦截请求和响应的 Web 组件，位于客户端和目标 Servlet 之间，形成“过滤器链”。
  • 核心方法: doFilter(request, response, chain)。
  • 应用场景: 统一编码、登录鉴权、日志记录、性能监控、XSS 防御等。

第三部分：Listener 详解

  • 定义: 用于监听 Web 应用中特定事件（如应用启动/销毁、会话创建/销毁）的发生，并执行相应代码。
  • 分类: ServletContext (应用级), HttpSession (会话级), ServletRequest (请求级)。
  • 应用场景: 应用启动时初始化资源（如连接池）、统计在线人数。

第四部分：HTTP/HTTPS 协议

1. HTTP (超文本传输协议)

  • 特点: 无连接、无状态、灵活、基于 TCP/IP。
  • 请求方法: GET, POST, PUT, DELETE 等。
  • 状态码: 2xx (成功), 3xx (重定向), 4xx (客户端错误), 5xx (服务器错误)。

2. HTTPS (安全的超文本传输协议)

  • 是什么: HTTPS = HTTP + SSL/TLS。在 HTTP 和 TCP 之间增加了安全层。
  • 核心功能:
    1. 内容加密: 防止中间人窃听。
    2. 身份认证: 验证服务器身份，防止钓鱼。
    3. 数据完整性: 校验数据在传输过程中是否被篡改。

3. HTTPS 握手详解 (以 TLS 1.2 为例)

HTTPS 握手的核心目标是：安全地协商出一套对称加密密钥，用于后续的 HTTP 数据传输。

第一阶段：客户端发起请求 (Client Hello)

  1. 客户端向服务器发送 Client Hello 消息，包含：
    • 支持的 TLS 版本。
    • 一个客户端生成的随机数 (Client Random)。
    • 支持的**加密套件 (Cipher Suites)**列表。

第二阶段：服务器响应 (Server Hello & Certificate)
2. 服务器响应 Server Hello 消息，包含：
* 确认的 TLS 版本和加密套件。
* 一个服务器生成的随机数 (Server Random)。
3. 服务器发送 Certificate 消息，其中包含服务器的数字证书（内含服务器公钥）。

第三阶段：客户端验证与密钥交换
4. 客户端验证服务器证书的合法性（是否由信任的CA颁发、是否过期、域名是否匹配等）。
5. 证书验证通过后，客户端生成第三个随机数 (Pre-master Secret)。
6. 客户端用服务器的公钥加密 Pre-master Secret，并通过 Client Key Exchange 消息发送给服务器。
7. 客户端发送 Change Cipher Spec 和 Finished 消息，通知服务器后续将使用协商好的密钥加密通信，并发送第一条加密的握手校验信息。

第四阶段：服务器完成握手
8. 服务器用自己的私钥解密，得到 Pre-master Secret。
9. 此时，客户端和服务器拥有相同的三个随机数，双方各自使用相同的算法，生成唯一的、对称的会话密钥 (Session Key)。
10. 服务器发送 Change Cipher Spec 和 Finished 消息，确认握手过程，并开始使用会话密钥加密通信。

握手结束，安全通信开始
后续的所有 HTTP 数据都将使用这个对称会话密钥进行加密传输。

第五部分：Web 服务器 (Tomcat)

  • 核心组件: Server -> Service -> Connector (处理连接) -> Container (处理业务)。
  • Container 四个级别: Engine -> Host -> Context -> Wrapper。
  • 请求处理流程: Connector -> Engine -> Host -> Context -> Wrapper -> Servlet.service()。
  • 线程模型: 使用线程池处理请求，高效处理并发。

第六部分：RESTful API 设计

  • 核心原则:
    1. 资源 (Resource): URI 代表资源，使用名词。
    2. 表现层 (Representation): 资源可以有多种表现形式（JSON, XML）。
    3. 状态转化 (State Transfer): HTTP 方法 (GET, POST, PUT, DELETE) 改变资源状态。
    4. 无状态 (Stateless): 服务器不保存客户端会话状态。
  • 最佳实践:
    • 在 URL 中使用名词。
    • 在 URL 中添加版本号（如 /api/v1）。
    • 使用合适的 HTTP 状态码。
    • 设计统一的响应数据格式。
