Web技术栈 - 核心概念

本篇笔记将聚焦于 Java 后端 Web 开发的基石，包括 Servlet 规范、HTTP 协议、Web 服务器以及 RESTful API 设计。

1. Servlet/JSP 核心概念

1.1 什么是 Servlet？

  • 定义: Servlet (Server Applet) 是运行在 Web 服务器（如 Tomcat）中的小型 Java 程序。它接收来自客户端的 HTTP 请求，进行处理，然后生成 HTTP 响应并将其发送回客户端。
  • 本质: Servlet 是 Java EE (现为 Jakarta EE) 提供的一套 API 规范。
  • 生命周期 (单例): init() -> service() (每次请求) -> destroy()。
  • 线程安全: Servlet 实例是单例的，应避免在其中使用可变的成员变量。

1.2 Filter (过滤器)

  • 定义: 一个可以拦截和处理请求和响应的对象，位于客户端和目标 Servlet 之间，形成“过滤器链”。
  • 核心方法: doFilter(request, response, chain)。
  • 应用场景: 统一编码、登录鉴权、日志记录、数据压缩。

1.3 Listener (监听器)

  • 定义: 用于监听 Web 应用中特定事件（如应用启动/销毁、会话创建/销毁）的发生，并执行相应代码。
  • 应用场景: 应用启动时初始化资源（如连接池）、统计在线人数。

2. HTTP/HTTPS 协议

2.1 HTTP (超文本传输协议)

  • 特点: 无连接、无状态、灵活、基于 TCP/IP。
  • 请求方法: GET, POST, PUT, DELETE, HEAD, OPTIONS。
  • 状态码:
    • 2xx (成功): 200 OK, 201 Created。
    • 3xx (重定向): 301 Moved Permanently, 302 Found, 304 Not Modified。
    • 4xx (客户端错误): 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found。
    • 5xx (服务器错误): 500 Internal Server Error。

2.2 HTTPS (安全的超文本传输协议)

  • 是什么: HTTPS = HTTP + SSL/TLS。
  • 核心功能:
    1. 内容加密: 采用混合加密（对称加密+非对称加密）。
    2. 身份认证: 通过数字证书验证服务器身份。
    3. 数据完整性: 通过 MAC (消息认证码) 校验数据。
  • 握手过程: 核心是客户端用服务器的公钥加密一个预主密钥，发送给服务器，服务器用私钥解密，然后双方根据约定的算法生成对称的会话密钥，用于后续通信。

3. Web 服务器 (以 Tomcat 为例)

Tomcat 是一个开源的、实现了 Java Servlet 和 JavaServer Pages (JSP) 规范的 Web 服务器和 Servlet 容器。

3.1 Tomcat 核心组件

  • Server: 代表整个 Tomcat 实例，是顶层组件。
  • Service: 将一个或多个 Connector 关联到一个 Engine。
  • Connector (连接器): 负责处理外部连接。它监听端口，接收请求，然后将请求转换为 HttpServletRequest 对象，并传递给容器处理。
  • Container (容器): 负责处理业务逻辑。有四个级别：
    1. Engine: 最高级别的容器。
    2. Host: 代表一个虚拟主机（域名）。
    3. Context: 代表一个 Web 应用。
    4. Wrapper: 代表一个单独的 Servlet。

请求处理流程:
Connector 接收请求 -> Engine -> Host -> Context -> Wrapper -> Servlet.service()

3.2 Tomcat 线程模型

  • Tomcat 使用线程池来处理请求，避免了为每个请求都创建新线程的开销，从而高效地处理大量并发请求。

4. RESTful API 设计

REST (Representational State Transfer)，即“表现层状态转化”，是一种软件架构风格，而非标准。它强调通过 HTTP 协议中已有的概念来设计和组织网络服务。

4.1 核心原则

  1. 资源 (Resource): 网络上的一切事物都被抽象为资源，通过唯一的 URI 标识。
  2. 表现层 (Representation): 资源可以有多种表现形式（JSON, XML 等），客户端和服务器之间传递的是资源的某种表现形式。
  3. 状态转化 (State Transfer): 客户端通过 HTTP 方法（GET, POST, PUT, DELETE等）实现服务端资源状态的改变。
  4. 无状态 (Stateless): 服务器不保存任何关于客户端的会话状态，每个请求都应包含所有必要信息。

4.2 最佳实践

  • 使用名词而非动词: URI 应表示资源。
    • 推荐: GET /users/123
    • 不推荐: GET /getUser?id=123
  • 版本控制: 将 API 版本号放入 URL 中，如 /api/v1/users。
  • 使用合适的 HTTP 状态码: 明确地告知客户端操作的结果。
  • 统一的数据格式: 设计统一的响应结构，如包含 code, message, data 等字段。
