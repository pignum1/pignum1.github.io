RabbitMQ 核心概念速查

RabbitMQ 是基于 AMQP (高级消息队列协议) 实现的一个成熟、稳定的消息队列。它由 Erlang 开发，天生具备高并发和高可用的特性。相比 Kafka，它更强调消息路由的灵活性。

1. 核心模型：Exchange -> Queue

在 RabbitMQ 中，生产者并不直接将消息发送到 Queue，而是发送给 Exchange (交换机)。

  • Producer: 消息生产者。
  • Exchange: 负责接收消息，并根据路由规则将消息转发到 Queue。
  • Queue: 消息存储的容器，等待消费者拉取。
  • Binding: Exchange 与 Queue 之间的关联关系。
  • Routing Key: 消息发送时携带的路由标志。

2. 四种交换机类型 (核心区别)

这是 RabbitMQ 路由逻辑的精华所在：

  1. Direct Exchange (直连):
    • 规则: Routing Key 必须完全匹配。
    • 场景: 具体的任务分发。
  2. Fanout Exchange (广播):
    • 规则: 忽略 Routing Key，将消息发送给所有绑定到该 Exchange 的 Queue。
    • 场景: 全网通告、配置更新推送。
  3. Topic Exchange (通配符):
    • 规则: 按照模式匹配。
      • * 匹配一个单词。
      • # 匹配零个或多个单词。
    • 场景: 日志系统，按级别和模块过滤（如 error.# 或 *.user.*）。
  4. Headers Exchange:
    • 规则: 不看 Key，看消息头中的属性。由于性能较低，很少使用。

3. 可靠性投递机制

  1. 生产者确认 (Confirm): 消息到达 Exchange 后给生产者的回执。
  2. 返回机制 (Return): 消息到达 Exchange 但无法根据路由键进入任何 Queue 时的回退。
  3. 消费者确认 (ACK): 消费者处理完业务后手动回复，防止消息丢失。
  4. 死信队列 (DLX): 处理失败、超时或被拒绝的消息会被转入此队列，用于延迟处理或人工审计。

4. RabbitMQ vs. Kafka (一分钟对比)

对比项 RabbitMQ Kafka
吞吐量 万级 / 秒 百万级 / 秒
持久性 内存优先，可选磁盘 基于磁盘顺序写
路由功能 极其强大 (Exchange) 简单 (按 Partition)
并发模型 异步、实时性高 批量、高吞吐
适用场景 传统业务系统、金融交易 日志收集、大数据处理

5. 总结

如果您需要极致的吞吐量和海量数据处理，选 Kafka。
如果您需要复杂的业务路由逻辑、严格的消息确认和更低的延迟，选 RabbitMQ。
