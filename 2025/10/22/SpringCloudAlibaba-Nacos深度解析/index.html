Nacos 深度解析：注册中心与配置中心的统一

Nacos (Naming and Configuration Service) 是阿里巴巴开源的一个致力于服务发现、配置管理和服务管理的平台。它不仅仅是一个注册中心，更是一个动态服务发现、配置和服务管理平台。

1. Nacos 核心架构

Nacos 的逻辑架构主要分为四层：

  1. 接入层 (Access Layer): 提供 HTTP 和 gRPC 接口，供客户端访问。
  2. 功能层 (Business Layer):
    • Naming Service: 服务注册、发现、健康检查。
    • Config Service: 配置的增删改查、版本管理、推送。
  3. 一致性协议层 (Consistency Protocol):
    • Distro: 阿里巴巴自研的 AP 协议，用于非持久化服务（临时实例）。
    • Raft: 标准的 CP 协议，用于持久化服务和配置数据。
  4. 存储层 (Storage Layer):
    • MySQL: 存储配置数据、用户信息等持久化数据。
    • Local File / Memory: 存储服务列表、缓存配置。

要真正理解 Nacos，必须看清一个注册请求是如何穿过 Spring Cloud 桥接层、Nacos SDK，最终进入服务端内存 Map 的。

2.2 关键源码步骤解析

  1. 触发阶段 (SC 适配层):
    • NacosAutoServiceRegistration: 它是 Spring Cloud 的标准化实现。它通过监听 WebServerInitializedEvent 事件，在 Web 容器启动后自动调用 registry.register()。
  2. 网络发送阶段 (SDK):
    • NamingClientProxyDelegate: Nacos 2.x 的 SDK 采用代理模式。它会维持一个到服务端的 gRPC 长连接。
    • 注册请求被封装为 InstanceRequest，通过 GrpcClientProxy 发送到服务端。
  3. 服务端接收与异步化 (Server):
    • InstanceRequestHandler: 负责处理所有实例相关的 gRPC 请求。
    • 异步解耦: Nacos 接收到请求后，并不会立即写内存，而是将请求包装成一个任务放入 BlockingQueue。这种设计让 Nacos 能够应对瞬时极高的注册峰值。
  4. 一致性协议处理 (Distro):
    • 对于临时实例，Nacos 走 Distro 协议。
    • 后台 Notifier 线程不断从队列拿任务，并负责将这些数据同步给集群中的其他 Nacos 节点。
  5. 最终写入 (CopyOnWrite):
    • 核心方法: Service.updateIPs(List<Instance> ips)。
    • 逻辑:
      1. 获取当前服务的细粒度锁。
      2. List<Instance> newIps = new ArrayList<>(oldIps) (复制副本)。
      3. 在 newIps 上执行更新。
      4. this.instances = newIps (引用替换)。
    • 效果: 这种 写时复制 机制实现了真正的“无锁读”，极大提升了服务发现的性能。

3. 配置管理 (Config Service) - 源码级深度剖析

Nacos 配置中心的核心挑战在于：如何让海量客户端实时感知配置变更，同时保持系统的高可用和低损耗。

3.1 客户端：分段轮询与 ClientWorker

客户端并不是简单地发一个请求，而是有一套复杂的管理机制：

  • ClientWorker: 客户端的核心总管。它启动了一个定时任务线程池，每隔 10ms 检查配置。
  • 分段检查 (3000条原则): 为了防止单个请求过大，Nacos 将监听的配置按每 3000 个 DataID 分为一组，每组启动一个独立的 LongPollingRunnable 线程。
  • 请求构造: 客户端将配置的 DataId+Group+MD5 拼接后，发送 HTTP POST 请求到服务端，设置 30s 的超时时间。

3.2 服务端：基于 AsyncContext 的“拖延”策略

服务端收到请求后，由 LongPollingService 负责处理：

  1. 立即对比: 检查 MD5。若不一致，立即返回变更。
  2. 异步挂起: 若一致，服务端并不立即返回，而是调用 request.startAsync() 获取异步上下文。
  3. 任务调度: 创建一个 ClientLongPolling 任务放入队列，并调度一个 29.5s 后执行的延时任务。
    • 29.5s 的用意: 留出 500ms 给网络传输，确保在客户端 30s 超时前响应。
  4. 双重唤醒机制:
    • 定时唤醒: 29.5s 到期，返回 304。
    • 事件唤醒: 期间配置若有变更，触发 LocalDataChangeEvent。服务端监听到事件后，立即从队列中定位到相关的 AsyncContext 并写回响应。

3.3 Nacos 2.x 的革命：从轮询到双向流 (gRPC)

在 2.x 版本中，配置推送逻辑发生了质变：

  • 长连接取代轮询: 客户端与服务端通过 gRPC 维持一条永久 TCP 链路。
  • 真正的 PUSH: 服务端不再“挂起 HTTP 请求”，而是通过 StreamObserver 直接向客户端推送 ConfigChangeNotifyRequest。
  • 效率提升: 彻底消除了 HTTP 建立连接和 30s 轮询带来的上下文切换开销，配置下发的延迟从秒级降至毫秒级。

4. 常见面试题 (Nacos 篇)

Q1: Nacos 注册中心为什么快？

  • 异步化: 注册请求先入队即返回，后台线程异步写内存。
  • 无锁读: 采用 CopyOnWrite 机制，服务发现操作完全不加锁。
  • gRPC: 2.x 使用长连接多路复用，减少了 TCP 握手。

Q2: Nacos 配置中心如何实现近乎实时的推送？

  • 1.x 靠长轮询: 服务端挂起请求 29.5s，配置变更时通过事件提前唤醒。
  • 2.x 靠 gRPC 流: 真正的双向通信，服务端检测到变更直接顺着长连接推给客户端。

Q3: 为什么 Nacos 1.x 长轮询是 29.5 秒而不是 30 秒？

  • 为了防止客户端 Socket 读超时。客户端默认超时 30s，服务端在 29.5s 返回，预留 500ms 缓冲，保证交互的稳定性。

5. 核心理论：CAP 定理与 Nacos 的 AP/CP 切换

理解 Nacos 的设计，必须首先理解分布式系统的基石 —— CAP 定理。

6.1 什么是 CAP？

在一个分布式系统中，以下三个特性不可兼得，最多只能满足其中两个：

  • C (Consistency - 一致性): 所有节点数据强一致。
  • A (Availability - 可用性): 系统始终能响应请求。
  • P (Partition Tolerance - 分区容错性): 网络故障时系统仍能运行。

在分布式网络中，P 是必然存在的。因此，系统的设计核心就是在 C（一致性） 和 A（可用性） 之间做权衡。

6.2 AP 模式 (可用性优先)

  • 理念: 宁可返回旧数据，也不允许服务中断。
  • 场景: 服务发现。如果注册中心为了同步数据而停止响应，整个微服务集群都会假死。拿到一个稍旧的 IP 列表通常不会导致系统崩溃。
  • 代表: Eureka、Nacos (临时实例)。

6.3 CP 模式 (一致性优先)

  • 理念: 宁可拒绝服务，也不允许数据不一致。
  • 场景: 配置管理、分布式锁、银行转账。如果配置信息在不同节点不一致，会导致极其严重的业务逻辑错误。
  • 代表: Zookeeper、Consul、Nacos (持久实例)。

6.4 Nacos 的动态切换

Nacos 是市面上极少数支持 AP 和 CP 模式切换的注册中心：

  1. 临时实例 (默认): 采用 AP 模式。心跳检测，数据存在内存，同步速度快，可用性极高。
  2. 持久实例: 采用 CP 模式。数据持久化到磁盘，采用 Raft 协议保证数据绝对一致。
  3. 切换方式: 客户端注册时通过 ephemeral=true/false 参数指定。

形象比喻:

  • AP 像看“朋友圈朋友圈”，晚几秒看到朋友发的照片没关系（可用性高），但一定要能刷得出来。
  • CP 像查“银行余额”，哪怕银行系统升级暂时查不到（不可用），也绝对不能查出昨天还没扣钱的错误余额（强一致性）。
