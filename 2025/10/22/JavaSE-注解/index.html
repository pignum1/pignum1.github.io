JavaSE - 注解 (Annotations)

注解（Annotations）是 Java 5 引入的一种元数据（metadata）机制。它提供了一种不影响程序逻辑的方式，为代码添加额外的信息。这些信息可以在编译时、类加载时或运行时被读取和处理，从而实现代码的自动化处理、检查或配置。

1. 注解的本质与作用

  • 本质: 注解是一种特殊的接口，继承自 java.lang.annotation.Annotation 接口。
  • 作用:
    • 编译检查: 告诉编译器如何检查代码（如 @Override, @Deprecated）。
    • 代码生成: 通过注解处理器在编译时生成新的代码（如 Lombok）。
    • 运行时处理: 在运行时通过反射读取注解信息，进行动态配置或行为修改（如 Spring 的 @Autowired, @Transactional）。

2. Java 内置注解

Java 提供了几个标准的内置注解，它们在日常开发中非常常见。

  • @Override:
    • 作用: 标记一个方法是重写父类或实现接口的方法。
    • 目的: 帮助编译器检查方法签名是否正确，如果父类中没有对应方法，则编译报错。
  • @Deprecated:
    • 作用: 标记一个类、方法或字段已过时。
    • 目的: 告诉开发者该元素不推荐使用，将来可能会被移除或有更好的替代方案。编译器会发出警告。
  • @SuppressWarnings:
    • 作用: 抑制编译器发出的特定警告。
    • 目的: 当开发者确认某个警告是无害的或已处理时，可以消除警告，使代码更整洁。
    • 参数: 可以是 "unchecked", "rawtypes", "deprecation", "all" 等。
  • @FunctionalInterface (Java 8+):
    • 作用: 标记一个接口是函数式接口（即只包含一个抽象方法的接口）。
    • 目的: 帮助编译器检查接口是否符合函数式接口的定义，并允许使用 Lambda 表达式。

3. 元注解 (Meta-Annotations)

元注解是用于注解其他注解的注解。它们定义了自定义注解的行为和特性。

  • @Target:
    • 作用: 声明注解可以应用于哪些程序元素。
    • 参数: ElementType 枚举，如 TYPE (类、接口、枚举), FIELD (字段), METHOD (方法), PARAMETER (参数), CONSTRUCTOR (构造器), LOCAL_VARIABLE (局部变量), ANNOTATION_TYPE (注解), PACKAGE (包), TYPE_PARAMETER (类型参数), TYPE_USE (类型使用)。
  • @Retention:
    • 作用: 声明注解的生命周期。
    • 参数: RetentionPolicy 枚举：
      • SOURCE: 注解只在源码中存在，编译时被丢弃。
      • CLASS: 注解在编译后的 .class 文件中存在，但运行时无法通过反射获取。
      • RUNTIME: 注解在运行时仍然存在，可以通过反射获取。这是最常用的类型，用于运行时处理。
  • @Documented:
    • 作用: 标记注解会被 Javadoc 工具包含在生成的文档中。
  • @Inherited:
    • 作用: 标记注解可以被子类继承。如果一个类被 @Inherited 注解修饰的注解标记，那么它的子类也会自动拥有这个注解。
  • @Repeatable (Java 8+):
    • 作用: 允许同一个注解在同一个程序元素上重复使用。

4. 自定义注解

我们可以根据需求创建自己的注解。

5. 注解的两大核心应用场景

注解的强大之处在于其处理时机不同，可以分为两大类：运行时处理和编译时处理。

场景一：运行时处理 (Runtime Processing)

这是最直观的用法，即我们之前的例子。

  • 生命周期: @Retention(RetentionPolicy.RUNTIME)
  • 工作方式: 程序在运行时，通过反射 API (isAnnotationPresent(), getAnnotation()) 来读取类、方法或字段上的注解信息，然后根据这些信息动态地改变程序的行为。
  • 优点: 灵活性极高，可以根据配置或环境动态调整行为。
  • 缺点: 依赖反射，有性能开销；错误只能在运行时发现。
  • 典型案例:
    • Spring 框架: @Autowired, @Service, @RestController 等，Spring 容器在启动时扫描这些注解，然后通过反射来创建 Bean、注入依赖、配置 Web 端点。
    • JUnit 4/5: @Test, @BeforeEach 等，JUnit 测试运行器在运行时通过反射找到并执行被标记的测试方法。
    • ORM 框架 (JPA/Hibernate): @Entity, @Table, @Column 等，框架在运行时读取这些注解，将对象和数据库表进行动态映射。

场景二：编译时处理 (Compile-time Processing)

这是注解更强大、更高效的用法，它在编译阶段就完成工作，避免了运行时的性能损耗。

  • 生命周期: @Retention(RetentionPolicy.SOURCE) 或 @Retention(RetentionPolicy.CLASS)
  • 工作方式: 程序员编写一个注解处理器 (Annotation Processor)。在 javac 编译代码的过程中，编译器会自动调用这个处理器。处理器会扫描源代码，查找它所关心的注解，然后可以生成全新的 Java 源代码文件。这些新生成的代码会和你的手写代码一起被编译成 .class 文件。
  • 优点:
    • 零运行时性能开销: 因为所有的工作（代码生成、检查）都在编译阶段完成了，运行时执行的是原生的高效代码，完全不涉及反射。
    • 错误提前暴露: 如果注解使用不当，可以在编译阶段就直接报错，而不是等到运行时才崩溃。
  • 缺点: 编程模型更复杂，需要编写专门的注解处理器。
  • 典型案例:
    • Lombok: 这是最著名的例子。你写的 @Getter, @Setter, @Data 等注解，在编译时被 Lombok 的注解处理器发现，然后它会自动为你生成对应的 getter/setter、toString() 等方法的源代码，并一起编译。你在源码里看不到这些方法，但在 .class 文件里它们是真实存在的。
    • MapStruct: 一个高性能的对象映射库。你只需定义一个带有 @Mapper 的接口，它就会在编译时自动生成该接口的实现类，完成对象之间属性的映射。性能远高于基于反射的 BeanUtils.copyProperties()。
    • Dagger 2 / Hilt: Google 开发的依赖注入框架。它们在编译时就处理 @Inject 等注解，生成所有依赖注入所需要的工厂类和组件代码，实现了零反射的依赖注入，在 Android 等对启动性能要求高的场景中非常受欢迎。

总结对比

对比维度 运行时处理 (反射) 编译时处理 (代码生成)
核心思想 运行时动态改变行为 编译时静态生成代码
性能 有反射开销，较慢 无运行时开销，高效
错误发现 运行时 (e.g., NullPointerException) 编译时 (e.g., 编译错误)
典型框架 Spring, JUnit, Hibernate Lombok, MapStruct, Dagger

因此，除了通过反射，注解的另一个强大之处在于它能驱动编译器在编译阶段就为我们完成大量重复、模板化的工作，既保证了源码的简洁，又获得了极高的运行效率。
