MySQL 核心原理与调优

本篇笔记聚焦于 MySQL 的底层物理存储、索引机制以及事务并发控制。

1. MySQL 逻辑架构

MySQL 采用了 Server层 与 存储引擎层 插件化分离的架构。

  • Server 层: 包括连接器、分析器、优化器（核心：估算成本、选择索引）、执行器。
  • 存储引擎层: 负责数据存储。InnoDB 是目前的主流，支持事务、行锁、外键。

2. 索引底层原理：B+ 树

2.1 物理存储结构

  • 页 (Page): InnoDB 磁盘管理的最小单位，默认 16KB。
  • B+ 树优势: 相比 B 树，B+ 树只在叶子节点存数据，非叶子节点存索引，使得树更矮（I/O次数少）；叶子节点双向链表，支持范围查询。

2.2 索引类型与优化

  • 聚簇索引: 叶子节点存整行数据（主键索引）。
  • 非聚簇索引: 叶子节点存主键值（二级索引）。回表是指先查二级索引拿到主键，再查聚簇索引拿到整行数据。
  • 联合索引: 遵循最左匹配原则。遇到范围查询匹配即停止。
  • 优化策略: 覆盖索引（消除回表）、索引下推 (ICP)（在引擎层过滤数据，减少回表次数）。

3. InnoDB 核心内存机制：Buffer Pool

Buffer Pool 实现了**“内存读写，异步刷盘”**，是性能的压舱石。

  1. WAL (预写日志): 修改内存脏页的同时顺序写 Redo Log，保证崩溃恢复。
  2. 改进型 LRU: 将链表分为 Young (热) 和 Old (冷) 区，防止全表扫描污染缓存。
  3. Change Buffer: 针对非唯一二级索引的写优化，减少随机 I/O。

4. 深入持久化：Redo Log 与两阶段提交

  1. Redo Log (物理日志): 记录页的物理修改，循环写入。
  2. 两阶段提交 (2PC):
    • Step 1 (Prepare): 写入 Redo Log，状态设为 prepare。
    • Step 2 (Binlog): Server 层写入 Binlog。
    • Step 3 (Commit): 引擎层将 Redo Log 状态改为 commit。
    • 意义: 保证 Redo Log 和 Binlog 数据一致，防止主从同步失效。

4.4 逻辑基石：Undo Log 机制

如果说 Redo Log 保证了“不丢”，那么 Undo Log 则保证了“能回去”。

  1. 定义与性质:
    • 逻辑日志: 记录的是与当前操作相反的逻辑（如：执行了 Insert，日志记录对应的 Delete）。
  2. 核心作用:
    • 实现回滚 (原子性): 当事务执行失败或执行 ROLLBACK 时，系统根据 Undo Log 将数据恢复到事务开始前的状态。
    • 支撑 MVCC (隔离性): 这是 Undo Log 的进阶用法。通过 roll_pointer 指针，Undo Log 将一行数据的历史版本串联成“版本链”。当快照读发现当前版本不可见时，会沿着链条去 Undo Log 中寻找符合条件的旧版本。
  3. 生命周期:
    • 事务提交后，Undo Log 并不会立即删除。因为可能还有其他长事务需要通过版本链读取该快照。
    • Purge 线程: 专门负责轮询，当确认没有任何事务再需要这些日志时，才执行物理删除。

Redo Log vs. Undo Log 对比表:

特性 Redo Log (重做日志) Undo Log (回滚日志)
保障特性 D (持久性) A (原子性)、I (隔离性)
记录内容 物理页的修改 (页号, 偏移量, 值) 逻辑上的相反操作 (原始旧值)
记录时机 脏页写回磁盘前 数据修改发生前
核心意义 掉电后恢复内存中未刷盘的修改 事务失败后撤销已执行的逻辑修改

5. 事务隔离机制与 MVCC 深度剖析

5.1 隔离级别与并发问题

在多线程并发访问时，如果没有隔离机制，会产生三种“病症”：

  1. 脏读 (Dirty Read): 读到了别人未提交的数据。
  2. 不可重复读 (Non-repeatable Read): 读到了别人已提交的修改，导致同一事务内两次读取结果不同。
  3. 幻读 (Phantom Read): 读到了别人已提交的新增/删除，导致同一事务内两次查询的行数不同。

隔离级别对照表:

隔离级别 脏读 不可重复读 幻读 形象比喻
读未提交 (RU) 允许 允许 允许 完全透明: 毫无遮拦。
读已提交 (RC) 禁止 允许 允许 阅后即焚: 提交了才能看。
可重复读 (RR) 禁止 禁止 基本禁止 静态快照: 事务内世界静止。
串行化 禁止 禁止 禁止 单人单间: 所有人必须排队。

5.2 MVCC (多版本并发控制) 底层原理

MVCC 实现了“读不加锁，读写不冲突”，极大提升了并发性能。

  1. 隐藏列: 每行记录都有 trx_id（最后修改事务 ID）和 roll_pointer（指向 Undo Log 的版本链指针）。
  2. ReadView (一致性视图):
    • m_ids: 活跃未提交事务列表。
    • min_trx_id: 活跃事务最小值。
    • max_trx_id: 系统即将分配的下一个 ID。
  3. 可见性算法 (RR 级别):
    • trx_id < min_trx_id: 事务已提交，可见。
    • trx_id >= max_trx_id: 事务在视图后开启，不可见。
    • min_trx_id <= trx_id < max_trx_id: 若不在 m_ids 中，说明已提交，可见。
  4. RC vs RR: RC 每次 SELECT 生成新视图；RR 仅第一次 SELECT 生成并复用。

6. 锁机制与并发控制深度剖析

如果说 MVCC 是“文斗”（非阻塞读），那么锁机制就是“武斗”（强行排队）。锁是数据库处理并发冲突的最后防线。

6.1 锁的粒度与分类

  • 全局锁: 锁定整个数据库，主要用于逻辑备份。
  • 表级锁:
    • 表锁: 锁定整张表。
    • 元数据锁 (MDL): 保护表结构不被修改。
    • 意向锁 (IS/IX): 为了更高效地加表锁。当事务给行加锁前，会先在表级别加意向锁。这样当另一个事务想锁表时，只需看表上的意向锁，而无需遍历全表查找是否有行锁。
  • 行级锁 (InnoDB 核心):
    1. 记录锁 (Record Lock): 仅锁定单行记录。
    2. 间隙锁 (Gap Lock): 锁定一个区间（不含记录本身），如 (5, 10)。唯一目的：防止幻插。
    3. Next-Key Lock (临键锁): 记录锁+间隙锁，锁定左开右闭区间，如 (5, 10]。这是 InnoDB 默认的行锁算法。

6.2 幻读的终极解决方案：快照读与当前读的“职能分工”

InnoDB 在 REPEATABLE READ (RR) 级别下，将读取操作“裂解”为两种模式，以平衡性能与一致性。

维度 快照读 (Snapshot Read) 当前读 (Current Read)
底层武器 MVCC (ReadView + Undo Log) Next-Key Lock (行锁 + 间隙锁)
读取版本 历史版本（快照） 最新版本
加锁情况 不加锁，读写不冲突 必须加锁，保证绝对安全
适用场景 普通 SELECT FOR UPDATE / UPDATE / DELETE
核心目的 追求极致的并发吞吐 保证业务逻辑的严密性

深度解析：为什么需要这种“裂解”？

  1. 快照读 (追求效率): 大多数查询只需看到“已提交的数据”。通过 MVCC 读取历史版本，避免了加锁开销，实现了高并发。
  2. 当前读 (追求正确性): 当我们要修改数据（或显式加锁读）时，必须看到数据的真实最新状态。此时通过 Next-Key Lock 不仅锁住现有记录，还封锁了它们之间的“间隙”，确保在操作期间不会有“幻影”数据插入，从而彻底终结幻读风险。

6.3 死锁检测与处理

当两个事务循环等待对方的资源时，产生死锁。

  • 超时回滚: innodb_lock_wait_timeout (默认 50s)。
  • 主动检测: 开启 innodb_deadlock_detect (默认开启)，系统会自动发现死锁环并强制回滚一个小事务，让大事务继续执行。
