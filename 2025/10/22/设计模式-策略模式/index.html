什么是策略模式？

策略模式 (Strategy Pattern) 是一种行为型设计模式，它定义了一系列算法，将每个算法都封装起来，并使它们可以相互替换。策略模式使得算法可以独立于使用它的客户端而变化。

通俗地讲，策略模式就像是解决“条条大路通罗马”的问题。你去罗马（目标），可以选择多种交通方式（策略），比如可以坐飞机、坐火车、或者骑自行车。每种交通方式都是一个独立的策略。你可以根据你的具体情况（如时间、预算）在运行时自由选择和切换策略，而你“去罗马”这个行为本身（客户端）并不需要关心每种交通方式的具体实现细节。

模式结构

策略模式主要包含以下角色：

  1. Context (上下文): 维护一个对 Strategy 对象的引用。它不执行具体策略，而是将工作委派给所引用的策略对象。它提供一个方法让客户端设置策略。
  2. Strategy (策略接口): 定义了所有支持的算法的公共接口。上下文使用这个接口来调用由具体策略定义的算法。
  3. ConcreteStrategy (具体策略): 实现了策略接口，封装了具体的算法或行为。

优缺点

优点

  1. 遵循开闭原则: 你可以在不修改现有上下文代码的情况下，轻松地增加新的策略。
  2. 避免 if-else 地狱: 将不同行为封装在各自的策略类中，避免了在上下文中使用大量的条件判断语句，使得代码更清晰、更易于维护。
  3. 算法可以自由切换: 客户端可以在运行时根据需要动态地改变对象的行为（算法）。
  4. 代码复用: 多个上下文可以共享同一个策略对象。

缺点

  1. 增加了类的数量: 每个策略都需要一个单独的类，这会增加系统中的类的总数。
  2. 客户端必须了解所有策略: 客户端需要知道有哪些可用的策略，并自行决定使用哪一个。这在一定程度上增加了客户端的职责。

经典应用

  1. java.util.Collections.sort(): sort 方法接受一个 Comparator 对象作为参数。这里的 Comparator 就是一个策略接口，它定义了比较两个对象的策略。你可以传入不同的 Comparator 实现（如 String.CASE_INSENSITIVE_ORDER 或自定义的比较器），来改变 sort 方法的排序行为。
  2. Spring Resource 接口: Spring 框架使用 Resource 接口来统一访问不同来源的资源（如文件系统、classpath、URL）。ClassPathResource, FileSystemResource, UrlResource 等都是 Resource 接口的具体策略实现。
  3. 线程池的拒绝策略: Java 的 ThreadPoolExecutor 允许你设置不同的拒绝策略（RejectedExecutionHandler），如 AbortPolicy (抛出异常)、CallerRunsPolicy (调用者自己执行)、DiscardPolicy (直接丢弃) 等。这也是策略模式的典型应用。
