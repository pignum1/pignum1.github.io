什么是责任链模式？

责任链模式 (Chain of Responsibility Pattern) 是一种行为型设计模式，它允许你将请求沿着处理者链进行发送。收到请求后，每个处理者均可对请求进行处理，或将其传递给链中的下一个处理者。这种模式通过将请求的发送者和接收者解耦，使得多个对象都有机会处理这个请求。

通俗地讲，责任链模式就像公司的报销审批流程。你提交一个报销单（请求），首先由你的直属经理（处理者A）审批。如果金额在经理的权限范围内，他就直接批准了；如果超出他的权限，他不会拒绝，而是将报销单往上传递给部门总监（处理者B）。总监再根据自己的权限进行处理或继续向上传递。你作为申请人，无需关心这个请求最终由谁处理，你只需要把它提交到这个流程（责任链）的起点即可。

模式结构

责任链模式主要包含以下角色：

  1. Handler (处理者): 定义了一个处理请求的接口。它通常持有一个指向链中下一个处理者的引用。
  2. ConcreteHandler (具体处理者): 实现了处理者接口。它负责处理它感兴趣的请求，如果它不能处理，则将请求转发给它的后继者。
  3. Client (客户端): 创建处理者链，并将请求发送到链的第一个处理者。

代码示例 (实战级重构)

之前的基础示例虽然解释了模式结构，但在链的组装和逻辑传递上显得有些僵硬。一个更优雅、更贴近实战的实现（类似 Spring 拦截器链）通常会有一个中心化的“链”对象来驱动流程。

1. 定义处理者接口 (Handler)

处理者接口变得更简洁，它只关心处理请求，并通过返回值告知“链”是否应该继续。

2. 创建具体处理者 (ConcreteHandler)

每个处理者实现单一职责，处理完后通过返回值决定流程是否继续。

3. 创建责任链 (HandlerChain)

这是本次重构的核心。HandlerChain 负责管理和执行链，使得处理者之间完全解耦。

4. 客户端使用

客户端现在可以非常优雅地构建和执行责任链，处理者顺序一目了然。

优缺点

优点

  1. 降低耦合度: 请求的发送者和接收者之间完全解耦。发送者无需知道请求被哪个接收者处理，也无需知道链的结构。
  2. 增强了系统的可扩展性和灵活性: 可以随时增加或修改处理者，也可以在运行时动态地改变链的结构，符合开闭原则。
  3. 单一职责原则: 每个处理者都只关心自己的职责范围，代码更清晰、更易于维护。

缺点

  1. 不能保证请求一定被处理: 如果链的末尾没有默认的处理者，请求可能会传递到链的末端而未被处理。
  2. 调试不便: 由于请求在链中传递，如果链条很长，可能会增加调试和追踪的难度。
  3. 性能问题: 在最坏的情况下，请求需要遍历整个链条才能被处理，可能会对性能产生一定影响。

经典应用

  1. Java Web 中的过滤器链 (Filter Chain): 这是责任链模式最经典的应用。在 Servlet 规范中，一个请求会经过一个由多个 Filter 组成的链。每个 Filter 都可以对请求进行处理（如权限检查、日志记录、编码转换），然后决定是中断请求还是将其传递给链中的下一个 Filter 或最终的 Servlet。
  2. Spring Security 过滤器链: Spring Security 使用一个过滤器链来保护 Web 请求。每个过滤器负责一种特定的安全职责，如认证 (UsernamePasswordAuthenticationFilter)、授权 (FilterSecurityInterceptor)、CSRF 防护 (CsrfFilter) 等。
  3. MyBatis 插件机制: MyBatis 的插件 (Interceptor) 机制也是基于责任链模式。你可以创建自定义插件来拦截 Executor、StatementHandler 等核心组件的方法调用，从而在 SQL 执行的不同阶段（如参数设置、结果集处理）插入自定义逻辑。
  4. 日志框架的 Appender: 在 Log4j 或 Logback 中，一个 Logger 可以关联多个 Appender。当记录日志时，日志事件会依次经过这些 Appender，形成一个责任链。
