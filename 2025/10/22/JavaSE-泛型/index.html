JavaSE - 泛型 (Generics)

泛型（Generics）是 Java 5 引入的一个重要特性，它允许在定义类、接口和方法时使用类型参数。泛型的核心目标是提高代码的类型安全性和消除强制类型转换，同时增强代码的可重用性。

1. 为什么需要泛型？

在泛型出现之前，Java 集合框架（如 ArrayList）只能存储 Object 类型的对象。这意味着：

  1. 类型不安全: 任何类型的对象都可以被添加到集合中，导致运行时可能出现 ClassCastException。
  2. 需要强制类型转换: 从集合中取出元素时，必须进行强制类型转换，代码冗余且容易出错。

泛型解决了这些问题，它在编译时就提供了类型检查，确保了类型安全。

2. 泛型的基本使用

2.1 泛型类

在类名后面添加 <T>（或其他大写字母，如 E 表示元素，K 表示键，V 表示值），T 就是类型参数。

2.2 泛型接口

接口也可以使用泛型。

2.3 泛型方法

在方法返回类型前面添加 <T>，T 就是类型参数。泛型方法可以定义在普通类中，也可以定义在泛型类中。

3. 类型擦除 (Type Erasure)

这是 Java 泛型最核心也是最容易混淆的概念。

  • 定义: Java 泛型是在编译时实现的。在编译完成后，所有泛型信息（即类型参数 <T>）都会被擦除，替换为它们的上界（通常是 Object）。
  • 目的: 为了兼容 Java 5 之前的旧版本代码。
  • 影响:
    1. 运行时无法获取泛型类型信息: 例如，List<String> 和 List<Integer> 在运行时实际上都是 List 类型。
    2. 不能使用 instanceof 判断泛型类型: if (obj instanceof List<String>) 是编译错误的。
    3. 不能创建泛型数组: T[] array = new T[10]; 是编译错误的，因为运行时 T 不存在。
    4. 不能直接捕获泛型异常: catch (MyGenericException<T> e) 是编译错误的。

4. 泛型通配符 (Wildcards) 与 PECS 原则

泛型通配符用于解决泛型类型之间不能直接赋值的问题。例如，List<String> 并不是 List<Object> 的子类型。为了理解通配符，我们必须先理解著名的 PECS 原则。

PECS: Producer Extends, Consumer Super

这个原则的命名非常容易引起混淆，关键在于要从泛型集合本身的角度来理解“生产者”和“消费者”。

  • Producer (生产者): 指的是一个向你的代码提供 T 类型实例的泛型集合。你的代码是从这个集合中读取 (get) 数据。因此，它是一个数据的来源或生产者。

    • 结论: 对于这样的生产者，我们使用 extends。

  • Consumer (消费者): 指的是一个需要接收 T 类型实例的泛型集合。你的代码是向这个集合中写入 (add) 数据。因此，它是一个数据的目的地或消费者。

    • 结论: 对于这样的消费者，我们使用 super。

我们用一个“水果盘”的例子来解释。假设有以下类：

? extends Fruit (上界通配符 - 用于读取)

List<? extends Fruit> 表示这是一个“装着某种水果的盘子”。它可能是 List<Fruit>，也可能是 List<Apple>。

  • 读取操作 (安全): 我们可以从这个盘子里取出水果。因为无论盘子里装的是 Apple 还是 Banana，我们都可以肯定地说“它是一种水果 (Fruit)”。

    • Fruit f = fruitPlate.get(0); 是安全的。

  • 写入操作 (危险): 我们不能往这个盘子里放入任何具体类型的水果。为什么？

    • 因为我们不确定盘子的具体类型。如果这个盘子实际上是 List<Apple>（一个苹果盘），你试图往里面放入一个 Banana，这显然是错误的。
    • 编译器为了保证类型安全，干脆禁止了所有写入操作（除了 null）。

总结 extends:

  • 功能: 主要用于安全地读取数据 (get)。
  • 限制: 不能写入数据 (add)。
  • 记忆: extends 用于生产者 (Producer)，即提供数据供你读取的集合。

? super Apple (下界通配符 - 用于写入)

List<? super Apple> 表示这是一个“能装苹果以及苹果父类的盘子”。它可能是 List<Apple>，也可能是 List<Fruit>，或者是 List<Object>。

  • 写入操作 (安全): 我们可以往这个盘子里放入一个 Apple。为什么？

    • 因为无论这个盘子是 List<Apple>（苹果盘）、List<Fruit>（水果盘）还是 List<Object>（什么都能装的盘），它都有能力装下一个 Apple。我们甚至可以放入 RedApple，因为 RedApple 也是一种 Apple。
    • applePlate.add(new Apple()); 和 applePlate.add(new RedApple()); 都是安全的。

  • 读取操作 (信息丢失): 我们不能确定从这个盘子里取出的东西是什么类型，除了 Object。

    • 因为这个盘子可能是 List<Object>，你从里面取出的东西，只能保证它是一个 Object，但不能保证它是一个 Fruit，更不能保证它是一个 Apple。
    • Object o = applePlate.get(0); 是唯一安全的操作。

总结 super:

  • 功能: 主要用于安全地写入数据 (add)。
  • 限制: 读取数据时类型信息丢失（只能得到 Object）。
  • 记忆: super 用于消费者 (Consumer)，即接收你写入的数据的集合。

? (无界通配符)

List<?> 实际上是 List<? extends Object> 的简写，其行为与 extends 一致：只能读（作为 Object），不能写。

实际应用：Collections.copy()

Java 源码中的 Collections.copy() 方法是理解 PECS 原则的最佳范例：

  • src (源列表): 它是生产者，我们只需要从里面读取 T 类型的元素，所以使用 ? extends T。
  • dest (目标列表): 它是消费者，我们需要往里面写入 T 类型的元素，所以使用 ? super T。

这个方法签名完美地诠释了 PECS 原则，使得 copy 方法具有最大的灵活性。例如，你可以将一个 ArrayList<Integer> 复制到一个 ArrayList<Number> 中。

5. 泛型方法与类型推断

Java 编译器在调用泛型方法时，会尝试根据方法参数的类型来推断类型参数 T 的具体类型。这种能力称为类型推断 (Type Inference)。

6. 泛型与继承

  • 泛型类不具备继承性: Box<Integer> 并不是 Box<Number> 的子类。即使 Integer 是 Number 的子类，Box<Integer> 和 Box<Number> 之间也没有继承关系。
  • 通配符解决泛型继承问题: 这正是泛型通配符 ? extends T 和 ? super T 存在的意义，它们允许我们表达泛型类型之间的“兼容性”或“子类型关系”。

7. 总结

泛型是 Java 语言中一个非常强大且复杂的特性。它通过在编译时进行类型检查，提高了代码的安全性、可读性和可重用性。理解类型擦除和泛型通配符是掌握泛型的关键。
