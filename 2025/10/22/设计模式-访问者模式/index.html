什么是访问者模式？

访问者模式 (Visitor Pattern) 是一种行为型设计模式，它的核心思想是将作用于数据结构中各种元素的操作分离出来，封装成独立的“访问者”对象。这使得你可以在不改变数据结构（元素类）的前提下，为这些元素定义新的操作。

通俗地讲，访问者模式就像一个保险推销员（访问者）去一个公司（对象结构）推销保险。公司里有工程师、经理、HR 等不同类型的员工（元素）。

  • 对于工程师，推销员会说：“你们经常加班，猝死风险高，需要买意外险。”
  • 对于经理，他会说：“您是公司中坚，身系家庭幸福，需要买高额寿险。”
  • 对于 HR，他会说：“贵公司可以为员工统一购买团体险，提升福利。”

在这个过程中，员工（元素）的类没有改变，但保险推销员（访问者）为不同类型的员工提供了完全不同的操作（推销不同的话术）。如果公司需要法律顾问服务，只需派一个律师（另一个访问者）来，他会用另一套话术（操作）来和这些员工打交道。

模式结构

访问者模式主要包含以下角色：

  1. Visitor (访问者接口): 定义了对每个 Element (元素) 的访问操作，其方法名通常与被访问的元素类名相关，例如 visitConcreteElementA()。
  2. ConcreteVisitor (具体访问者): 实现了访问者接口，封装了对数据结构中具体元素的一组相关操作。
  3. Element (元素接口): 定义了一个 accept() 方法，它以一个访问者作为参数。
  4. ConcreteElement (具体元素): 实现了元素接口。它的 accept() 方法通常只有一行代码：visitor.visit(this);，将自身传递给访问者。
  5. ObjectStructure (对象结构): 这是一个元素的容器，通常包含一个 accept() 方法，用于遍历所有元素并让它们接受访问者的访问。

优缺点

优点

  1. 符合开闭原则: 你可以在不修改现有元素类的情况下，增加新的操作（即增加新的访问者）。
  2. 集中相关操作: 将对同一组元素的相关操作集中到一个访问者类中，而不是分散在各个元素类里。
  3. 易于扩展: 可以轻松地增加新的元素类，只需在访问者接口和所有具体访问者中增加相应的 visit 方法。

缺点

  1. 破坏封装: 访问者模式要求元素类提供一个 accept 方法来暴露自身，这在一定程度上破坏了元素的封装性。
  2. 难以增加新的元素: 这与优点3看起来矛盾，但实际上是该模式最大的缺点。每当增加一个新的具体元素类时，你都需要修改所有访问者接口和所有具体访问者类，在其中加入对新元素的 visit 方法。这违反了开闭原则。

适用场景

访问者模式适用于以下情况：

  1. 数据结构稳定，但操作易变: 当你有一个复杂的数据结构（如组合模式构成的对象树），并且需要对它进行多种不同的、不相关的操作时。使用访问者模式，可以将这些操作独立出来。
  2. 需要对一个对象结构中的对象进行很多不同的并且不相关的操作，而你想避免让这些操作“污染”这些对象的类。

总结

访问者模式是一个“双刃剑”。它在增加新操作时非常方便，但在增加新元素时却非常痛苦。因此，只有在元素结构非常稳定，但需要频繁添加新操作的场景下，才适合使用访问者模式。

经典应用

  1. 编译器: 编译器在构建了抽象语法树 (AST) 后，需要对树进行多种操作，如类型检查、语法分析、代码生成等。每一种操作都可以实现为一个访问者，遍历 AST 并执行相应的操作。
  2. javax.lang.model: Java 的注解处理 API 中就使用了访问者模式。Element 接口（代表程序元素，如类、方法、字段）有一个 accept(ElementVisitor<R, P> v, P p) 方法，你可以实现自己的 ElementVisitor 来处理不同类型的程序元素。
