Spring Bean 生命周期全过程 (源码深度剖析)

本文档旨在从源码层面，按时间线性详细阐述 Spring Bean 从容器启动到最终可用的完整生命周期。

Phase 1: 容器启动与准备 (Bootstrap)

在 Bean 真正被创建之前，Spring 容器 (ApplicationContext) 需要进行一系列的环境准备和配置加载。入口方法为 AbstractApplicationContext.refresh()。

1.1 环境准备 (prepareRefresh)

  • 核心类: AbstractApplicationContext
  • 动作:
    1. 设置启动时间、活动状态 (active=true)。
    2. 初始化属性源 (initPropertySources)：加载 Web 环境配置。
    3. 校验必选属性 (validateRequiredProperties)：检查关键环境变量，失败则直接抛出异常。
    4. 初始化早期监听器 (earlyApplicationListeners)：备份监听器列表，确保幂等性。

1.2 工厂创建与图纸加载 (obtainFreshBeanFactory)

  • 核心类: AbstractRefreshableApplicationContext
  • 动作: refreshBeanFactory()
    1. 销毁旧工厂: 确保容器的纯净性。
    2. 创建新工厂: 实例化 DefaultListableBeanFactory。这是 Spring 的核心 Bean 工厂。
    3. 加载 BeanDefinition: 调用 loadBeanDefinitions（模板方法）。
      • XML 模式: XmlBeanDefinitionReader 解析 XML。
      • 注解 模式: AnnotatedBeanDefinitionReader / ClassPathBeanDefinitionScanner 扫描 @Component。
  • 产物: 此时内存中只有 BeanDefinition (Bean 的元数据图纸)，Bean 实例尚未创建。

1.3 厂房装修 (prepareBeanFactory)

  • 核心类: AbstractApplicationContext
  • 动作: 配置工厂的基础设施。
    1. 设置类加载器 (ClassLoader) 和表达式解析器 (SPEL)。
    2. 添加 ApplicationContextAwareProcessor：用于处理 Aware 回调。
    3. 设置忽略接口 (ignoreDependencyInterface)：如 ApplicationContextAware，禁止自动装配这些接口的 Setter。
    4. 注册可解析依赖 (registerResolvableDependency)：允许直接注入 BeanFactory, ApplicationContext 等容器对象。

1.4 配置增强 (invokeBeanFactoryPostProcessors)

  • 核心类: PostProcessorRegistrationDelegate
  • 动作: 执行 BeanFactoryPostProcessor (BFPP)。
    • ConfigurationClassPostProcessor: 解析 @Configuration, @Bean, @ComponentScan, @Import。这是 Spring Boot 能够自动配置的核心。
    • PropertySourcesPlaceholderConfigurer: 处理配置文件中的占位符 ${...}。
  • 意义: 对 BeanDefinition 进行修改和补充。

1.5 注册后置处理器 (registerBeanPostProcessors)

  • 核心类: PostProcessorRegistrationDelegate
  • 动作: 实例化并注册 BeanPostProcessor (BPP)。
    • 严格排序: 按 PriorityOrdered -> Ordered -> Non-Ordered 的顺序注册。
    • 意义: 确保 AOP 自动代理创建器 (AnnotationAwareAspectJAutoProxyCreator) 等核心基础设施在普通 Bean 创建前就位。

Phase 2: 实例化 (Instantiation)

容器启动的最后一步 finishBeanFactoryInitialization 会调用 preInstantiateSingletons，触发所有非懒加载单例 Bean 的创建。

2.0 批量生产的起点 (preInstantiateSingletons)

这是 DefaultListableBeanFactory 中的核心方法，它负责按下“启动按钮”。

  1. 防御性复制: List<String> beanNames = new ArrayList<>(this.beanDefinitionNames);
    • 原因: 防止在 Bean 创建过程中，因某些 Bean 的初始化逻辑触发了新 Bean 的注册，导致遍历时抛出 ConcurrentModificationException。
  2. 严苛筛选: 遍历列表，只有同时满足 非抽象、单例、非懒加载 的 Bean 才有资格在此刻创建。
  3. 触发生产:
    • 普通 Bean: 直接调用 getBean(beanName)。
    • FactoryBean: 先获取工厂本身 (&name)。只有当它实现了 SmartFactoryBean 且 isEagerInit() 为真时，才立即调用 getObject() 创建目标对象。
  4. 谢幕演出: 当所有 Bean 创建完成后，再次遍历所有 Bean。
    • 如果 Bean 实现了 SmartInitializingSingleton 接口，调用其 afterSingletonsInstantiated()。
    • 意义: 这是 Spring MVC 等框架在所有 Bean 就位后进行最终配置（如扫描 Controller 注册 URL）的关键时机。

2.0.1 核心入口 (doGetBean)

getBean 最终会调用 doGetBean，这是一个巨大的分发器，处理各种边界情况。

  1. 名称转换: 去除 & 前缀（如果是 FactoryBean），并将别名 (Alias) 解析为规范名称。
  2. 缓存查找: 尝试从一级/二级/三级缓存中获取。如果命中，直接进入第 6 步。
  3. 父工厂委托: 如果当前容器没有该 Bean 定义，且存在父容器，则委托给父容器查找（Spring MVC 父子容器特性的基础）。
  4. Depends-On: 如果配置了 @DependsOn，会先递归调用 getBean 强制创建被依赖的 Bean。
  5. 分类创建:
    • Singleton: 调用 getSingleton(..., factory)，在 Lambda 中执行 createBean，并放入一级缓存。
    • Prototype: 直接调用 createBean，不存缓存。
    • Other Scope: 从 Request/Session 等 Scope 中获取。
  6. FactoryBean 处理: 如果拿到的对象是 FactoryBean 但用户需要的是普通 Bean（无 &），则调用 getObject() 创建目标对象。

核心流程进入 AbstractAutowireCapableBeanFactory.doCreateBean。

2.1 实例化 (createBeanInstance)

  • 核心类: SimpleInstantiationStrategy / CglibSubclassingInstantiationStrategy
  • 动作:
    1. 推断构造函数: ConstructorResolver 根据参数类型和 @Autowired 注解选择合适的构造函数。
    2. 执行实例化: 通过反射 (Constructor.newInstance) 或 CGLIB 创建 Bean 的原始对象。
  • 状态: 此时对象已在堆内存中分配，但所有属性均为 null (或默认值)。

2.2 解决循环依赖 (addSingletonFactory)

  • 核心类: DefaultSingletonBeanRegistry
  • 动作: 将当前 Bean 包装为一个 ObjectFactory，放入三级缓存 (singletonFactories)。
  • 逻辑: 如果后续有其他 Bean 依赖当前 Bean，可以通过这个工厂获取到当前 Bean 的早期引用 (Early Reference)，从而打破循环等待。

2.3 循环依赖解决全流程推演 (缓存流动)

假设 A 依赖 B，B 依赖 A。看数据如何在三个缓存间流转：

  1. A 出生: createBean(A) -> 实例化 A -> A 工厂放入三级缓存。
    • 缓存状态: [1级: 空] [2级: 空] [3级: A工厂]
  2. A 需注入 B: populateBean(A) 发现依赖 B -> 调用 getBean(B)。
  3. B 出生: createBean(B) -> 实例化 B -> B 工厂放入三级缓存。
    • 缓存状态: [1级: 空] [2级: 空] [3级: A工厂, B工厂]
  4. B 需注入 A: populateBean(B) 发现依赖 A -> 调用 getBean(A)。
    • 1级无，2级无，3级找到 A工厂。
  5. A 晋升 (关键): 调用 A工厂的 getObject()。
    • 生成 A 的早期引用（若 A 需要 AOP，此时提前创建代理）。
    • A 移入二级缓存，移出三级缓存。
    • 缓存状态: [1级: 空] [2级: A早期] [3级: B工厂]
  6. B 完成: B 注入 A 成功 -> 初始化 B -> B 移入一级缓存。
    • 缓存状态: [1级: B成品] [2级: A早期] [3级: 空]
  7. A 完成: A 注入 B 成功 (拿的是1级的 B) -> 初始化 A -> A 移入一级缓存 (从2级取出)。
    • 缓存状态: [1级: A成品, B成品] [2级: 空] [3级: 空]

2.4 深度思考：为什么非要三级缓存？

Q: 如果只用两级缓存（成品 + 半成品），能解决循环依赖吗？
A: 能，但无法支持 AOP。
如果只用两级，A 实例化后直接将原始对象放入二级缓存。B 注入 A 时拿到的是原始对象。但如果 A 配置了切面（AOP），B 应该注入 A 的代理对象。如果注入了原始对象，AOP 就失效了。

Q: 那为什么不在实例化后直接创建代理，放入二级缓存？
A: 破坏了 Bean 的生命周期。
Spring 的设计原则是：AOP 代理应该在 Bean 初始化完成之后生成。如果一实例化就创建代理，那是对正常流程的破坏。

Q: 三级缓存的精妙之处

  • 三级缓存 (Factory): 这是一个延迟决策。
    • 正常情况: 工厂永远不被调用。AOP 代理在生命周期最后一步（初始化后）生成。
    • 循环依赖: B 需要 A，调用工厂。工厂内部逻辑 (getEarlyBeanReference) 会判断：如果 A 需要 AOP，就提前创建代理返回。
  • 二级缓存: 用于确保单例。
    • 假如 A 被 B 依赖，又被 C 依赖。B 调用工厂拿到了 A 的代理。如果没有二级缓存，C 再次调用工厂，会生成另一个 A 的代理。这是不对的。
    • 有了二级缓存，B 拿到 A 代理后放入二级。C 再来拿，直接从二级拿同一个 A 代理。

Phase 3: 属性填充 (Population)

3.1 属性注入 (populateBean)

  • 核心类: AutowiredAnnotationBeanPostProcessor
  • 动作:
    1. InstantiationAwareBeanPostProcessor: 在属性设置前给 BPP 一个修改 Bean 定义的机会。
    2. 解析依赖: 处理 @Autowired, @Value, @Resource 注解。
    3. 递归注入: 调用 getBean() 获取依赖的 Bean。如果依赖的 Bean 尚未创建，则触发其创建流程。

Phase 4: 初始化 (Initialization)

Bean 的属性填充完毕后，需要执行各种初始化逻辑，使其成为一个完全可用的对象。

4.1 Aware 接口回调 (invokeAwareMethods)

  • 动作: 如果 Bean 实现了以下接口，容器会调用对应方法注入资源：
    • BeanNameAware: 注入 Bean 的名字。
    • BeanClassLoaderAware: 注入类加载器。
    • BeanFactoryAware: 注入 BeanFactory 本身。

4.2 初始化前置处理 (applyBeanPostProcessorsBeforeInitialization)

  • 核心类: 所有注册的 BeanPostProcessor
  • 动作: 遍历执行 postProcessBeforeInitialization。
    • ApplicationContextAwareProcessor: 在此处理 ApplicationContextAware 等高级 Aware 接口的回调。
    • InitDestroyAnnotationBeanPostProcessor: 在此处理 @PostConstruct 注解的方法。

4.3 执行初始化方法 (invokeInitMethods)

  • 动作: 按顺序执行 Bean 的初始化逻辑。
    1. InitializingBean: 调用 afterPropertiesSet()。
    2. 自定义 init-method: 调用 XML 或 @Bean 中配置的 initMethod。

4.4 初始化后置处理 (applyBeanPostProcessorsAfterInitialization)

  • 核心类: 所有注册的 BeanPostProcessor
  • 动作: 遍历执行 postProcessAfterInitialization。
  • 关键点 (AOP): AnnotationAwareAspectJAutoProxyCreator 在这一步介入。
    • 它检查当前 Bean 是否匹配某个切面。
    • 如果匹配，它会创建并返回一个代理对象 (JDK Dynamic Proxy 或 CGLIB) 来替换原始 Bean。
    • 这是 Spring AOP 生效的关键时刻。

4.5 注册销毁 (registerDisposableBeanIfNecessary)

  • 动作: 如果 Bean 实现了 DisposableBean 或配置了 @PreDestroy，容器会记录下来，以便在关闭时调用销毁逻辑。

常见面试题精选

Q1: Spring 是如何解决循环依赖的？

  • 核心: 利用三级缓存。
    1. 一级 (singletonObjects): 存放成品 Bean。
    2. 二级 (earlySingletonObjects): 存放半成品（早期）Bean。
    3. 三级 (singletonFactories): 存放 Bean 工厂 lambda。
  • 流程: A 创建 -> 实例化 -> 放入三级缓存 -> 注入 B -> B 创建 -> 注入 A -> 从三级缓存获取 A 的工厂，生成 A 的早期引用，放入二级缓存 -> B 完成 -> A 获取 B -> A 完成。
  • 限制: 只能解决单例 (Singleton) 且是 Setter/Field 注入 的循环依赖。构造器注入无法解决。

Q2: AOP 代理对象是在 Bean 生命周期的哪一步生成的？

  • 通常是在 初始化后置处理 (postProcessAfterInitialization) 阶段。
  • 例外：如果发生了循环依赖，且该 Bean 需要被代理，那么在三级缓存的 ObjectFactory.getObject() 被调用时（即被依赖注入时），代理对象会提前生成。

Q3: BeanFactory 和 FactoryBean 有什么区别？

  • BeanFactory: 是 Spring 容器的顶层接口，是管理 Bean 的工厂。
  • FactoryBean: 是一个特殊的 Bean。它自己是一个工厂，负责生产其他的对象。通过 getBean("name") 拿到的是它生产的对象；通过 getBean("&name") 拿到的是 FactoryBean 本身。

Q4: @PostConstruct, afterPropertiesSet, init-method 的执行顺序？

  1. @PostConstruct: 由 BeanPostProcessor (Before) 处理，最先执行。
  2. afterPropertiesSet: InitializingBean 接口方法，中间执行。
  3. init-method: XML/注解配置的方法，最后执行。
