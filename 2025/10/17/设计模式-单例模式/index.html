经典场景：唯一的配置管理器

想象一下，在您的应用程序中，需要一个统一的配置管理器（ConfigManager），它负责从文件或数据库加载配置信息（如数据库URL、线程池大小等），并提供给系统的所有部分使用。

问题来了：如果系统中任何部分都可以随意 new ConfigManager()，会导致什么后果？

  1. 状态不一致：每个实例都可能持有不同的配置信息。
  2. 资源浪费：多次从文件或数据库加载配置，造成不必要的 I/O 和内存开销。

因此，我们必须保证 ConfigManager 这个类在整个应用程序的生命周期中，只有一个实例。这就是单例模式的用武之地。

什么是单例模式 (Singleton Pattern)？

单例模式是一种创建型设计模式，它确保一个类只有一个实例，并为该实例提供一个全局访问点。

核心要素

要实现单例模式，一个类必须做到以下三点：

  1. 拥有一个私有的构造函数，以防止外部代码通过 new 操作符直接创建实例。
  2. 拥有一个私有的、静态的该类自身的引用变量，用于持有唯一的实例。
  3. 拥有一个公共的、静态的方法（通常命名为 getInstance()），用于获取这个唯一的实例。如果实例不存在，该方法会负责创建它。

Java 中的多种实现方式

在 Java 中实现单例模式有多种方法，它们在线程安全和性能方面各有优劣。

1. 饿汉式 (Eager Initialization)

这是最简单的实现方式。实例在类加载时就被创建。

  • 优点：实现简单，并且由 JVM 保证线程安全（因为类加载过程是线程安全的）。
  • 缺点：无法实现懒加载（Lazy Loading）。即使从未使用过这个实例，它也会在类加载时被创建，可能造成资源浪费。

2. 懒汉式 (Lazy Initialization) - 线程不安全

只在第一次被请求时才创建实例。

  • 优点：实现了懒加载，节省了资源。
  • 缺点：在多线程环境下是完全错误的！ 如果两个线程同时通过了 if (instance == null) 的判断，它们会各自创建一个实例，从而破坏了单例的唯一性。

3. 懒汉式 - 同步方法 (Thread-Safe)

通过在 getInstance() 方法上添加 synchronized 关键字来保证线程安全。

  • 优点：简单地解决了线程安全问题。
  • 缺点：性能低下。synchronized 会对整个方法加锁，导致每次调用 getInstance() 都会发生同步，即使在实例已经被创建、不再需要同步的情况下也是如此，造成了不必要的性能开销。

4. 双重检查锁定 (Double-Checked Locking, DCL)

这是对同步方法懒汉式的一种优化，旨在减少不必要的同步开销。

  • 优点：兼顾了懒加载、线程安全和高性能。
  • 缺点：实现相对复杂。volatile 关键字是必不可少的，它能防止因 JVM 指令重排而导致的问题。

5. 静态内部类 (Initialization-on-demand holder idiom) - 推荐

这是目前广泛推荐的一种实现方式，它利用了 JVM 的类加载机制来保证线程安全和懒加载。

  • 优点：
    • 线程安全：由 JVM 保证，当一个类被初始化时，其他线程无法进入。
    • 懒加载：只有当 getInstance() 方法被第一次调用时，JVM 才会加载 SingletonHolder 类，从而创建 INSTANCE。
    • 实现简单：代码比 DCL 简洁得多。

6. 枚举 (Enum) - 最佳实践

《Effective Java》作者 Joshua Bloch 极力推荐的方式，它被认为是实现单例模式的最佳方法。

  • 优点：
    • 代码最简洁。
    • 线程安全：由 JVM 保证。
    • 防止反序列化：Java 枚举类天然地解决了在反序列化时重新创建实例的问题。
  • 缺点：
    • 无法实现懒加载（与饿汉式类似）。
    • 如果单例类需要继承其他类，则无法使用。

总结

实现方式 线程安全 懒加载 推荐指数
饿汉式 是 否 ★★★☆☆
懒汉式 (基础) 否 是 ☆☆☆☆☆ (错误)
懒汉式 (同步) 是 是 ★★☆☆☆ (性能差)
双重检查锁定 是 是 ★★★★☆ (实现复杂)
静态内部类 是 是 ★★★★★ (推荐)
枚举 是 否 ★★★★★ (最佳实践)

在绝大多数情况下，直接使用 静态内部类 或 枚举 就是实现单例模式的最好选择。
